# GitOps CD -- GitHub Actions + ArgoCD Integration -- 2-Hour Study Plan

**Total Time:** ~120 minutes
**Created:** 2026-05-10
**Last verified:** 2026-05-10
**Difficulty:** Intermediate -> Advanced (assumes the prior 3 GHA docs from May 5-7 -- workflow/job model, reusable workflows, container CI with build/scan/sign/push to ECR -- and the Apr 16-19 ArgoCD trio: fundamentals, App-of-Apps + ApplicationSets, and Helm + Argo Rollouts. This plan is the **glue between the two halves**: how a build pipeline on GitHub hands off to a pull-based CD controller on the cluster without ever crossing the streams.)

## Overview

This plan covers the *handoff contract* between CI and CD in a real GitOps pipeline: why the CI workflow MUST NOT `kubectl apply` (and what to do instead), the three legitimate ways to update an image tag in the config repo (CI opens a PR, CI direct-pushes to a dev branch, ArgoCD Image Updater does the write-back for you), how to gate environment promotions with **GitHub Environments** (required reviewers, wait timer, environment secrets, deployment branch policies -- the same primitives you use for OIDC scoping), how to wire **ArgoCD notifications** back to GitHub so the PR shows green only after the cluster has actually reconciled, two-repo vs single-repo trade-offs, and the modern alternative -- **Kargo** -- when manual PR promotion stops scaling. After completing it, you should be able to defend, in a code review, the design choice of "CI ends at `git push` to the config repo; ArgoCD takes it from there" and explain exactly which GitHub primitive (environment, branch protection, OIDC trust policy `sub` claim) enforces each gate in your pipeline.

## The CI/CD Handoff Mental Model

```
  app repo                                      config repo (GitOps repo)
+-----------+                                +-----------------------------+
| src/      |                                | apps/                       |
| Dockerfile|                                |   <app>/                    |
| .github/  |     1. push -> release.yml     |     base/   (shared)        |
+-----------+ ----------------------------+  |     overlays/               |
                                          |  |       dev/    values.yaml   |
   build -> scan -> sign -> push to ECR   |  |       staging/values.yaml   |
                                          |  |       prod/   values.yaml   |
                                          |  +-----------------------------+
                                          |               ^
                  2. update image digest  |               |
                  in overlays/dev/values  |               | 4. ArgoCD watches
                  via PR or direct commit +---------------+    config repo,
                                                               syncs cluster

                                           3. PR merged -> ArgoCD detects ->
                                              (auto-sync for dev, manual for prod)
                                           5. ArgoCD notifications -> GitHub
                                              commit status / PR comment
```

The cardinal rule: **CI writes to the config repo; CI never writes to the cluster.** Anything that breaks this rule (a `kubectl apply` step, a `helm upgrade` from CI, a Terraform `kubernetes_manifest` resource managed by the same pipeline that builds the image) is by definition not GitOps -- it's push-based CD wearing a GitOps T-shirt.

## Resources

### 1. GitOps Principles & CI vs CD Separation -- OpenGitOps (CNCF) -- 10 min -- MUST READ

- **URL:** <https://opengitops.dev/>
- **Type:** Official Spec (CNCF)
- **Summary:** The four principles of GitOps -- **Declarative**, **Versioned & Immutable**, **Pulled Automatically**, **Continuously Reconciled** -- as ratified by the CNCF GitOps Working Group. Read the full one-page spec (`Principles` tab) and the FAQ. The principles are short but the implications are deep, and they answer every architectural question you will hit today: "Can my CI pipeline run `kubectl apply`?" -- No (violates *pulled automatically*: the cluster is the puller, not the pushee). "Can I store image tags as environment variables in a Jenkins job?" -- No (violates *declarative* + *versioned*: state must live in Git). "Can my CD tool drift from Git for a hotfix and reconcile later?" -- No (violates *continuously reconciled*: drift detection and auto-remediation are non-negotiable). The "GitOps Continuous Delivery" portion of the FAQ explicitly clarifies that CI and CD are separate concerns: CI's output is an immutable artifact (a signed container image with a digest); CD's input is a Git commit that references that digest. Anything that conflates the two -- a single workflow that builds AND deploys -- is the anti-pattern this whole study plan is fighting. Skim, but internalize: every YAML decision you make today either upholds or violates one of these four lines.

### 2. How to Model Your GitOps Environments -- Kostis Kapelonis (Octopus / formerly Codefresh blog) -- 18 min -- MUST READ

- **URL:** <https://octopus.com/blog/how-to-model-your-gitops-environments>
- **Type:** Authoritative Blog (the canonical environment-promotion article)
- **Summary:** The single most-cited article on GitOps promotion modeling, by Kostis Kapelonis (developer advocate at Codefresh, now Octopus -- the article was migrated when Codefresh was acquired). The thesis you must internalize: **use folders, not branches, for environments.** A `main` branch with `envs/dev/`, `envs/staging/`, `envs/prod/` directories beats `dev`/`staging`/`prod` long-lived branches every single time, because (a) promotions become a `cp envs/dev/version.yml envs/staging/version.yml` -- a *file copy* in a single commit, reviewable as a normal PR diff -- not a `git merge` that risks dragging unrelated changes from a stale dev branch into prod; (b) cherry-picks for hotfixes don't fragment history; (c) promotion can go in any direction (forward for releases, backward for rollbacks) with the same mechanic; (d) `git log envs/prod/` gives you a perfect audit trail of "every change that ever hit prod" without the noise of dev iterations. The article also lays out the four config categories you must keep separate per environment -- (1) container image versions, (2) Kubernetes platform settings (replicas, resource limits), (3) static business config (URLs, region), (4) dynamic business config (feature flags meant to be promoted with the image) -- and warns that mixing categories is what breaks promotions in practice (you can't `cp version.yml` if version, replicas, and feature flags share a file). The article is from 2022 but every principle is still load-bearing in 2026 -- ApplicationSets, Kargo, and GitOps Promoter all assume folder-per-environment as the substrate.

### 3. ArgoCD Image Updater -- Update Methods & Write-Back (Official Docs) -- 15 min -- MUST READ

- **URL:** <https://argocd-image-updater.readthedocs.io/en/stable/basics/update-methods/>
- **Type:** Official Docs
- **Summary:** The component that closes the loop **without** requiring CI to know anything about your config repo. Two write-back methods you must understand: (a) **`argocd`** (the default) -- writes parameter overrides via the ArgoCD API directly into the Application CR's `spec.source.helm.parameters` (or `kustomize.images`) field. **Do not use this in production**: the override lives in cluster state, not Git, which violates the GitOps "versioned in Git" principle and means a `kubectl delete application` wipes your tag history. (b) **`git`** -- writes to a `.argocd-source-<appName>.yaml` file in your config repo's manifest path on a tracked branch. This is the production-correct choice: every image bump becomes a real commit by `argocd-image-updater <noreply@argoproj.io>` with a configurable message template. Critical constraints: your Application MUST track a *branch* (`targetRevision: main`), not a tag or a SHA -- if it tracks `HEAD` you must explicitly set `writeBackConfig.gitConfig.branch`. Update strategies (configured per-image via annotations on the Application): **`semver`** (constrains to a SemVer range like `~v1.2` -- the only safe strategy if your CI tags releases with `v1.2.3`), **`latest`** / **`newest-build`** (sorts by image push timestamp -- only safe if you trust your registry's clock and never re-push old tags), **`digest`** (watches a single mutable tag like `main` and updates the deployed digest whenever the tag is repointed -- pairs naturally with a CI pipeline that pushes `main` after every merge), and **`name`** / **`alphabetical`** (lexical sort -- use for `YYYY-MM-DD-HHMMSS` calver tags, not for SemVer). The write-back-to-git method **does not natively open a PR** -- it commits directly to the tracked branch. If you want PR-based promotion you either use the GitLab pull-request integration, switch to Kargo (resource #8), or have CI open the PR itself (resource #5).

### 4. Mastering Argo CD Image Updater with Helm -- CNCF Blog -- 15 min

- **URL:** <https://www.cncf.io/blog/2024/11/05/mastering-argo-cd-image-updater-with-helm-a-complete-configuration-guide/>
- **Type:** CNCF Official Blog (Nov 2024 -- recent, accurate)
- **Summary:** The end-to-end configuration walkthrough that the official docs hand-wave over. Read this for the **annotation reference** specifically -- Image Updater is configured almost entirely via annotations on the ArgoCD `Application` CR, and the syntax is non-obvious: `argocd-image-updater.argoproj.io/image-list: myalias=ghcr.io/org/api` (the alias `myalias` is what you reference in subsequent annotations -- think of it as a per-image variable name), `argocd-image-updater.argoproj.io/myalias.update-strategy: semver`, `argocd-image-updater.argoproj.io/myalias.allow-tags: regexp:^v\d+\.\d+\.\d+$` (the safety net that prevents accidentally pulling in a `v1.2.3-rc1` pre-release if your CI pushes them), `argocd-image-updater.argoproj.io/myalias.helm.image-name: image.repository`, `argocd-image-updater.argoproj.io/myalias.helm.image-tag: image.tag` (the dotted paths into your `values.yaml`, not the keys themselves -- this is the most common copy-paste error). For ECR specifically: registry credentials are configured via a **script** because IAM role assumption can't be expressed as a static secret -- the `argocd-image-updater-config` ConfigMap references a `credentials: ext:/scripts/auth1.sh` shell script that calls `aws ecr get-login-password`. Also covers the `pull_secret` annotation (`pullsecret:<namespace>/<secret>`) when the cluster pulls from a private registry that ArgoCD itself doesn't have credentials for. The article's biggest practical contribution is the section on testing: `argocd-image-updater test ghcr.io/org/api --update-strategy semver --allow-tags 'regexp:^v\d+'` runs the resolver locally and tells you which tag *would* be selected -- run this before adding annotations to a real Application or you will spend an hour wondering why your prod image isn't updating.

### 5. From Commit to Production -- GitOps Promotion with GitHub Actions, ArgoCD, Helm & Kargo (freeCodeCamp) -- 25 min

- **URL:** <https://www.freecodecamp.org/news/from-commit-to-production-hands-on-gitops-promotion-with-github-actions-argo-cd-helm-and-kargo/>
- **Type:** Hands-On Tutorial (long-form, 2025)
- **Summary:** The most complete *runnable* end-to-end example tying every piece of this plan together: `release.yml` builds and pushes the image to a registry; a second workflow (`promote-to-dev.yml`) checks out the **separate config repo** with a fine-grained PAT or a GitHub App token, runs `yq -i '.image.tag = strenv(NEW_TAG)' apps/api/overlays/dev/values.yaml`, commits, and pushes directly to `main`; ArgoCD's dev Application has `syncPolicy.automated.prune: true` and reconciles within 3 minutes; for staging and prod, a third workflow opens a **pull request** instead of direct-pushing, with `peter-evans/create-pull-request@v6` (the de-facto action -- handles the branch creation, commit, and PR open in a single step); the PR description embeds a link to the staging ArgoCD UI for the reviewer to inspect the diff. The tutorial then layers on **Kargo** to replace the per-environment workflow files with a single `Stage` CRD -- if you finish this tutorial you will understand both the "manual GitHub Actions promotion" model and the "controller-driven Kargo promotion" model, and which trade-offs push you from one to the other (typically: ~5 environments + ~10 services + drift detection across all 50 combinations is the line where hand-rolled promotion workflows become unmaintainable). Skim the Kargo section if you're short on time -- the GHA + ArgoCD pieces are the load-bearing part for today.

### 6. GitHub Environments -- Deployment Protection Rules (GitHub Docs) -- 12 min -- MUST READ

- **URL:** <https://docs.github.com/en/actions/reference/workflows-and-actions/deployments-and-environments>
- **Type:** Official Docs
- **Summary:** The four protection-rule primitives you must know: (1) **Required reviewers** -- up to six users or teams; only one needs to approve; reviewers must have at least read access; the job sits in `waiting` state and the assignees get an email + a UI banner; **environment secrets are NOT available to the job until approval is granted** (this is the critical security boundary -- a malicious PR cannot exfiltrate prod secrets even if it triggers the workflow, because the secrets aren't injected until a human approves). (2) **Wait timer** -- 1 to 43,200 minutes (30 days max); compounds with required reviewers; useful for "let prod soak in canary for 30 minutes before letting the next stage proceed"; **wait time is not billable** (so a 24-hour wait timer doesn't burn 24 hours of your Actions minutes quota). (3) **Deployment branch policies** -- restrict which branches can deploy to this environment via name patterns (`main`, `release/*`) or to "protected branches only"; this is what prevents a feature branch from accidentally being able to assume the prod IAM role even if the workflow YAML doesn't gate on `github.ref`. (4) **Custom protection rules** (GA in 2024) -- third-party gates like ServiceNow change tickets, Datadog SLO health, or your own webhook receiver; the deployment is paused and a webhook is fired with a callback URL the rule must hit to approve or reject. **Critical plan caveat**: on Free / Pro / Team plans, required reviewers and wait timers **only work for public repositories**; private-repo support requires GitHub Enterprise or one of the higher-tier plans. Most production teams hit this surprise the first time they try to gate a private monorepo -- check before you architect around it. Pair with **environment secrets** (separate from repo secrets, scoped to the environment, only available to jobs that name `environment: production` and have passed all protection rules) and with the **OIDC `sub` claim** -- `repo:org/repo:environment:production` is the trust-policy condition that tells AWS "only mint creds when GitHub Actions is running against the protected `production` environment" -- the cross-platform handshake that makes GitHub Environments and AWS IAM cooperate.

### 7. ArgoCD Notifications -- GitHub Service & Triggers (Official Docs) -- 10 min

- **URL:** <https://argo-cd.readthedocs.io/en/stable/operator-manual/notifications/services/github/>
- **Type:** Official Docs
- **Summary:** The *return path* from cluster state back to the GitHub PR. ArgoCD Notifications is a controller (built into ArgoCD core since v2.3 -- the standalone `argocd-notifications` repo is archived; do not install it separately) that watches Application state changes and fires templates to one of ~25 destinations. The GitHub service authenticates as a **GitHub App** (preferred -- you upload the App's private key as a Kubernetes secret; per-installation tokens are minted on demand, scoped to specific repos) or via a PAT (acceptable for testing only). It supports three GitHub-native event types: **commit statuses** (`POST /repos/{owner}/{repo}/statuses/{sha}` -- the green/yellow/red dot next to a commit; this is what you wire to "ArgoCD has finished syncing the commit that triggered this PR" so the PR's "Checks" section reflects cluster reality, not just CI), **deployments** (`POST /repos/{owner}/{repo}/deployments` -- the GitHub Deployments API; populates the "Environments" sidebar on the repo home page so you can see "production: deployed 4 minutes ago, sha abc1234" without leaving GitHub), and **pull request comments** (rendered as Markdown in the PR conversation -- useful for embedding the ArgoCD diff or the rollout health). Companion reading: the **Triggers** doc (<https://argo-cd.readthedocs.io/en/stable/operator-manual/notifications/triggers/>) lists the built-in triggers (`on-deployed`, `on-health-degraded`, `on-sync-failed`, `on-sync-status-unknown`) -- these are what you bind to your templates. The killer pattern: trigger `on-deployed` -> service `github` -> template that posts a commit status of "argocd/sync" success on the SHA of the *manifest commit* (which, if your CI pipeline opened the PR with that SHA in the body, will appear as a check on the original feature-branch PR via cross-repo status linking). The user is then looking at the PR they opened in the *app* repo, and seeing a green check from ArgoCD that the *config* repo's commit has reconciled in the cluster -- the full loop closed without a single human "is it deployed yet?" Slack message.

### 8. Promotion Made Easy with Kargo (Akuity Blog) -- 10 min -- (When PRs Stop Scaling)

- **URL:** <https://akuity.io/blog/promotion-made-easy-with-kargo>
- **Type:** Vendor Blog (Akuity is the company behind ArgoCD)
- **Summary:** The "what comes after hand-rolled GHA promotion workflows" answer. Kargo is a CNCF Sandbox project (incubated at Akuity, now community-governed) that introduces three CRDs: **`Warehouse`** (subscribes to image registries, Git repos, and Helm charts; produces immutable **Freight** -- a versioned bundle of artifact references with an auto-generated alias like `sleepy-swan-123`); **`Stage`** (a promotion target -- typically one per environment-cluster combination -- with a `requestedFreight` selector and a `verification` block that runs an Argo Workflows job to check health/SLOs before declaring the promotion successful); **`Promotion`** (the imperative request to move a specific Freight into a specific Stage; can be auto-created by a `PromotionPolicy` or manually via the Kargo UI / `kargo promote` CLI). Promotion **steps** are the unit of work -- `git-clone`, `kustomize-set-image`, `helm-update-image`, `git-commit`, `git-push`, `argocd-update` are the built-ins; you compose them per-stage in the `Stage` spec. Why use Kargo instead of `peter-evans/create-pull-request` + ArgoCD Image Updater + N copies of `promote.yml`? Three reasons: (1) **Centralized visibility** -- one UI shows "freight `sleepy-swan-123` is in dev + staging, blocked at prod by failed verification check" across all your services; with hand-rolled workflows you have N PRs in N repos and must mentally aggregate; (2) **Verification as a first-class concept** -- the `verification` block runs *before* the promotion is marked complete, so you can encode "wait 30 min, then check Datadog SLO, then check Argo Rollouts analysis" without writing custom GHA YAML; (3) **Multi-stage pipelines** -- Kargo's `Stage` graph supports diamond/fan-out shapes ("dev -> [staging-us, staging-eu] -> prod") that are awkward to express as GHA `needs:` dependencies. Read this article for the conceptual model only -- you don't need to install Kargo today; just understand it exists and when to reach for it.

### 9. (Reference) peter-evans/create-pull-request -- The Action You'll Use Most (GitHub) -- 5 min

- **URL:** <https://github.com/peter-evans/create-pull-request>
- **Type:** Official Action README (de-facto community standard)
- **Summary:** The action that powers ~90% of "CI opens a PR against the config repo" workflows in the wild. Skim the README for two things: (1) **token requirements** -- the default `GITHUB_TOKEN` cannot trigger downstream workflows (a security feature: workflows can't recursively fire workflows or you get infinite loops), so the PR it opens won't run your config repo's CI checks. To make ArgoCD's commit-status-on-PR notification work, you must use a **GitHub App token** (`actions/create-github-app-token` is the official way to mint one) or a fine-grained PAT with `pull_requests: write` + `contents: write` on the *config* repo specifically. (2) **The `signoff: true` option** -- adds a `Signed-off-by:` trailer (DCO compliance) to the auto-generated commit; required by some org's branch protection rules. The action's `commit-message`, `branch`, `delete-branch: true` (cleans up the branch after merge), and `body` inputs are the five fields you'll fill in every time. The combination "GHA workflow on app repo's release -> mints config-repo App token -> `yq` updates `values.yaml` in the config repo working tree -> `peter-evans/create-pull-request` opens a PR" is the canonical pattern; the alternative ("CI runs `git push` directly to the config repo's main branch") works for dev environments where you want auto-deploy, but anything past dev should be PR-gated.

## Study Tips

- **Memorize the four GitOps principles as a checklist for every CI design choice.** Declarative, Versioned & Immutable, Pulled, Continuously Reconciled. Anytime you're about to add a step to a workflow, ask: which of these does it violate? `kubectl apply` -> violates "Pulled". Storing the image tag in a Helm `--set` flag passed via CI -> violates "Versioned" (the value isn't in Git). Letting your workflow exit successfully before ArgoCD has reconciled -> violates "Continuously Reconciled" because you've now set up a system where the green PR check doesn't actually mean prod is updated.
- **Folders, not branches.** The single most common mistake is creating `dev`/`staging`/`prod` long-lived branches in the config repo and trying to "promote" by merging dev into staging. This **always** ends in a merge nightmare within six months. One branch (`main`), folders per environment, promotions are PRs that copy/edit values files. If you're sketching a config repo layout on a whiteboard and you hear yourself say "we'll have a staging branch", stop and re-read resource #2.
- **Pick exactly one image-update mechanism per repo, not three.** The three options -- (a) CI direct-pushes to dev branch, then a separate `promote.yml` opens PRs for staging and prod; (b) ArgoCD Image Updater handles all environments via annotations + write-back; (c) Kargo orchestrates everything via Stage CRDs -- all *work*, but they conflict if you mix them. Image Updater overwriting a value in `dev/values.yaml` while a CI workflow tries to bump the same file at the same time is a classic merge-conflict-storm, and debugging "who pushed this commit" across two automations is misery. Pick one mental model per service.
- **The OIDC `sub` claim and GitHub Environments are designed to compose.** From yesterday's plan you know `repo:org/repo:ref:refs/heads/main` scopes a role to main-branch pushes. Today: `repo:org/repo:environment:production` scopes it to "any workflow run that has `environment: production` in its job spec AND has passed all the environment's protection rules (reviewers, wait timer, branch policy)". This is the single best practice for prod IAM trust policies -- it forces the AWS role to be assumable only when GitHub has *already* enforced human approval, so even a malicious PR that somehow forges a workflow file cannot mint prod creds. Use environment-scoped subs for anything that touches prod state; reserve branch-scoped subs for read-only or dev-only roles.
- **Trace the loop end-to-end before you build it.** On paper, draw: "developer pushes to feature branch in app repo" -> ... -> "production pod is running new image". Number every arrow. Now annotate which system owns each arrow (GHA workflow runner, GitHub UI, ECR, config repo, ArgoCD controller, Kubernetes API, ArgoCD Notifications, GitHub Deployments API). For every numbered arrow that crosses a system boundary, identify the auth mechanism (OIDC token, App token, ArgoCD ServiceAccount, etc.). If you can't fill in every arrow's owner and auth, your design isn't done yet. The result should be 8-12 arrows; if it's >20, you have too many systems.

## Next Steps

- **Day 5 (Progressive Delivery -- Argo Rollouts canary + analysis)** -- the natural follow-on. The `release.yml` -> config-repo PR -> ArgoCD sync chain you've now wired up deploys *all* of prod at once. Argo Rollouts (covered briefly in the Apr 19 Helm prod-patterns doc) replaces the `Deployment` resource with a `Rollout` that supports canary, blue-green, and progressive traffic-shifting via service mesh integration, plus `AnalysisTemplate` resources that auto-rollback on Prometheus/Datadog metric degradation. Combined with Kargo's verification steps, you get "promotion succeeds only if the canary's p99 latency stays below 200ms for 10 minutes" as a declarative gate.
- **Tying it back to supply chain** -- yesterday's Cosign keyless signature is what makes today's image-updater write-back trustworthy. The full chain: GHA workflow signs image with `cosign sign --yes` -> writes digest into config repo PR -> ArgoCD pulls the manifest -> a Sigstore Policy Controller (or Kyverno with the `verifyImages` rule) admission-webhook in the cluster verifies the signature against the GHA workflow identity (`certificate-identity-regexp: '^https://github.com/my-org/'`) before letting the pod schedule. If the signature is missing or the identity doesn't match, the deployment is rejected -- even if ArgoCD itself was compromised and tried to inject an arbitrary image. This is SLSA Level 3 in practice: ArgoCD is a trusted reconciler, but not a trusted *signer*; the trust root is the build platform's OIDC identity.
- **GitOps Promoter (argoproj-labs)** -- the new-er alternative to Kargo, focused specifically on the "PR-based promotion across environments with status checks" use case. Smaller scope than Kargo (no Freight/Warehouse abstraction; just promotion as PRs with health gates), but tighter ArgoCD integration. Worth comparing once you've used Kargo on a real service for a quarter. <https://github.com/argoproj-labs/gitops-promoter>
- **Multi-cluster promotion patterns** -- when "production" is `[us-east-1, us-west-2, eu-west-1]`, the Stage graph in Kargo (or the Application set in ArgoCD) becomes a fan-out: one `staging` stage feeds three `prod-<region>` stages, each with its own verification. Read the ArgoCD `ApplicationSet` Cluster generator docs (already covered Apr 17) with this lens and you'll see why ApplicationSets and Kargo are complementary, not competing -- ApplicationSet generates the *Applications* (one per cluster), Kargo controls the *promotion of versions* through them.
