# GitHub Actions -- Core Review & Advanced Workflows -- 2-Hour Study Plan

**Total Time:** ~120 minutes
**Created:** 2026-05-05
**Difficulty:** Intermediate -> Advanced (assumes you can already write a basic `.github/workflows/*.yml` with a job and a few steps; this plan reinforces fundamentals quickly and then goes deep on production patterns: contexts, conditionals, concurrency, reusable workflows, and matrix strategies)

## Overview

This plan moves past "hello world" and into the mechanics that production workflows actually depend on: how the github/env/needs/matrix/inputs contexts compose, why `if: always()` makes a job uncancellable and what to use instead, how concurrency groups serialize deploys vs auto-cancel stale PR builds, when to reach for a reusable workflow over a composite action, and how matrix include/exclude lets you collapse 12 hand-written jobs into a 6-line strategy block. After completing it, you will be able to read any production GitHub Actions workflow file and predict its behavior on `push`, `pull_request`, `workflow_dispatch`, and re-run scenarios -- and you will know which 2026 syntax patterns (`GITHUB_OUTPUT`, environment files, Node 20 default, `secrets: inherit`) replace older deprecated forms.

## Resources

### 1. Workflow Syntax Reference (Official Docs) -- 20 min

- **URL:** <https://docs.github.com/en/actions/reference/workflows-and-actions/workflow-syntax>
- **Type:** Official Docs (reference)
- **Summary:** The single source of truth for every key you can put in a workflow file. Skim, do not read top-to-bottom -- focus on these sections: `on.<event>` (event filters like `branches`, `paths`, `types`), `concurrency` (group + cancel-in-progress), `jobs.<id>.needs` (DAG dependencies), `jobs.<id>.if` (job-level conditional), `jobs.<id>.strategy` (matrix, fail-fast, max-parallel), `jobs.<id>.timeout-minutes` (default is 360, almost always too high), `jobs.<id>.environment` (deployment env + protection rules), and `jobs.<id>.outputs` (how a job exposes values to downstream jobs via `needs.<id>.outputs.<name>`). Treat this page as the YAML grammar you Cmd-F into when reviewing a real workflow -- you do not need to memorize it, you need to know it exists.

### 2. Contexts Reference -- github / env / vars / secrets / needs / matrix / runner / inputs (Official Docs) -- 15 min

- **URL:** <https://docs.github.com/en/actions/writing-workflows/choosing-what-your-workflow-does/accessing-contextual-information-about-workflow-runs>
- **Type:** Official Docs (reference)
- **Summary:** The eight contexts are the variables you read inside `${{ ... }}` expressions, and 90% of "why doesn't my workflow work" debugging is "wrong context, wrong scope". Critical distinctions: `github.event_name` vs `github.event.action` (the latter only exists for events with subtypes like `pull_request: [opened, synchronize]`), `github.ref` (`refs/heads/main`) vs `github.ref_name` (`main`) vs `github.head_ref` (only set on `pull_request` events, the source branch), `env` is repo/job/step-level env vars while `vars` is repo/org/environment configuration variables (added 2023, the non-secret counterpart to `secrets`), `needs.<job>.outputs` flows data forward in the DAG, and `matrix.<key>` is only available inside a matrix job. Understand the lifecycle: contexts available at job level differ from those at step level (e.g., `steps.<id>.outputs` only exists after that step runs).

### 3. Events That Trigger Workflows -- workflow_dispatch / repository_dispatch / schedule (Official Docs) -- 15 min

- **URL:** <https://docs.github.com/en/actions/reference/events-that-trigger-workflows>
- **Type:** Official Docs (reference)
- **Summary:** Read the sections on `push`, `pull_request`, `pull_request_target` (the dangerous one -- runs against base repo with secrets, used carefully for forked-PR labeling), `schedule` (cron syntax in UTC, runs on default branch only, can be skipped during high GitHub load), `workflow_dispatch` (manual trigger with up to 25 typed inputs as of late 2025 -- choice/boolean/string/number/environment), and `repository_dispatch` (external API trigger with arbitrary `client_payload` JSON, accessible as `github.event.client_payload.*`). The mental model: `workflow_dispatch` = "a human or another GHA workflow kicks this off"; `repository_dispatch` = "an external system (Jenkins, a webhook, a Lambda) kicks this off"; `schedule` = "the clock kicks this off". Each has different security implications -- `pull_request_target` in particular is a known foot-gun because it provides `secrets` to code that may have been modified by an untrusted contributor.

### 4. GitHub Actions Concurrency Patterns -- cancel-in-progress: false for Parallel Deployments (Blog) -- 12 min

- **URL:** <https://dev.to/kanta13jp1/github-actions-concurrency-patterns-cancel-in-progress-false-for-parallel-deployments-1bjm>
- **Type:** Blog (practitioner deep dive)
- **Summary:** The cleanest explanation of the four concurrency patterns you will actually use in production. (1) **Per-PR auto-cancel for CI:** `group: ci-${{ github.workflow }}-${{ github.ref }}`, `cancel-in-progress: true` -- new pushes to a PR cancel the in-flight CI run, saving runner minutes. (2) **Serialized prod deploys:** `group: deploy-production`, `cancel-in-progress: false` -- multiple deploy attempts queue rather than cancel each other, but note GitHub's queue depth is 1 (any third pending job evicts the second). (3) **Per-environment serialization:** `group: deploy-${{ inputs.environment }}` -- staging and prod queues are independent, so a long staging deploy does not block prod hotfix. (4) **Matrix-aware parallel deploys:** `group: deploy-${{ matrix.region }}`, `cancel-in-progress: false` -- each region serializes independently. The article also covers the trap: if you set `cancel-in-progress: true` on a deploy workflow and someone re-runs CI, you can cancel a half-finished production deploy mid-rollout.

### 5. Composite Actions vs Reusable Workflows -- What is the Difference (DEV.to) -- 12 min

- **URL:** <https://dev.to/n3wt0n/github-composite-actions-vs-reusable-workflows-updated-2023-bl8>
- **Type:** Blog (practitioner comparison)
- **Summary:** The decision article you reference every time someone says "let's DRY this up". Composite actions = reusable **steps** within a job (live in `.github/actions/<name>/action.yml`, called as `uses: ./.github/actions/<name>`); reusable workflows = reusable **whole jobs** (live in `.github/workflows/*.yml` with `on.workflow_call`, called as `uses: ./.github/workflows/<file>.yml` or `uses: org/repo/.github/workflows/<file>.yml@<ref>`). Critical differences to internalize: composite actions **cannot use `secrets`** (must be passed as `with:` inputs in plaintext -- never pass real secrets this way), composite actions **cannot have `if:` on individual steps controlled from outside**, composite actions run on the caller's runner (no `runs-on:` of their own), reusable workflows can call other actions but **cannot call other reusable workflows beyond 4 levels of nesting**, and reusable workflows expose `outputs` at the job level via `jobs.<id>.outputs`. Rule of thumb: if you need a different runner, secrets, environments, or a multi-job pipeline, use a reusable workflow; if you need 3-5 shell/action steps glued together (like "checkout + setup-node + cache + install"), use a composite action.

### 6. Reuse Workflows -- workflow_call inputs / secrets / outputs (Official Docs) -- 15 min

- **URL:** <https://docs.github.com/en/actions/how-tos/reuse-automations/reuse-workflows>
- **Type:** Official Docs (how-to)
- **Summary:** The mechanics of building a production reusable workflow. Define `on.workflow_call.inputs` with explicit `type` (string/boolean/number) and `required` flags -- without `type`, the workflow will not load. Define `on.workflow_call.secrets.<name>` to declare which secrets the called workflow expects; the caller passes them via `secrets:` (one per line) or the magic `secrets: inherit` (passes ALL caller secrets, the right choice when the called workflow lives in the same org). Define `on.workflow_call.outputs.<name>.value: ${{ jobs.<id>.outputs.<name> }}` to bubble a value from a step inside the called workflow back to the caller -- two layers of mapping: step output -> job output -> workflow output. Pin called workflows to a SHA (`@a1b2c3d`) or a version tag (`@v2.1.0`), never `@main` in production -- a malicious or buggy commit in the central platform repo otherwise instantly affects every caller. The page also covers limitations (matrix on a calling job, max 20 unique reusable workflows per workflow run, environment-secrets behavior).

### 7. The Matrix Strategy in GitHub Actions (RunsOn Blog) -- 12 min

- **URL:** <https://runs-on.com/github-actions/the-matrix-strategy/>
- **Type:** Blog (practitioner deep dive)
- **Summary:** The best concrete walk-through of `include` / `exclude` / `fail-fast` / `max-parallel` semantics, with diagrams of the cross-product expansion. Key mechanics: `matrix:` keys form a Cartesian product (3 OS x 4 Node versions = 12 jobs), `include:` adds extra dimensions to specific combinations (e.g., add `experimental: true` only when `os: ubuntu-latest, node: 22`) OR appends new combinations that are not in the cross-product, `exclude:` removes specific combinations (the "skip Windows + Node 18" case), `fail-fast: true` (default) cancels all sibling jobs the moment any matrix cell fails -- usually wrong for production (one flaky test cancels 11 valid runs), so set `fail-fast: false` for cross-platform CI and `continue-on-error: true` on individual experimental cells. `max-parallel` throttles how many cells run concurrently -- essential when each cell hits a shared rate-limited resource (DockerHub pulls, an external API). The article also shows dynamic matrices: a setup job emits a JSON array via `GITHUB_OUTPUT`, a downstream job consumes it as `strategy.matrix: ${{ fromJson(needs.setup.outputs.matrix) }}` -- the pattern for "test only the packages whose files changed".

### 8. Mastering GitHub Actions -- Insights and Pitfalls of `if` Conditions (DEV.to) -- 10 min

- **URL:** <https://dev.to/cloud-sky-ops/mastering-github-actions-insights-and-pitfalls-of-if-conditions-1j87>
- **Type:** Blog (practitioner deep dive)
- **Summary:** The `if:` gotchas reference. Four status check functions: `success()` (default -- "all needed jobs succeeded"), `failure()` ("any needed job failed"), `cancelled()` ("workflow was cancelled"), `always()` ("ALWAYS run, including on cancellation"). The trap: `always()` makes a job **uncancellable** -- a user clicking "Cancel workflow" cannot stop it, which matters for cleanup/notify jobs that you actually do want to skip on cancel. The recommended replacement is `if: !cancelled()` -- runs on success or failure but respects user cancellation. Second trap: `success()` returns **false** if any needed job is `skipped`, so a downstream cleanup job conditioned on `if: success()` will be skipped if an upstream job was conditionally skipped, even though nothing failed -- the fix is `if: !failure()` (or the more explicit `!failure() && !cancelled()`). Third trap: when combining `always()` with another condition, you must wrap as `if: ${{ always() && <other> }}` because the parser otherwise treats the whole expression as a literal. Fourth trap: step-level `if:` does not need `${{ }}`, but job-level `if:` is parsed identically -- both work, the inconsistency is in older docs/examples. Read this once carefully; you will reference it forever.

### 9. GitHub Actions: Deprecating save-state and set-output Commands (GitHub Changelog) -- 8 min

- **URL:** <https://github.blog/changelog/2022-10-11-github-actions-deprecating-save-state-and-set-output-commands/>
- **Type:** Official Changelog (currency check)
- **Summary:** The 2026-currency cheatsheet. The old workflow command `echo "::set-output name=foo::bar"` is deprecated -- replace with `echo "foo=bar" >> $GITHUB_OUTPUT`. Same shape for the related deprecations: `::save-state name=foo::bar` -> `echo "foo=bar" >> $GITHUB_STATE`, `::set-env name=FOO::bar` -> `echo "FOO=bar" >> $GITHUB_ENV`, `::add-path::/some/path` -> `echo "/some/path" >> $GITHUB_PATH`. Why it changed: the stdout-parsing form was a log-injection vulnerability (an attacker who controlled a log line could set arbitrary outputs); the file-based form is parsed only from a runner-local file. Although the actual removal has been postponed multiple times (still emitting deprecation warnings as of late 2025), every workflow you write today should use the env-file form -- it is the only form documented in current Actions docs. Pair this with Node 20 being the default for JavaScript actions (Node 16 actions emit deprecation warnings since late 2023) and the broader pattern: GitHub Actions deprecations are slow but inevitable, and they always favor more explicit/secure forms.

## Study Tips

- **Build a "context cheatsheet" as you go.** Open a scratch buffer and write down which contexts are available where: `github.*` everywhere, `env.*` everywhere (after declaration), `secrets.*` only at workflow/job level (not in `if:` conditions on the workflow trigger), `needs.*.outputs.*` only in jobs that declare `needs:`, `matrix.*` only inside matrix jobs, `inputs.*` only when triggered by `workflow_dispatch` or `workflow_call`, `steps.<id>.*` only after that step has run. Most "expression evaluates to empty string" bugs come from referencing a context that does not exist in that scope -- and the runner silently coerces to empty rather than erroring.
- **Trace the conditional truth table for your real workflow.** For each job, ask: what happens on (a) push to main, (b) PR opened, (c) PR push, (d) workflow_dispatch, (e) prior job success, (f) prior job failure, (g) prior job skipped, (h) workflow cancelled? Most teams discover their cleanup/notify jobs do the wrong thing in 2-3 of these 8 cases. The combination `if: ${{ !cancelled() && (success() || failure()) }}` is what you usually want for "run regardless, but stop if I hit Cancel".
- **For concurrency, write the group key on a whiteboard before writing YAML.** What is the unit of serialization? Per-PR (`${{ github.workflow }}-${{ github.ref }}`)? Per-environment (`deploy-${{ inputs.env }}`)? Per-region (`deploy-${{ matrix.region }}`)? Globally (`deploy-prod`)? Then ask: do I want to cancel-in-progress (cheap CI feedback) or queue (paid deploys you cannot interrupt)? Almost every concurrency bug is "wrong group key granularity" or "cancel-in-progress: true on a deploy".
- **Pick reusable-workflow vs composite by the secret + runner test.** If the reused unit needs `secrets:` or its own `runs-on:`, it MUST be a reusable workflow. If it is "5 steps that always run together on whatever runner the caller picked", it should be a composite action -- composite actions are cheaper (no extra job, no separate runner spin-up) and integrate into the caller's job logs as nested step groups. The wrong choice is usually "reusable workflow for what should have been a composite", which fragments your workflow run UI into 6 tiny jobs.
- **Pin everything to a SHA in production.** `uses: actions/checkout@v4` is convenient; `uses: actions/checkout@a1b2c3d4...` is safe. The Tj-actions/changed-files supply-chain incident in early 2025 (a popular third-party action's tag was repointed at a malicious commit, exfiltrating secrets from thousands of repos) is the canonical "this is why we pin SHAs" cautionary tale. Dependabot can keep SHA-pinned actions updated.
