## Overview

This study plan covers AWS Step Functions as a serverless workflow orchestration service -- from the foundational Standard vs Express workflow decision and Amazon States Language (ASL) state types through error handling with Retry/Catch, service integration patterns (Request Response, .sync, .waitForTaskToken), Parallel and Distributed Map states for fan-out processing, input/output data manipulation, versioning and aliases for safe deployments, and cost optimization strategies. ## Resources

### 1. Choosing Between Standard and Express Workflows -- AWS Official Docs ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/step-functions/latest/dg/choosing-workflow-type.html
- **Type:** Official Docs
- **Summary:** The definitive comparison page that drives the most fundamental Step Functions architectural decision. Standard Workflows run up to 1 year with exactly-once execution semantics, persist state between transitions, store execution history for 90 days (retrievable via API), and support all integration patterns including .sync, .waitForTaskToken, Distributed Map, and Activity tasks -- priced per state transition at $0.025 per 1,000 transitions. Express Workflows run up to 5 minutes with at-least-once semantics (asynchronous) or at-most-once (synchronous), do NOT persist state between transitions, require CloudWatch Logs for execution history (not accessible via Step Functions API), and only support Request Response integrations -- priced per execution count, duration, and memory consumed. Express Workflows come in two variants: asynchronous (returns confirmation immediately, results in CloudWatch Logs, triggered by EventBridge or other async sources) and synchronous (waits for completion and returns result, invokable from API Gateway or Lambda, ideal for microservice orchestration). Critical constraint: workflow type is immutable after creation -- you cannot convert Standard to Express or vice versa. The decision framework: use Standard for long-running durable workflows, non-idempotent operations (payments, EMR cluster creation), workflows needing execution history audit trails, and any workflow requiring .sync or callback patterns; use Express for high-volume event processing (IoT data ingestion, streaming transformation), short-duration idempotent operations, and cost-sensitive high-throughput workloads where at-least-once delivery is acceptable.

### 2. Workflow States and Amazon States Language Structure -- AWS Official Docs ⏱️ 20 min
- **URLs:**
  - State types overview: https://docs.aws.amazon.com/step-functions/latest/dg/workflow-states.html
  - ASL structure: https://docs.aws.amazon.com/step-functions/latest/dg/statemachine-structure.html
  - Input/output processing: https://docs.aws.amazon.com/step-functions/latest/dg/concepts-input-output-filtering.html
- **Type:** Official Docs (three pages, ~7 min each)
- **Summary:** The foundational pages that establish the vocabulary and mental model for building workflows. The state types overview introduces all eight state types: (1) **Task** -- invokes an AWS service or activity, the workhorse state, (2) **Choice** -- conditional branching based on input values (if/else logic), (3) **Wait** -- delays execution for a fixed duration or until a timestamp, (4) **Parallel** -- creates multiple branches that execute concurrently, (5) **Map** -- iterates over a collection, running a set of steps for each item (Inline mode for small datasets, Distributed mode for massive-scale S3 processing), (6) **Pass** -- passes input to output with optional transformation (useful for injecting fixed data or restructuring JSON), (7) **Succeed** -- terminates execution successfully, (8) **Fail** -- terminates execution with an error name and cause message. The ASL structure page covers the top-level fields of a state machine definition: Comment, StartAt, States, QueryLanguage (JSONata vs JSONPath), and TimeoutSeconds. The input/output processing page is critical for understanding the five-stage data flow pipeline that applies to every state using JSONPath: InputPath (filter raw input) -> Parameters (construct new JSON with static values and selected input fields) -> ResultSelector (reshape the task result) -> ResultPath (merge result back into original input or replace it) -> OutputPath (filter final output passed to next state). Note that AWS introduced JSONata as a recommended alternative to JSONPath at re:Invent 2024, using Arguments/Output/Assign fields instead -- understand both since existing state machines use JSONPath and new ones may use JSONata. After reading these three pages, you will have the complete mental model for how data flows through a Step Functions workflow.

### 3. Error Handling -- Retry and Catch Mechanisms -- AWS Official Docs ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/step-functions/latest/dg/concepts-error-handling.html
- **Type:** Official Docs
- **Summary:** One of the most important pages for production Step Functions design and a frequent interview topic. Step Functions provides two declarative error handling mechanisms: **Retry** (automatic retries with exponential backoff) and **Catch** (fallback state transitions when retries are exhausted). Retry fields: ErrorEquals (array of error names to match), IntervalSeconds (initial wait, default 1), MaxAttempts (default 3), BackoffRate (multiplier, default 2.0), MaxDelaySeconds (caps the maximum wait between retries), and JitterStrategy (FULL or NONE, randomizes delays to prevent thundering herd). Catch fields: ErrorEquals (errors to match), Next (fallback state name), ResultPath (where to place error information in state output). Built-in error types to memorize: States.ALL (matches any error, must be alone and last in the array), States.TaskFailed (any error except States.Timeout), States.Timeout (task exceeded TimeoutSeconds or HeartbeatSeconds), States.HeartbeatTimeout (specifically heartbeat timeout), States.Runtime (unprocessable exceptions that cannot be retried), States.Permissions (insufficient IAM permissions), and States.DataLimitExceeded (payload exceeds 256 KB limit). Evaluation order: Step Functions tries all Retry rules first (top to bottom, first match wins), and only when retries are exhausted does it evaluate Catch rules (also top to bottom). Always put specific error matchers before States.ALL. Critical billing insight: each retry attempt counts as a state transition in Standard Workflows, so aggressive retry configurations directly increase cost. This page connects to yesterday's Lambda error handling study -- Step Functions Retry/Catch replaces the need for Lambda-side retry logic when Lambda is invoked synchronously via a Task state, moving error handling from imperative code to declarative workflow definition.

### 4. Service Integration Patterns -- AWS Official Docs ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/step-functions/latest/dg/connect-to-resource.html
- **Type:** Official Docs
- **Summary:** The core mechanics page explaining the three integration patterns that determine how Step Functions communicates with AWS services. (1) **Request Response** (default): Step Functions calls the service API and immediately proceeds to the next state after receiving the HTTP response -- the service may still be processing. Use for fire-and-forget operations like publishing to SNS or sending to SQS. Supported by both Standard and Express Workflows. Resource ARN format: arn:aws:states:::serviceName:apiAction. (2) **Run a Job (.sync)**: Step Functions calls the service and pauses until the job completes, polling automatically. Append .sync to the resource ARN. Supported services include Lambda (Invoke), ECS (RunTask), Batch (SubmitJob), Glue (StartJobRun), SageMaker (CreateTrainingJob), CodeBuild (StartBuild), and others. Standard Workflows only -- Express Workflows do NOT support .sync. Resource ARN: arn:aws:states:::serviceName:apiAction.sync. (3) **Wait for Callback (.waitForTaskToken)**: Step Functions pauses indefinitely (up to 1 year for Standard) until an external process calls SendTaskSuccess or SendTaskFailure with the task token. The token is accessed via the context object ($$.Task.Token in JSONPath). Use for human approval workflows, external system integration, and asynchronous processing where the target system calls back. Standard Workflows only. Resource ARN: arn:aws:states:::serviceName:apiAction.waitForTaskToken. Always set HeartbeatSeconds to detect stuck tasks. Beyond these three patterns, understand the two integration categories: **Optimized integrations** (Lambda, DynamoDB, ECS, SQS, SNS, Step Functions, Glue, SageMaker, etc.) provide a curated API surface with Step-Functions-specific features, and **AWS SDK integrations** (200+ services, 10,000+ API actions) let you call virtually any AWS API directly from a Task state without Lambda in the middle -- a key cost optimization pattern you already encountered with API Gateway's direct AWS service integrations.

### 5. Distributed Map for Large-Scale Processing -- AWS Official Docs ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/step-functions/latest/dg/state-map-distributed.html
- **Type:** Official Docs
- **Summary:** The Map state in Distributed mode is Step Functions' answer to massive-scale parallel processing -- processing millions of items from S3 by running up to 10,000 concurrent child workflow executions. Unlike Inline Map (which runs iterations within the parent execution, subject to the 25,000 execution history event limit and 256 KB payload limit), Distributed Map launches each iteration as a separate child workflow execution with its own isolated execution history, enabling effectively unlimited scale. Key configuration: ItemReader reads from S3 data sources (CSV, JSON, JSON Lines, Parquet, or S3 object inventory with up to 100 million items and 10 GB file size), ItemBatcher groups items into batches for efficiency (reducing child execution count), MaxConcurrency controls parallelism (default 10,000, set lower to respect downstream service limits like Lambda concurrency), ToleratedFailurePercentage/ToleratedFailureCount define how many child failures are acceptable before the entire Map state fails, and ResultWriter writes consolidated results back to S3. Distributed Map is Standard Workflows only. The three Map modes form a progression: Inline Map for small in-memory collections (up to hundreds of items), Distributed Map for S3-scale datasets (millions of items), and Parallel state for fixed-branch fan-out (known number of concurrent paths). Use case examples: processing S3 inventory files, ETL on thousands of CSV files, batch image processing, large-scale data validation.

### 6. Building Cost-Effective Step Functions Workflows -- AWS Compute Blog ⏱️ 20 min
- **URL:** https://aws.amazon.com/blogs/compute/building-cost-effective-aws-step-functions-workflows/
- **Type:** Blog (AWS Official)
- **Summary:** An essential blog post from the AWS Compute team that bridges the Standard vs Express decision from Resource 1 into practical cost optimization strategies with concrete pricing comparisons. Covers the pricing models side-by-side: Standard at $0.025 per 1,000 state transitions (each Retry counts as a transition, each Map iteration counts as transitions, Parallel branches multiply transitions) versus Express at $1.00 per million requests plus duration-based charges (per 100ms, tiered by memory from 64 MB to 3 GB). The key insight: for a workflow with 5 state transitions running 1,000,000 times monthly, Standard costs approximately $125 while Express costs approximately $3.10 -- a 40x difference. The blog then covers three cost optimization patterns: (1) **Migrate Standard to Express** when your workflow completes in under 5 minutes and can tolerate at-least-once semantics (the post provides a migration checklist), (2) **Nest Express inside Standard** to get the best of both worlds -- the outer Standard workflow handles long-running orchestration and .sync/.waitForTaskToken patterns while inner Express workflows handle high-volume, short-duration sub-workflows at lower cost, (3) **Use direct SDK integrations** to eliminate Lambda invocations for simple AWS API calls (DynamoDB PutItem, SQS SendMessage, SNS Publish) -- this mirrors the API Gateway direct integration pattern you studied yesterday. The nested workflow pattern is particularly powerful: a Standard workflow orchestrating an order process can call an Express sub-workflow for the high-frequency payment validation steps, dramatically reducing cost while maintaining exactly-once guarantees for the outer flow.

### 7. Versioning, Aliases, and Gradual Deployments -- AWS Official Docs ⏱️ 10 min
- **URLs:**
  - Overview: https://docs.aws.amazon.com/step-functions/latest/dg/concepts-cd-aliasing-versioning.html
  - Gradual deployment: https://docs.aws.amazon.com/step-functions/latest/dg/version-rolling-deployment.html
- **Type:** Official Docs (two pages, ~8 min each)
- **Summary:** Launched in June 2023, versioning and aliases bring the same deployment safety patterns you studied for Lambda (Apr 6) to Step Functions. Key concepts: (1) **Versions** are numbered, immutable snapshots of a state machine definition created when you publish -- each version gets a unique ARN (arn:aws:states:region:account:stateMachine:name:version). Unlike Lambda where $LATEST is the default, Step Functions revisions are internal snapshots that must be explicitly published as versions. (2) **Aliases** are named pointers to one or two state machine versions, enabling traffic routing for gradual deployments. An alias can route 100% to one version or split traffic between two versions with configurable weights (e.g., 90% to version 3, 10% to version 4). (3) **Deployment strategies**: canary (route small percentage to new version, monitor, then shift 100%), blue-green (immediate full cutover with instant rollback capability), and linear (gradual percentage increase over time). Aliases integrate with CloudWatch Alarms for automatic rollback -- if the alarm triggers during the canary period, traffic shifts back to the original version. (4) **Execution association**: when an execution starts via an alias, it runs the version resolved at start time and continues on that version even if the alias is updated mid-execution. This mirrors Lambda alias behavior exactly -- and in production, you would combine both: Step Functions alias routes to a new workflow version that references Lambda aliases pointing to new function versions, enabling coordinated canary deployments across the entire orchestration stack.

### 8. Best Practices and Terraform Patterns -- AWS Official Docs + AWS DevOps Blog ⏱️ 10 min (split reading)
- **URLs:**
  - Best practices: https://docs.aws.amazon.com/step-functions/latest/dg/sfn-best-practices.html
  - Terraform best practices: https://aws.amazon.com/blogs/devops/best-practices-for-writing-step-functions-terraform-projects/
- **Type:** Official Docs + Blog (AWS Official)
- **Summary:** Two complementary resources that cover operational excellence and infrastructure-as-code patterns. The best practices page covers eight production-critical guidelines: (1) nest Express Workflows inside Standard to optimize cost while maintaining durability, (2) always set TimeoutSeconds on Task states to prevent stuck executions (default is no timeout -- executions can hang indefinitely until the 1-year Standard limit), (3) always set HeartbeatSeconds for callback and activity tasks to detect failures faster than the task timeout, (4) pass S3 ARNs instead of large payloads to stay within the 256 KB payload limit (a common production failure), (5) manage the 25,000 execution history event hard limit by using Distributed Map or nesting child executions, (6) configure Lambda-specific retry policies for transient errors (Lambda.ServiceException, Lambda.AWSLambdaException, Lambda.SdkClientException, Lambda.TooManyRequestsException), (7) maintain at least 100 open polls per activity ARN, (8) handle CloudWatch Logs resource policy limits using the /aws/vendedlogs/ prefix. The Terraform blog (September 2023) covers project structure (separate directories for state machines, modules, and Lambda functions), ASL definition management (use templatefile() to parameterize ASL JSON files instead of inline heredoc -- never hardcode ARNs), IAM policy creation using aws_iam_policy_document data sources for compile-time validation, and CloudWatch log group configuration with proper IAM permissions. The key Terraform pattern: define ASL in a separate .asl.json file, use templatefile() to inject Lambda ARNs and DynamoDB table names as variables, and reference the rendered definition in aws_sfn_state_machine.definition.

## Study Tips

- Build a Step Functions integration pattern decision matrix with three columns (Request Response, Run a Job .sync, Wait for Callback .waitForTaskToken) and rows for: when to use, workflow type support, execution behavior, timeout handling, IAM requirements, and example services. This single artifact captures the most commonly confused Step Functions topic -- candidates frequently mix up which patterns Express Workflows support (only Request Response) and when to use .sync vs .waitForTaskToken (synchronous job completion polling vs external system callback).

- Map the error handling hierarchy end-to-end for a Task state invoking Lambda: Lambda function throws an error -> Step Functions receives the error with name and cause -> evaluates Retry rules top-to-bottom (first match wins, applies exponential backoff with optional jitter) -> if all retries exhausted, evaluates Catch rules top-to-bottom (first match wins, transitions to fallback state with error info at ResultPath) -> if no Catch matches, the entire execution fails. Compare this to Lambda's own three invocation model error contracts from Day 1: when Lambda is invoked via a Step Functions Task state, it is a synchronous invocation -- Lambda returns the error directly to Step Functions, which then applies its Retry/Catch logic. This means Step Functions Retry replaces the need for client-side retry logic that synchronous Lambda callers normally implement.

- Draw the five-stage input/output processing pipeline (InputPath -> Parameters -> ResultSelector -> ResultPath -> OutputPath) and trace a concrete JSON payload through each stage. This is the hardest conceptual topic in Step Functions because each stage has a subtly different purpose: InputPath filters what the state sees, Parameters constructs the task's input, ResultSelector reshapes the task's output, ResultPath merges the result back into the original input (or replaces it), and OutputPath filters what the next state receives. The most common mistake is confusing ResultPath (where to put the result within the original input) with OutputPath (what subset of the combined result to pass forward).
