# RDS & Aurora Deep Dive -- 2-Hour Study Plan

**Total Time:** ~120 minutes
**Created:** 2026-03-25
**Difficulty:** Intermediate-Advanced

## Overview

This study plan covers Amazon RDS high availability and read scaling (Multi-AZ deployments vs Read Replicas), Aurora's revolutionary distributed storage architecture (6-copy 4/6 write quorum across 3 AZs), Aurora Serverless v2 ACU-based auto-scaling, Aurora Global Database for cross-region DR with sub-second replication lag, Aurora cloning via copy-on-write protocol, RDS Proxy for connection pooling and failover resilience, and RDS backup strategies (automated backups with PITR vs manual snapshots) -- culminating in an Aurora vs RDS decision framework. After completing this plan, you will be able to explain why Aurora's storage-level replication eliminates the traditional database replication bottleneck, articulate precisely when Multi-AZ (synchronous HA) differs from Read Replicas (asynchronous read scaling), choose between Aurora Serverless v2 and provisioned capacity based on workload patterns, and design database architectures that leverage Aurora Global Database for multi-region resilience -- building on your EBS storage knowledge (Mar 24) where you learned that RDS uses EBS volumes under the hood, your KMS encryption deep dive (Mar 9) for RDS encryption at rest, and your load balancer knowledge (Mar 16) which parallels how RDS Proxy distributes database connections.

## Resources

### 1. Amazon RDS Multi-AZ Deployments -- Official Feature Page ⏱️ 20 min
- **URL:** https://aws.amazon.com/rds/features/multi-az/
- **Type:** Official Docs
- **Summary:** The definitive starting point that explains both Multi-AZ deployment types side by side: the classic Multi-AZ DB instance deployment (one primary + one passive standby in a different AZ, synchronous replication, automatic failover completing as quickly as 60 seconds with zero data loss, standby cannot serve read traffic) and the newer Multi-AZ DB cluster deployment (one writer + two readable standbys across 3 AZs, semisynchronous replication using transaction log shipping, failover typically under 35 seconds, up to 2x faster transaction commit latency, standbys serve read traffic via a reader endpoint) -- includes a comparison table contrasting Single-AZ, one-standby, and two-readable-standby deployments across engine availability, read capacity, latency, and failover duration; this is the conceptual foundation for understanding why Multi-AZ is purely a high availability mechanism (not a scaling solution) and how it differs architecturally from Read Replicas, which you will read about next; also note that Multi-AZ DB clusters currently support only MySQL and PostgreSQL (not Oracle, SQL Server, or MariaDB)

### 2. Working with DB Instance Read Replicas -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_ReadRepl.html
- **Type:** Official Docs
- **Summary:** The comprehensive reference for RDS Read Replicas covering asynchronous replication from a source DB instance, the ability to create replicas in the same region or cross-region (critical for DR), promotion of a read replica to a standalone DB instance (which breaks replication permanently), creating read replicas of read replicas (daisy-chaining with increased replication lag), and the key constraint that read replicas serve only read traffic by default -- use this page to internalize the fundamental distinction from Multi-AZ: Read Replicas use asynchronous replication for horizontal read scaling and can exist cross-region, while Multi-AZ uses synchronous replication for automatic failover within a region; also note that you can make a read replica Multi-AZ-enabled, combining both patterns (a read replica in another region that itself has a standby for HA), and that cross-region read replicas incur data transfer charges between regions; supported for MySQL, MariaDB, PostgreSQL, Oracle (with caveats), and SQL Server (limited)

### 3. Amazon Aurora Storage and Reliability -- Official Documentation ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/Aurora.Overview.StorageReliability.html
- **Type:** Official Docs
- **Summary:** The official page explaining Aurora's cluster volume architecture -- a single virtual SSD-backed volume that automatically replicates data across three Availability Zones, with storage growing automatically up to 128 TiB (or 256 TiB for some versions) without manual provisioning; covers the two storage pricing configurations: Aurora I/O-Optimized (no per-request I/O charges, best when I/O spending exceeds 25% of total Aurora costs -- eliminates the surprise I/O billing problem that plagued early Aurora adopters) and Aurora Standard (per-million-request pricing for lower I/O workloads); the key architectural insight is that Aurora separates compute from storage entirely -- adding a read replica does not copy data, it just connects another compute instance to the shared cluster volume, which is fundamentally different from RDS where each read replica maintains its own separate copy of data via asynchronous replication; after reading this page, proceed immediately to the next resource which explains the quorum mechanism that makes this distributed storage fault-tolerant

### 4. Amazon Aurora Under the Hood: Quorums and Correlated Failure -- AWS Database Blog ⏱️ 15 min
- **URL:** https://aws.amazon.com/blogs/database/amazon-aurora-under-the-hood-quorum-and-correlated-failure/
- **Type:** Blog (AWS Official)
- **Summary:** The first post in a four-part AWS blog series that explains Aurora's quorum-based replication in detail -- each 10 GB segment of the database volume is replicated six ways across three AZs (two copies per AZ), writes are issued to all six copies and acknowledged when four respond (4/6 write quorum), reads require three copies (3/6 read quorum), enabling Aurora to tolerate the loss of an entire AZ plus one additional node without losing write availability, and to tolerate the loss of an entire AZ plus two additional nodes without losing read availability (the "AZ+1" failure model); at 10 GB segments on a 10 Gbps network, quorum repair takes under a minute, keeping the Mean Time To Repair (MTTR) low enough that the probability of correlated double failures within the repair window is negligible; this is the intellectual core of Aurora's architecture -- understanding why 4/6 and 3/6 quorum numbers were chosen (they overlap, guaranteeing any read quorum includes at least one member of the most recent write quorum) will give you a deep answer for any interview question about Aurora durability; also read the second post in the series at https://aws.amazon.com/blogs/database/amazon-aurora-under-the-hood-quorum-reads-and-mutating-state/ if time permits

### 5. How Aurora Serverless v2 Works -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/aurora-serverless-v2.how-it-works.html
- **Type:** Official Docs
- **Summary:** The conceptual guide to Aurora Serverless v2 covering how capacity is measured in Aurora Capacity Units (each ACU provides approximately 2 GiB of RAM plus corresponding CPU and networking), scaling ranges from 0 ACUs (auto-pause, available on platform v3) to 256 ACUs in 0.5 ACU increments, scaling is continuous and granular (unlike Serverless v1 which doubled capacity at discrete steps), each writer and reader DB instance scales independently, readers in promotion tiers 0-1 scale with the writer while tiers 2-15 scale independently based on their own load; also covers mixed-configuration clusters where you can combine Serverless v2 instances with provisioned instances in the same cluster (use provisioned for the writer with predictable baseline load, Serverless v2 for readers that handle variable traffic spikes); the critical decision point is: use Serverless v2 for variable, unpredictable, or spiky workloads (dev/test environments, new applications, seasonal traffic) and provisioned for steady, predictable workloads where you can forecast capacity and want lower per-ACU cost

### 6. Using Amazon Aurora Global Database -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/aurora-global-database.html
- **Type:** Official Docs
- **Summary:** The comprehensive guide to Aurora Global Database which uses storage-level replication (not engine-level) to replicate data from a primary cluster in one region to up to five secondary read-only clusters in different regions with latency typically under one second -- covers the architecture (primary region has one read/write cluster, each secondary region has a read-only cluster with up to 16 read replicas), switchover vs failover (managed planned switchover for operational rotation with zero data loss vs managed unplanned failover for disaster recovery with RPO in seconds and RTO in minutes), write forwarding (secondary region instances forward write requests to the primary region, enabling global write acceptance with slightly higher latency), and limitations; this is distinct from RDS cross-region read replicas in that Aurora Global Database replicates at the storage layer rather than the database engine layer, resulting in much lower replication lag (sub-second vs seconds-to-minutes for engine-level replication) and zero performance impact on the primary; this resource connects directly to the DR strategies you will study in Week 5

### 7. Cloning a Volume for an Amazon Aurora DB Cluster -- Official Documentation ⏱️ 10 min
- **URL:** https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/Aurora.Managing.Clone.html
- **Type:** Official Docs
- **Summary:** Explains Aurora's fast database cloning using the copy-on-write protocol -- when you create a clone, Aurora initially points both the source and clone at the same data pages (zero additional storage, completes in minutes regardless of database size), and only allocates new storage when either the source or clone modifies data (each modified page is copied to a new location while the original remains for the other cluster); covers use cases (creating dev/test environments from production data without doubling storage costs, running analytics queries against a clone without impacting production performance, blue/green deployments), limitations (maximum 15 copy-on-write clones per source, must be in the same AWS Region, cannot clone across accounts for Serverless v1), and the critical distinction from snapshots: cloning is faster and more space-efficient because it avoids a full data copy upfront, whereas restoring from a snapshot requires copying all data before the new cluster becomes available

### 8. Aurora vs. RDS: Engineering Guide to Choose the Right AWS Database -- Bytebase ⏱️ 10 min
- **URL:** https://www.bytebase.com/blog/aurora-vs-rds/
- **Type:** Blog (Technical)
- **Summary:** A regularly updated (last revised April 2025) engineering comparison of RDS vs Aurora across nine dimensions: supported databases (RDS supports MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, Db2; Aurora supports only MySQL and PostgreSQL), architecture (RDS runs vanilla engines on EBS; Aurora separates compute from a purpose-built distributed storage layer), performance (Aurora claims up to 5x MySQL and 3x PostgreSQL throughput), elasticity (Aurora Serverless v2 provides auto-scaling that RDS lacks natively), high availability (Aurora's storage replication vs RDS Multi-AZ), and pricing (Aurora's per-I/O charges surprised early adopters until I/O-Optimized was introduced in 2023); the practical decision framework: choose RDS when you need an engine Aurora does not support (Oracle, SQL Server, MariaDB, Db2), for small predictable workloads where cost matters most, or for dev/test where HA is not critical; choose Aurora when you need high read throughput with fast replica addition, sub-second failover, storage auto-scaling, or Serverless v2 elasticity; this synthesizes everything from the previous seven resources into an actionable selection guide

## Study Tips

- Build a clear mental separation between two completely different replication concepts: **RDS Multi-AZ replication** (synchronous, same region, standby cannot serve reads in classic mode, purpose is automatic failover for high availability) vs **RDS Read Replicas** (asynchronous, same or cross-region, serve read traffic, purpose is horizontal read scaling and optionally DR via promotion). The newer Multi-AZ DB cluster deployment blurs this line by giving you two readable standbys, but it is still fundamentally an HA mechanism with read capacity as a bonus -- it does not replace read replicas for cross-region read scaling. In interviews, if someone asks "how do you scale reads for RDS?", the answer is read replicas; if they ask "how do you achieve HA for RDS?", the answer is Multi-AZ. Aurora changes the game entirely because its shared storage architecture means replicas do not need to replicate data -- they just mount the same volume.

- When studying Aurora's 4/6 write quorum, work through the math yourself: with 6 copies across 3 AZs (2 per AZ), losing an entire AZ removes 2 copies, leaving 4 -- exactly enough for a write quorum. Losing an AZ plus one more node leaves 3 copies -- not enough to write (need 4) but enough to read (need 3). This is why Aurora can tolerate "AZ+1" failures for writes and "AZ+2" failures for reads. The 10 GB segment size is chosen so that repair (re-replicating a failed segment to a healthy node) takes under a minute on a 10 Gbps network, keeping the window where a second failure could cause data loss extremely small. Connect this to your EBS knowledge from yesterday: RDS standard runs on EBS volumes that replicate within a single AZ, while Aurora built its own distributed storage layer that spans three AZs -- this is the fundamental architectural difference that justifies Aurora's higher price.

- For the Aurora vs RDS decision framework, remember three scenarios that immediately point to RDS over Aurora: (1) you need an engine Aurora does not support (Oracle, SQL Server, MariaDB, Db2), (2) your workload is small and cost-sensitive where Aurora's minimum costs are not justified, (3) you need exact open-source MySQL/PostgreSQL compatibility without any cloud-native modifications. Three scenarios that point to Aurora: (1) you need fast read scaling without replicating data to each replica, (2) you need sub-second cross-region replication via Global Database, (3) you have unpredictable workloads that benefit from Serverless v2 auto-scaling. The I/O-Optimized pricing tier (introduced 2023) is a critical exam topic -- it eliminated Aurora's unpredictable I/O billing problem and made cost comparison with RDS much more straightforward.
