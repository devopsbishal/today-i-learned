# AWS Global Accelerator & Edge Services -- 2-Hour Study Plan

**Total Time:** ~120 minutes
**Created:** 2026-03-18
**Difficulty:** Intermediate

## Overview

This study plan covers AWS Global Accelerator -- the service that uses Anycast static IP addresses and the AWS global backbone network to route TCP/UDP traffic to optimal regional endpoints with sub-30-second failover. You will learn the component hierarchy (accelerators, listeners, endpoint groups, endpoints), traffic management mechanisms (traffic dials, endpoint weights, client affinity), health checking and cross-region failover, custom routing accelerators for gaming and VoIP use cases, and the critical three-way comparison of Global Accelerator vs CloudFront vs Route53 latency-based routing -- building directly on your CloudFront deep dive (Mar 13), Route53 routing policies (Mar 11), and ALB/NLB knowledge (Mar 16) to complete your edge services understanding for Week 3.

## Resources

### 1. What Is AWS Global Accelerator -- Official Conceptual Overview ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/global-accelerator/latest/dg/what-is-global-accelerator.html
- **Type:** Official Docs
- **Summary:** The foundational page that establishes the "what" and "why" of Global Accelerator -- it provides two static Anycast IPv4 addresses (or four for dual-stack) that serve as fixed entry points announced from AWS edge locations worldwide, routing traffic over the AWS backbone to optimal regional endpoints instead of traversing the unpredictable public internet; covers the two accelerator types (standard for load-balanced routing to ALB/NLB/EC2/EIP endpoints, custom routing for deterministic mapping to specific EC2 instances), BYOIP support, and the core value proposition of instant failover without DNS caching delays -- read this first to understand how Global Accelerator complements the DNS-based routing you learned with Route53 (Mar 11) and the content caching you learned with CloudFront (Mar 13)

### 2. How AWS Global Accelerator Works -- Architecture and Traffic Flow ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/global-accelerator/latest/dg/introduction-how-it-works.html
- **Type:** Official Docs
- **Summary:** The critical "how it works" page that explains the mechanics you need for both implementation and interviews: Anycast IP addresses are announced from multiple edge locations simultaneously so BGP routes clients to the nearest edge, then traffic travels over the AWS backbone to the closest healthy regional endpoint; covers TCP termination at the edge (Global Accelerator terminates client TCP connections at edge locations and establishes new connections to endpoints -- the same proxy pattern as ALB from your Mar 16 study, but at the global edge level), the two independent network zones per edge location for fault tolerance (if one zone fails, clients retry on the second static IP from the isolated zone), connection idle timeouts (TCP: 340 seconds, UDP: 30 seconds), traffic dials (percentage control at the endpoint group level for blue-green deployments), endpoint weights (proportional distribution within an endpoint group, default 128, max 255), client affinity (NONE uses 5-tuple hash for equal distribution, SOURCE_IP uses 2-tuple hash for session stickiness), and the fail-open behavior where unhealthy-endpoint-groups trigger failover to the nearest healthy group in another region

### 3. Global Accelerator Components -- Accelerators, Listeners, Endpoint Groups, Endpoints ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/global-accelerator/latest/dg/introduction-components.html
- **Type:** Official Docs
- **Summary:** The component reference page that maps the four-level hierarchy you need to internalize: accelerator (top-level resource, owns the two static IPs, has a DNS name like a1234567890abcdef.awsglobalaccelerator.com), listener (processes inbound connections on a protocol and port or port range -- TCP, UDP, or both), endpoint group (associated with exactly one AWS Region, contains one or more endpoints, has a traffic dial for percentage-based control and configurable health check settings), and endpoint (the actual resource receiving traffic -- ALB, NLB, EC2 instance, or Elastic IP for standard accelerators; VPC subnets for custom routing); this hierarchy parallels the ALB structure you already know (ALB -> listener -> listener rules -> target groups -> targets) but operates at the global rather than regional level -- understanding this mapping will help you reason about Global Accelerator architectures quickly

### 4. Understanding Global Accelerator Use Cases -- Official Scenarios ⏱️ 10 min
- **URL:** https://docs.aws.amazon.com/global-accelerator/latest/dg/introduction-benefits-of-migrating.html
- **Type:** Official Docs
- **Summary:** Covers the five canonical use cases you should be ready to articulate in interviews: (1) scaling without IP changes -- add/remove endpoints behind the same two static IPs without updating clients, critical for IoT and mobile apps where client updates are difficult; (2) latency-sensitive applications -- gaming, ad-tech, financial services where Anycast routing and backbone transit reduce latency by 25-30% compared to public internet paths; (3) disaster recovery and multi-region resiliency -- automatic detection of failing endpoints with instant rerouting to the next closest healthy region, no DNS TTL wait; (4) origin protection -- mask your ALBs and EC2 instances behind two static IPs with built-in AWS Shield Standard DDoS protection (connecting to your Shield knowledge from Mar 10); (5) VoIP and online gaming -- custom routing accelerators that map clients to specific EC2 instances for matchmaking, session management, and real-time communication

### 5. How Custom Routing Accelerators Work -- Deterministic Port-to-Instance Mapping ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/global-accelerator/latest/dg/about-custom-routing-how-it-works.html
- **Type:** Official Docs
- **Summary:** The deep dive on custom routing -- the second accelerator type that most people skip but is a powerful interview differentiator: instead of load-balancing traffic across endpoints, custom routing creates a static, deterministic mapping between Global Accelerator listener port ranges and specific EC2 private IP addresses and ports within VPC subnets; your application calls the Global Accelerator API once to retrieve this mapping, then uses application logic (matchmaking servers, session border controllers) to tell clients which GA port to connect to, and GA routes them to exactly the right EC2 instance; critically, all destinations are denied by default and you must explicitly allow traffic via AllowCustomRoutingTraffic API (subnet-level or socket-level granularity); custom routing does NOT support health checks or automatic failover -- your application logic owns that responsibility; endpoints are VPC subnets only (not ALBs or NLBs); ideal for multiplayer gaming (assign players to game servers by skill/region), VoIP (route callers to specific media servers), and real-time collaboration platforms

### 6. AWS Global Accelerator vs CloudFront vs Route 53 -- A Practical Guide for Architects ⏱️ 20 min
- **URL:** https://dev.to/t2c/aws-global-accelerator-vs-cloudfront-vs-route-53-a-practical-guide-for-architects-53mb
- **Type:** Article (Dev.to)
- **Summary:** An excellent three-way comparison article (Sep 2025) that synthesizes the decision framework you need after studying all three services: Route53 is the DNS switchboard (domain resolution, routing policies, health-check-based failover -- but subject to TTL caching delays you learned about on Mar 11), CloudFront is the application edge for HTTP/HTTPS content with caching and WAF integration (your Mar 13 knowledge), and Global Accelerator is the network-level edge for TCP/UDP acceleration with static IPs and instant failover; provides three concrete architectural patterns -- (1) content-heavy single-region: Route53 + CloudFront, (2) low-latency multi-region API: Global Accelerator fronting regional ALBs/NLBs with Route53 for naming, (3) mixed workloads: split by hostname with HTTP/static to CloudFront and real-time/custom protocols to Global Accelerator; includes the key constraint that Global Accelerator cannot front a CloudFront distribution -- when you need both, route different hostnames with Route53

### 7. AWS Global Accelerator Pricing ⏱️ 10 min
- **URL:** https://aws.amazon.com/global-accelerator/pricing/
- **Type:** Official Docs
- **Summary:** The pricing page you need to understand the cost model and make informed architecture decisions: Global Accelerator charges two components -- (1) a fixed hourly fee of $0.025 per accelerator ($18/month running 24/7), and (2) a Data Transfer Premium (DT-Premium) per GB that varies by source region and destination edge location (US/Canada to Europe: $0.015/GB, US/Canada to Asia Pacific: $0.035/GB, Australia to other regions: $0.070-$0.101/GB); the DT-Premium is charged only on the dominant traffic direction (inbound or outbound, not both), and is in addition to standard EC2 data transfer charges -- so Global Accelerator always costs more than standard data transfer, making it a premium service justified by latency reduction and instant failover; the example calculation shows 10,000 GB/month at $128 total ($18 fixed + $110 DT-Premium); compare this with CloudFront's per-request + per-GB model (no fixed fee) and Route53's per-query model ($0.40-$0.70 per million queries) to understand when the GA premium is worth it

## Study Tips

- Build a three-column comparison table with Route53 Latency Routing, CloudFront, and Global Accelerator as columns, and these rows: OSI layer, protocols supported, caching, static IPs, failover speed, health checking, pricing model, and primary use cases. This is the interview cheat sheet for edge services. The single most important distinction: Route53 operates at the DNS layer (subject to TTL caching, cheapest), CloudFront operates at the HTTP application layer (caching is the differentiator, good for static + dynamic HTTP), and Global Accelerator operates at the network transport layer (TCP/UDP acceleration with instant failover, most expensive). When an interviewer asks "how would you reduce global latency?", your answer depends entirely on the protocol and caching requirements.
- Trace the complete traffic flow for a Global Accelerator request and compare it to what you learned about CloudFront: with GA, the client resolves the GA DNS name, connects to the nearest edge via Anycast, GA terminates TCP at the edge and opens a new connection to the regional endpoint over the AWS backbone -- there is NO caching at any layer. With CloudFront (your Mar 13 study), traffic hits POP -> REC -> Origin Shield -> origin with caching at every tier. This mental model of "GA = backbone shortcut without caching" vs "CloudFront = backbone shortcut WITH caching" is the key to answering comparison questions correctly.
- Pay attention to how Global Accelerator integrates with services you already know: ALB and NLB (Mar 16) serve as endpoints behind GA, AWS Shield Standard (Mar 10) protects GA static IPs automatically, Route53 (Mar 11) can point a friendly domain name at the GA DNS name using an alias record, and traffic dials + endpoint weights provide blue-green deployment capabilities similar to Route53 weighted routing but without DNS TTL delays. These cross-service connections demonstrate the architectural thinking interviewers want to see.
