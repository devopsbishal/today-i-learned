# Load Balancers -- ALB vs NLB vs GLB -- 2-Hour Study Plan

**Total Time:** ~120 minutes
**Created:** 2026-03-16
**Difficulty:** Intermediate

## Overview

This study plan covers the three Elastic Load Balancing types in AWS -- Application Load Balancer (Layer 7), Network Load Balancer (Layer 4), and Gateway Load Balancer (Layer 3 for appliance insertion) -- along with target group types (instance, IP, Lambda, ALB), listener rules and advanced request routing, health check configuration and behavior, cross-zone load balancing (and why the default differs between ALB and NLB), connection draining / deregistration delay, and sticky sessions (duration-based vs application-based cookies). After completing it, you will be able to articulate exactly when to choose ALB vs NLB vs GLB for a given architecture, explain how GLB uses GENEVE encapsulation and Gateway Load Balancer Endpoints to transparently insert third-party appliances into a VPC traffic path, configure target groups and health checks correctly, and understand the operational behaviors (cross-zone distribution, deregistration draining, session affinity) that directly affect availability and user experience -- all of which builds on your VPC advanced networking (Feb 19-25), Transit Gateway (Feb 23), CloudFront (Mar 13, where ALB serves as an origin), Network Firewall (Mar 10, which GLB complements for third-party appliance patterns), and Auto Scaling (Feb 12, which registers targets into these target groups).

## Resources

### 1. How Elastic Load Balancing Works -- Official Architecture Overview ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/elasticloadbalancing/latest/userguide/how-elastic-load-balancing-works.html
- **Type:** Official Docs
- **Summary:** The foundational page that explains how ELB works across all load balancer types -- load balancer nodes deployed per Availability Zone, DNS resolution with 60-second TTL, the critical difference between internet-facing (public IPs on nodes) and internal (private IPs only) schemes, and the routing algorithm each type uses (ALB: round robin per target group, NLB: flow hash using protocol + source/dest IP + port + TCP sequence number, GLB: 5-tuple flow hash); most importantly, this page contains the definitive explanation of cross-zone load balancing with a concrete example showing that with 2 AZ targets in AZ-A and 8 in AZ-B, enabling cross-zone gives each target 10% of traffic while disabling it gives AZ-A targets 25% each and AZ-B targets only 6.25% each -- and the default behavior differs by type: ALB always has cross-zone enabled at the load balancer level (can be disabled per target group), while NLB and GLB have it disabled by default, which matters for cost (NLB cross-zone traffic incurs inter-AZ data transfer charges, ALB does not) and for zonal isolation patterns you learned about in your VPC and Transit Gateway studies

### 2. ALB vs NLB vs GLB -- Official AWS Comparison Page ⏱️ 15 min
- **URL:** https://aws.amazon.com/compare/the-difference-between-the-difference-between-application-network-and-gateway-load-balancing/
- **Type:** Official Docs
- **Summary:** AWS's own comparison page that puts all three load balancer types side by side across every dimension that matters for architecture decisions: OSI layer (ALB at Layer 7 inspecting HTTP headers, NLB at Layer 4 examining IP/port, GLB at Layer 3 as a transparent network gateway), proxy behavior (ALB terminates connections and creates new ones to targets enabling SSL offloading and session stickiness, NLB passes connections through preserving client source IP, GLB forwards without terminating), supported protocols (ALB: HTTP/HTTPS/gRPC/WebSocket, NLB: TCP/UDP/TLS, GLB: all IP protocols via GENEVE encapsulation), target types (ALB: instance/IP/Lambda, NLB: instance/IP/ALB, GLB: instance/IP), and routing algorithms; use this page to build the decision matrix you will need in interviews -- choose ALB when you need content-based routing (path, host, headers, query strings), choose NLB when you need static IPs, extreme performance, or non-HTTP protocols, choose GLB when inserting third-party security appliances transparently into the traffic path

### 3. Target Groups for Application Load Balancers -- Official Deep Dive ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/elasticloadbalancing/latest/application/load-balancer-target-groups.html
- **Type:** Official Docs
- **Summary:** The comprehensive reference for ALB target groups covering everything you need for both implementation and interviews: target types (instance targets route via primary private IP, IP targets enable multiple apps on same instance using different IPs or cross-VPC targets via peering/PrivateLink which connects to your Feb 20 VPC Peering study, Lambda targets invoke a single function per target group), routing algorithms (round robin, least outstanding requests for latency-sensitive workloads, weighted random with Automatic Target Weights for anomaly mitigation), slow start mode (ramps new targets from 0 to full share over 30-900 seconds so freshly launched instances behind Auto Scaling can warm caches), deregistration delay / connection draining (default 300 seconds, range 0-3600 -- when a target deregisters, it enters "draining" state where in-flight requests complete but no new requests arrive, critical for graceful deployments and Auto Scaling scale-in), and all target group attributes; pay special attention to the interaction between deregistration delay and Auto Scaling that you studied on Feb 12 -- if your scale-in terminates an instance before draining completes, users get errors

### 4. Sticky Sessions for Application Load Balancers -- Official Guide ⏱️ 10 min
- **URL:** https://docs.aws.amazon.com/elasticloadbalancing/latest/application/sticky-sessions.html
- **Type:** Official Docs
- **Summary:** Covers the two sticky session mechanisms that ALB supports and when to use each: duration-based stickiness uses a load-balancer-generated cookie (AWSALB) with a configurable expiration from 1 second to 7 days, while application-based stickiness lets your application set its own cookie (e.g., JSESSIONID) and ALB creates a companion cookie (AWSALBAPP) that maps to it -- the key difference is that application-based cookies give your app control over when stickiness expires (e.g., on logout) while duration-based cookies use a fixed timer; critical constraints to remember: sticky sessions require cross-zone load balancing to be enabled (attempting to enable stickiness with cross-zone disabled will fail), stickiness works at the target group level not the listener level, and sticky sessions can cause uneven load distribution so they should only be used when your application truly requires session affinity (e.g., in-memory session state that has not been externalized to ElastiCache or DynamoDB, which you will study in Week 4)

### 5. Health Checks for ALB Target Groups -- Official Configuration Reference ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/elasticloadbalancing/latest/application/target-group-health-checks.html
- **Type:** Official Docs
- **Summary:** The definitive reference for ALB health check settings: protocol (HTTP/HTTPS), port (defaults to traffic port but can be overridden for a dedicated health endpoint), path (defaults to / but should be a dedicated /health endpoint that checks dependencies), interval (5-300 seconds, default 30), timeout (2-120 seconds, default 5, must be less than interval), healthy threshold (2-10 consecutive successes, default 5), unhealthy threshold (2-10 consecutive failures, default 2), and success codes (matcher, default 200 but can be a range like 200-299); the asymmetry between healthy threshold (5) and unhealthy threshold (2) is intentional -- targets fail fast (2 checks) but recover slowly (5 checks) to prevent flapping; also covers the critical "fail-open" behavior where if ALL targets in ALL enabled AZs become unhealthy, the load balancer routes to all targets regardless (last-resort availability), and the target health states lifecycle (initial -> healthy/unhealthy -> draining -> unused); compare this page with the NLB health checks page (https://docs.aws.amazon.com/elasticloadbalancing/latest/network/target-group-health-checks.html) to note that NLB health checks can also use TCP (not just HTTP), which matters for non-HTTP backends

### 6. NLB: Using ALB as a Target -- Official Docs ⏱️ 10 min
- **URL:** https://docs.aws.amazon.com/elasticloadbalancing/latest/network/application-load-balancer-target.html
- **Type:** Official Docs
- **Summary:** Covers the fourth target type that is unique to NLB -- registering an ALB as a target, which is a pattern that appears in AWS exam questions and real architectures; the use case is combining NLB's static IP addresses and AWS PrivateLink support with ALB's Layer 7 routing capabilities: for example, you need to expose a microservices application via PrivateLink (which only works with NLB, as you learned in your VPC Peering study on Feb 20) but your application needs path-based routing that only ALB provides, so you put NLB in front of ALB; key constraints: only one ALB per target group, both must be in the same VPC and account, the ALB-type target group protocol must be TCP with port matching the ALB listener port, each registered ALB reduces the maximum targets per AZ by 50, and an ALB can be a target of at most 2 NLBs; this NLB-in-front-of-ALB pattern is also how you get a static IP address for an ALB (ALB IPs change, NLB IPs are static per AZ)

### 7. What is a Gateway Load Balancer? -- Official Introduction ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/elasticloadbalancing/latest/gateway/introduction.html
- **Type:** Official Docs
- **Summary:** The foundational page for understanding GLB's unique architecture that is fundamentally different from ALB and NLB: GLB operates at Layer 3 as a transparent network gateway combined with a load balancer, listening for ALL IP packets across ALL ports (not specific listeners like ALB/NLB), and forwarding traffic to target appliances using GENEVE encapsulation on port 6081 which preserves the original packet headers so appliances see real source/destination IPs; the architecture uses Gateway Load Balancer Endpoints (GWLBe), a special VPC endpoint type powered by PrivateLink, that act as route table next-hops in the service consumer VPC -- traffic flows from the consumer VPC through GWLBe to the GLB in the security/appliance VPC, then to the appliance fleet, then back through the same path; this connects directly to your Network Firewall study (Mar 10) where you learned about centralized inspection -- GLB solves the same problem but for third-party appliances (Palo Alto, Fortinet, Check Point) rather than AWS-native Network Firewall; key requirement: GWLBe and application servers must be in different subnets to enable the route table next-hop pattern

### 8. Centralized Inspection with GLB and Transit Gateway -- AWS Whitepaper ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/using-gwlb-with-tg-for-cns.html
- **Type:** Official Docs (Whitepaper)
- **Summary:** This whitepaper section brings GLB together with the Transit Gateway architecture you studied on Feb 23, showing the complete centralized inspection pattern for multi-VPC environments: a Security VPC contains appliance subnets (one per AZ) with GLB endpoints and virtual appliances plus Transit Gateway attachment subnets, while spoke VPCs route their default traffic through Transit Gateway to the Security VPC where GLB distributes it across the appliance fleet for inspection before forwarding to the destination; the critical detail is that Transit Gateway appliance mode must be enabled (which you already learned ensures return traffic goes through the same AZ as the original flow for stateful inspection), and GLB defaults to AZ-affinity for cross-zone behavior; this is the enterprise-grade pattern that combines your Transit Gateway knowledge with GLB to create a scalable, multi-VPC inspection architecture -- exactly the kind of architecture question that comes up in cloud engineer interviews when asked "how would you implement centralized network security across 50 VPCs?"

## Study Tips

- Build a three-column comparison table as you read, with ALB/NLB/GLB as columns and these rows: OSI layer, protocols, target types, routing algorithm, cross-zone default, static IP support, connection termination behavior, PrivateLink support, and primary use cases. This table will be your interview cheat sheet and forces you to internalize the differences rather than just reading about them. The single most important row is the proxy behavior difference: ALB terminates and re-establishes connections (enabling header inspection, SSL offload, sticky sessions), NLB passes connections through (preserving source IP, enabling ultra-low latency), and GLB transparently encapsulates packets (preserving everything for appliance inspection).
- Connect load balancers to services you already know by thinking about where each type appears in a real architecture: CloudFront uses ALB as an origin (your Mar 13 study), Auto Scaling groups register targets into ALB/NLB target groups (your Feb 12 study), ECS services create and manage ALB target groups automatically (your Feb 16-17 ECS studies), VPC PrivateLink only works with NLB or GLB not ALB (your Feb 20 study), and AWS Network Firewall is the AWS-managed alternative to GLB + third-party appliances (your Mar 10 study). Drawing these connections solidifies your architectural thinking.
- For health check configuration, remember the interview-critical defaults and their rationale: unhealthy threshold (2) is deliberately lower than healthy threshold (5) because fast failure detection matters more than fast recovery detection (you want to stop sending traffic to a failing target quickly, but you want to be confident a recovering target is truly stable before sending it production traffic). When designing health checks, always use a dedicated /health endpoint that checks downstream dependencies (database connectivity, cache availability) rather than just returning 200 -- a target that responds to HTTP but cannot reach its database is effectively unhealthy and should be marked as such.
