## Overview

This study plan covers the storage-layer mechanics of disaster recovery on AWS -- the deep operational details of S3 replication (CRR/SRR) failure scenarios and what does versus does not replicate, Replication Time Control for compliance-grade SLAs, Batch Replication for backfilling existing objects, DynamoDB Point-in-Time Recovery with the new configurable retention periods, DynamoDB export/import to S3 for cross-account DR, and DynamoDB Global Tables MRSC strong consistency constraints and trade-offs. After completing this plan, you will be able to design storage-tier DR architectures with precise knowledge of replication edge cases, recovery procedures, monitoring strategies, and the critical gotchas that separate exam-ready understanding from surface-level familiarity.

## Resources

### 1. What Does Amazon S3 Replicate? -- Official Documentation ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication-what-is-isnot-replicated.html
- **Type:** Official Docs
- **Summary:** The single most important page for understanding S3 replication edge cases that your Mar 23 study only touched at a high level. This page is the definitive reference for exactly what replicates and what does not -- and the "does not" list is where exam questions and production surprises live. Critical details to internalize: (1) existing objects before replication was enabled do NOT replicate (requires Batch Replication), (2) objects encrypted with SSE-KMS require explicit opt-in with the KMS key ARN in the replication config, (3) delete markers are NOT replicated by default in V2 replication configs (Filter element) but CAN be enabled via DeleteMarkerReplication, (4) version ID deletes (permanent deletions of specific versions) are NEVER replicated to the destination -- this is an intentional protection against malicious cross-region deletion, (5) lifecycle actions do NOT replicate -- you need independent lifecycle policies on each bucket, (6) objects in Glacier/Deep Archive storage classes cannot be replicated until restored, (7) replicas created by another replication rule do NOT cascade (A->B->C chaining does not work), and (8) bucket-level subresources (lifecycle configs, notification configs, etc.) are never replicated. Also read the linked "Requirements and considerations" page at https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication-requirements.html for the full prerequisites list (versioning on both buckets, IAM role permissions, same-owner or cross-account grant requirements). This page alone answers the majority of S3 replication interview questions.

### 2. S3 Replication Time Control and Replication Metrics -- Official Documentation ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication-time-control.html
- **Type:** Official Docs
- **Summary:** Builds on your Mar 23 conceptual understanding of RTC with the operational details that matter for DR design and compliance auditing. S3 RTC provides an SLA-backed guarantee that 99.99% of objects replicate within 15 minutes -- the only value you can set for both ReplicationTime and EventThreshold (always 15 minutes, no customization). Key operational details: RTC automatically enables CloudWatch replication metrics (ReplicationLatency as the maximum seconds behind, BytesPendingReplication, OperationsPendingReplication, and OperationsFailedReplication), plus two S3 event notifications (OperationMissedThreshold when an object fails to replicate within 15 minutes, and OperationReplicatedAfterThreshold when it catches up). Critical cost and capacity considerations: the default data transfer quota is 1 Gbps (requestable increase via Support), the SLA does NOT apply when you exceed request rate guidelines or transfer quotas, and you are charged for CloudWatch custom metrics plus one PUT request per replicated object on the destination. After this page, skim the companion "Using S3 Replication metrics" page at https://docs.aws.amazon.com/AmazonS3/latest/userguide/repl-metrics.html for the CloudWatch dashboard setup and how to view replication progress per rule. Understanding RTC monitoring is essential for designing DR architectures where you need to prove RPO compliance to auditors.

### 3. S3 Batch Replication -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-batch-replication-batch.html
- **Type:** Official Docs
- **Summary:** Covers the operational gap that live replication leaves: existing objects, previously failed replications, and newly added destinations. S3 Batch Replication uses S3 Batch Operations under the hood to retroactively replicate objects that live replication does not cover. Three scenarios where Batch Replication is required: (1) objects created before the replication rule was enabled, (2) objects that failed to replicate (status FAILED), and (3) objects already replicated to existing destinations when you add a new destination to the replication config (status REPLICA). Key mechanics: the source bucket must have an existing replication configuration, Batch Replication generates a manifest automatically from the source bucket inventory, you can filter by object creation date and replication status (NONE, FAILED, REPLICA) to scope the job, and the IAM role needs both replication permissions and Batch Operations permissions. Critical DR implication: when you add a new DR region by adding a replication destination, live replication only handles new writes -- you MUST run a Batch Replication job to bring existing data into the new DR region, and during this backfill window your RPO is effectively unbounded for existing objects. Also read the "Create a Batch Replication job for new replication rules" page at https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-batch-replication-new-config.html for the workflow when setting up replication for the first time. Disable lifecycle rules during Batch Replication to prevent source/destination divergence.

### 4. DynamoDB Point-in-Time Recovery -- Official Documentation + AWS Database Blog ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/PointInTimeRecovery_Howitworks.html
- **Type:** Official Docs
- **Summary:** Your Mar 26 DynamoDB study did not cover PITR in depth -- today's the day to internalize it as a critical DR mechanism. PITR provides continuous, automatic backups with per-second granularity enabling restore to any point between EarliestRestorableDateTime and LatestRestorableDateTime (typically 5 minutes before current time). The January 2025 update introduced configurable recovery periods from 1 to 35 days per table (previously fixed at 35 days) -- important for compliance scenarios requiring shorter retention windows. Critical operational details: (1) restore ALWAYS creates a new table (not in-place -- same constraint as RDS PITR that you studied Apr 2, requiring application-level cutover to the new table), (2) pricing is based on table size including LSIs regardless of recovery period length (setting 1 day vs 35 days costs the same), (3) when you decrease the recovery period, older backup data is immediately truncated and unrecoverable, (4) when a PITR-enabled table is deleted, DynamoDB automatically creates a system backup retained for 35 days at no cost -- a safety net against accidental table deletion, (5) restored tables do not automatically inherit DynamoDB Streams settings, auto-scaling policies, IAM policies, CloudWatch alarms, or tags -- you must reconfigure these manually or via IaC after restore. After the docs page, read the AWS Database Blog announcement at https://aws.amazon.com/blogs/database/announcing-configurable-point-in-time-recovery-periods-for-amazon-dynamodb/ for practical guidance on choosing the right recovery period. Compare mentally with RDS PITR (snapshot + transaction log replay with variable RTO) vs DynamoDB PITR (continuous backups with per-second granularity but always-new-table constraint).

### 5. DynamoDB Multi-Region Strong Consistency (MRSC) -- Official Documentation ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/multi-region-strong-consistency-gt.html
- **Type:** Official Docs
- **Summary:** Your Mar 26 and Mar 31 studies covered MRSC at a conceptual level -- today you go deep into the constraints that determine whether MRSC is viable for a given architecture. MRSC uses synchronous replication (write completes only after data reaches at least one other region), delivering RPO=0 and strongly consistent reads from any replica -- the only DynamoDB option that provides CP behavior in CAP terms. Deployment constraints: exactly three regions required (three full replicas OR two replicas plus one witness), restricted to predefined region sets (US: Virginia/Ohio/Oregon, EU: Ireland/London/Paris/Frankfurt, AP: Tokyo/Seoul/Osaka) -- you CANNOT mix regions across sets, which is a hard architectural constraint for global applications. Witness architecture: a lightweight component in a third region that stores data to satisfy the quorum but does not support read/write operations and does not appear in your AWS account. Critical limitations that eliminate MRSC for many workloads: no Local Secondary Indexes, no TTL, no transactions (TransactWriteItems/TransactGetItems return errors), cannot change consistency mode after creation, cannot add replicas to an existing MRSC table, source table must be empty when converting to MRSC, and write conflicts produce ReplicatedWriteConflictException (retry-safe but adds application complexity). The MREC vs MRSC comparison table on this page is essential -- memorize it. MRSC is a narrow but powerful tool: use it only when you genuinely need cross-region strong consistency with RPO=0 and your workload tolerates the latency and feature restrictions.

### 6. DynamoDB Export/Import with S3 -- Official Documentation ⏱️ 10 min
- **URL:** https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/S3forDynamoDB.html
- **Type:** Official Docs
- **Summary:** Covers the DynamoDB-to-S3 export and S3-to-DynamoDB import features that serve as the "Backup and Restore" tier of DynamoDB DR -- complementing Global Tables (active replication) and PITR (continuous point-in-time). Export to S3 requires PITR to be enabled, consumes zero RCUs (reads from PITR continuous backups, not the live table), supports full exports (entire table at a point in time) and incremental exports (only changed data, minimum 10 MB charge), outputs in DynamoDB JSON or Amazon Ion format, and can export to S3 buckets in different accounts and regions -- enabling cross-account DR patterns. Import from S3 creates a new DynamoDB table (not in-place -- matching the PITR constraint), accepts CSV, DynamoDB JSON, or Ion format with optional GZIP/ZSTD compression, and consumes zero WCUs. DR pattern: export daily to a cross-region S3 bucket as a cold backup tier alongside live Global Table replication, providing defense against data corruption that propagates across replicas (the "replication-is-not-backup" principle from your Mar 30 study). After the overview page, read the export mechanics at https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/S3DataExport.HowItWorks.html and import mechanics at https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/S3DataImport.HowItWorks.html for format and quota details.

### 7. Designing a Resilient and Cost-Effective Backup Strategy for Amazon S3 -- AWS Storage Blog ⏱️ 15 min
- **URL:** https://aws.amazon.com/blogs/storage/designing-a-resilient-and-cost-effective-backup-strategy-for-amazon-s3/
- **Type:** Blog (AWS Official)
- **Summary:** Bridges the gap between understanding individual S3 features and composing them into a coherent data protection strategy. This AWS Storage Blog post compares two complementary approaches: AWS Backup for S3 (centralized policy management with cross-region and cross-account copy, Vault Lock for immutability, restore testing -- connecting to your Apr 1 AWS Backup study) versus S3 Cross-Region Replication (continuous, near-real-time replication for low-RPO DR). The key insight is that these are not competing strategies but complementary layers: CRR provides low-RPO operational recovery (seconds to minutes of data loss), while AWS Backup provides immutable point-in-time snapshots that protect against data corruption propagating through replication (the same "replication-is-not-backup" principle that applies to DynamoDB Global Tables). Also covers versioning as the foundation of both strategies (required for CRR, used by AWS Backup for point-in-time recovery), lifecycle policies for managing version accumulation costs, and the decision framework for when to use each approach based on RPO/RTO requirements and compliance needs. This is the "putting it all together" resource that connects your S3, AWS Backup, and DR strategy knowledge into a unified storage protection architecture.

## Study Tips

- Build a replication comparison matrix with these columns: feature, S3 CRR, S3 SRR, DynamoDB Global Tables MREC, DynamoDB Global Tables MRSC. For each, fill in: replication type (async vs sync), RPO, what triggers replication (new objects only vs all writes), what does NOT replicate (lifecycle actions, version ID deletes, TTL deletes that DO replicate for DynamoDB), conflict resolution mechanism (last-writer-wins for MREC, ReplicatedWriteConflictException for MRSC, not applicable for S3 since it is single-writer), monitoring metrics (S3 CloudWatch replication metrics vs DynamoDB ReplicationLatency), and region constraints (any region for S3/MREC vs predefined region sets for MRSC). This single matrix will anchor every storage DR discussion in interviews.

- Trace the failure and recovery sequence for each service. For S3: what happens when you enable CRR on a bucket with 1 million existing objects? (Answer: only new objects replicate; you must run Batch Replication for existing ones, and during the backfill your DR region is incomplete.) What happens when someone deletes a specific version in the source bucket? (Answer: the destination retains that version -- intentional protection.) For DynamoDB: what happens when you restore from PITR? (Answer: new table created, you must reconfigure Streams, auto-scaling, tags, alarms, then update application endpoints or DNS.) What happens when an MRSC write conflicts with a concurrent write in another region? (Answer: ReplicatedWriteConflictException -- application must retry.) Thinking through these failure scenarios is what separates operational readiness from theoretical knowledge.

- Connect the "replication-is-not-backup" principle across every service you have studied this week. S3 CRR replicates corrupted data just as fast as good data -- you need versioning plus either AWS Backup or MFA Delete as a corruption defense. DynamoDB Global Tables replicate corrupt writes to all regions within seconds -- you need PITR as a separate recovery mechanism. Aurora Global Database replicates at the storage layer, so application-level corruption propagates to all secondaries -- you need Aurora Backtrack (MySQL, 72-hour max) or PITR as a defense. Every replication mechanism in AWS protects against infrastructure failure (AZ/region outage) but NOT against logical errors (bad code, accidental deletes, ransomware). Your DR architecture must address both failure modes with separate mechanisms.
