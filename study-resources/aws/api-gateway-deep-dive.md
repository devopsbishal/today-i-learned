## Resources

### 1. Amazon API Gateway Concepts -- AWS Official Docs ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-basic-concept.html
- **Type:** Official Docs
- **Summary:** The foundational glossary page that defines every API Gateway concept you need to internalize before diving into specific features. Covers the three API types (REST API, HTTP API, WebSocket API), the complete request/response pipeline (method request, integration request, integration response, method response), deployment model (deployments as point-in-time snapshots, stages as logical lifecycle references like dev/prod/v2), three endpoint types (edge-optimized routed through CloudFront, regional for same-region clients, private accessible only via VPC endpoints), integration types (proxy passing requests directly to backends, mock generating responses without a backend, private integrations via VPC Link), data transformation via mapping templates using Velocity Template Language (VTL), models for request/response validation, usage plans for per-client throttling and quotas, and API keys as developer identifiers (explicitly NOT for authentication). This page establishes the vocabulary and mental model that every subsequent resource builds upon -- read it carefully and note the four-step request pipeline (method request -> integration request -> integration response -> method response) as this is the core architectural concept that distinguishes REST API non-proxy integrations from the simpler proxy pattern.

### 2. Choose Between REST APIs and HTTP APIs -- AWS Official Docs ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/apigateway/latest/developerguide/http-api-vs-rest.html
- **Type:** Official Docs
- **Summary:** The definitive comparison table that drives the REST API vs HTTP API architectural decision -- one of the most frequently asked API Gateway interview questions. REST APIs ($3.50/million requests) offer the full feature set: API keys, per-client throttling via usage plans, request validation, AWS WAF integration, private API endpoints via VPC endpoints, response caching at the gateway layer, canary release deployments, mapping templates with VTL for request/response transformation, mock integrations, edge-optimized endpoints, and execution logs to CloudWatch. HTTP APIs ($1.00/million requests, up to 71% cheaper) sacrifice these features for lower cost and lower latency (~10ms overhead at p99 vs tens of milliseconds for REST APIs) but add native JWT authorization with any OpenID Connect provider (Cognito, Auth0, Okta) without needing a Lambda authorizer. The decision framework: use HTTP API for new serverless REST APIs when you need low cost, low latency, and JWT-based auth; use REST API when you need caching, WAF, API keys/usage plans, request validation, VTL transformations, private endpoints, or canary deployments. After reading the comparison table, also note that WebSocket APIs are a separate product entirely (not a variant of REST or HTTP API) with their own pricing model ($1.00/million messages + $0.25/million connection minutes). You studied Lambda Function URLs yesterday -- combine that knowledge with this comparison to form a complete three-option decision matrix: Function URLs (free, zero-feature, webhook/internal use), HTTP API (cheap, JWT auth, simple routing), REST API (full-featured, enterprise patterns).

### 3. API Gateway Integration Types and Lambda Proxy vs Non-Proxy -- AWS Official Docs ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-api-integration-types.html
- **Type:** Official Docs
- **Summary:** The core mechanics page explaining how API Gateway connects to backends -- five integration types that determine how much transformation happens at the gateway layer versus in your code. (1) AWS_PROXY (Lambda proxy): the recommended default for Lambda -- API Gateway passes the entire HTTP request (headers, query strings, path parameters, body, request context) as a structured JSON event to Lambda, and Lambda returns a response object with statusCode, headers, and body. Zero configuration, no mapping templates, no integration request/response setup. The trade-off: all transformation logic lives in your function code. (2) AWS (Lambda non-proxy / AWS service): you configure integration request mapping templates to transform the incoming request into the format your backend expects, and integration response mapping templates to transform the backend response before returning it to the client. This is how you build direct AWS service integrations (e.g., API Gateway -> DynamoDB PutItem without Lambda in between, or API Gateway -> SQS SendMessage, or API Gateway -> Step Functions StartExecution). (3) HTTP_PROXY: passes requests straight to an HTTP backend with no transformation. (4) HTTP: like HTTP_PROXY but with mapping templates for transformation. (5) MOCK: returns hardcoded responses without calling any backend, useful for CORS preflight responses, health checks, or API prototyping. After reading this page, also read the Lambda proxy integration page at https://docs.aws.amazon.com/apigateway/latest/developerguide/set-up-lambda-proxy-integrations.html to understand the exact event format Lambda receives and the response format it must return -- this is the integration you will use 90% of the time and it directly connects to the Lambda invocation models you studied yesterday.

### 5. Throttling, Caching, and Canary Deployments -- AWS Official Docs ⏱️ 20 min
- **URLs:**
  - Throttling: https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-request-throttling.html
  - Caching: https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-caching.html
  - Canary releases: https://docs.aws.amazon.com/apigateway/latest/developerguide/canary-release.html
- **Type:** Official Docs (three pages, ~7 min each)
- **Summary:** Three interconnected stage-level features that control traffic, performance, and deployment safety. **Throttling** uses the token bucket algorithm with a four-tier hierarchy applied in order: (1) per-client per-method limits in usage plans, (2) per-method limits in stage settings, (3) account-level limits per Region (default 10,000 requests/second with 5,000 burst, requestable increase), (4) AWS Regional limits. Understanding this hierarchy is critical -- a per-method throttle of 100 RPS cannot exceed the account-level limit, and usage plan throttles for individual API keys take highest priority. Throttled requests return 429 Too Many Requests. **Caching** is per-stage with configurable cluster sizes (0.5 GB to 237 GB), default TTL of 300 seconds (max 3,600), and only GET methods cached by default. Key operational details: cache keys are formed from method/integration parameters (query strings, headers, path variables), cache creation/deletion takes ~4 minutes, changing capacity deletes all cached data, clients can invalidate entries with Cache-Control: max-age=0 header (requires execute-api:InvalidateCache IAM permission), and caching is best-effort monitored via CacheHitCount/CacheMissCount CloudWatch metrics. Caching is charged hourly by size and is NOT in the free tier. **Canary releases** split traffic at the stage level between the current deployment (production) and a new deployment (canary) with configurable percentage, separate CloudWatch log groups for monitoring (API-Gateway-Execution-Logs/{api-id}/{stage}/Canary), and optional stage variable overrides for canary-specific configuration. This is complementary to Lambda weighted alias routing you studied yesterday -- API Gateway canary tests the entire API deployment (routes, integrations, authorizers), while Lambda alias routing tests individual function code.

### 6. API Gateway Authorizers: Lambda Authorizers and Cognito User Pools -- AWS Official Docs ⏱️ 15 min
- **URLs:**
  - Lambda authorizers: https://docs.aws.amazon.com/apigateway/latest/developerguide/apigateway-use-lambda-authorizer.html
  - Cognito user pools: https://docs.aws.amazon.com/apigateway/latest/developerguide/apigateway-integrate-with-cognito.html
- **Type:** Official Docs (two pages, ~8 min each)
- **Summary:** The two primary authorization mechanisms for REST APIs (plus IAM authorization as a third option). **Lambda authorizers** (formerly "custom authorizers") invoke a Lambda function to authenticate callers and return an IAM policy document that API Gateway caches for up to 1 hour (configurable, default 300 seconds). Two types: TOKEN-based (receives a bearer token from a specified header, e.g., Authorization) and REQUEST-based (receives the entire request context including headers, query strings, path parameters, stage variables, and request context -- more flexible, recommended for new implementations). The Lambda function returns a policy document with Allow/Deny and optionally a context map of key-value pairs passed to the integration (useful for injecting user metadata into downstream Lambda functions without re-parsing the token). Caching is keyed by the identity source (token value or specified request parameters) -- understand this because misconfigured caching can return another user's authorization policy. **Cognito user pool authorizers** validate JWT tokens from Amazon Cognito directly at the API Gateway layer without invoking a Lambda function (lower latency, no Lambda cost). You configure the user pool ID and token source header, and API Gateway validates the token signature, expiration, and optionally required OAuth scopes. The trade-off: Cognito authorizers only validate tokens (authentication), while Lambda authorizers can implement arbitrary authorization logic (e.g., check database permissions, enforce attribute-based access control). Decision framework: use Cognito authorizers when you use Cognito and only need token validation; use Lambda authorizers when you need custom authorization logic, non-Cognito identity providers, or want to inject context into downstream integrations; use IAM authorization for AWS service-to-service calls or when callers have IAM credentials.

### 7. WebSocket APIs in API Gateway -- AWS Official Docs ⏱️ 10 min
- **URL:** https://docs.aws.amazon.com/apigateway/latest/developerguide/apigateway-websocket-api-overview.html
- **Type:** Official Docs
- **Summary:** WebSocket APIs enable bidirectional, stateful communication between clients and backends -- fundamentally different from REST/HTTP APIs' request-response model. Key concepts: (1) **Routes** direct incoming messages to backend integrations based on a routeSelectionExpression evaluated against the message body (e.g., ${request.body.action} routes messages with {"action": "sendMessage"} to the sendMessage route). Three special routes: $connect (triggered when client initiates connection, use for authentication/connection registration in DynamoDB), $disconnect (triggered on disconnect, use for cleanup), $default (fallback when no route matches). (2) **Connection management**: clients have a 10-minute idle timeout and 2-hour maximum connection lifetime. The @connections API allows backends to push messages to specific connected clients by connection ID (POST to https://{api-id}.execute-api.{region}.amazonaws.com/{stage}/@connections/{connection-id}). (3) **Integration pattern**: $connect route typically invokes Lambda to store the connection ID in DynamoDB, custom routes invoke Lambda to process messages, and Lambda uses the @connections API to push responses to specific or all connected clients. Common use cases: chat applications, real-time dashboards, multiplayer games, collaborative editing. WebSocket APIs are priced differently: $1.00/million messages plus $0.25/million connection minutes, making them cost-effective for high-frequency bidirectional communication compared to polling REST APIs.

### 8. When to Use API Gateway vs Lambda Function URLs -- Yan Cui (The Burning Monk) ⏱️ 5 min
- **URL:** https://theburningmonk.com/2024/03/when-to-use-api-gateway-vs-lambda-function-urls/
- **Type:** Blog (Expert Community)
- **Summary:** A concise, practical decision guide from one of the most respected serverless practitioners (published March 2024, updated November 2024) that builds directly on the Function URLs knowledge from yesterday's Lambda study. Yan Cui cuts through the marketing to provide a clear decision framework: Function URLs are ideal when cost and raw performance are primary concerns (zero latency overhead, zero additional cost), you are building a "Lambdalith" (monolithic Lambda serving an entire API), or you need to handle large payloads or long-running requests. API Gateway is preferred when you need flexibility through its feature set -- direct AWS service integrations (skip Lambda entirely), multiple authentication methods, usage plans, request validation, or WebSocket support. The key insight: "Because of its flexibility, I prefer API Gateway" for most production APIs, but Function URLs have clear wins for private webhooks, internal microservice-to-microservice calls, and simple backends where API Gateway features add cost without value. Combine this with your Lambda Function URLs comparison table from yesterday and the REST vs HTTP API comparison from Resource 2 to form a complete four-option decision matrix: Function URLs -> HTTP API -> REST API -> ALB, ordered by increasing features and cost.

## Study Tips

- Build a four-column decision matrix comparing Function URLs, HTTP API, REST API, and ALB across these rows: cost per million requests, latency overhead, authentication options, throttling granularity, caching, WAF support, WebSocket support, request validation, private endpoint support, and maximum payload size. This single artifact captures the most commonly asked serverless architecture interview question and forces you to internalize the trade-offs rather than memorizing feature lists. Your Lambda doc from yesterday already has the Function URLs vs API Gateway vs ALB comparison table -- extend it with HTTP API as a separate column.

- Map the API Gateway request lifecycle end-to-end for a REST API with Lambda proxy integration: client request -> CloudFront (if edge-optimized) -> API Gateway endpoint -> resource policy evaluation -> method request (request validation) -> authorizer (Lambda/Cognito/IAM, cached) -> throttle check (usage plan -> method -> stage -> account) -> integration request (mapping template if non-proxy, passthrough if proxy) -> Lambda invocation (synchronous, from yesterday's study) -> integration response -> method response -> cache storage (if GET and caching enabled) -> client response. Drawing this pipeline once will clarify where each feature (throttling, caching, authorization, transformation) sits in the request flow.

- Connect API Gateway canary deployments to Lambda weighted alias routing you studied yesterday. They operate at different levels: API Gateway canary tests the entire API configuration (new routes, changed authorizers, modified integrations) while Lambda alias routing tests new function code behind the same API configuration. In production, you might use both: deploy a new Lambda version behind a weighted alias (90/10), then later deploy an API Gateway canary when changing the API structure itself. Understanding this layered deployment model is critical for system design interviews.
