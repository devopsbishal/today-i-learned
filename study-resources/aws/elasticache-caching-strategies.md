# ElastiCache & Caching Strategies -- 2-Hour Study Plan

**Total Time:** ~120 minutes
**Created:** 2026-03-27
**Difficulty:** Intermediate-Advanced

## Overview

This study plan covers Amazon ElastiCache as a general-purpose in-memory caching layer: Redis (now Valkey) vs Memcached engine selection, the four core caching strategies (lazy loading/cache-aside, write-through, write-behind, and TTL-based expiration), Redis Cluster Mode Enabled vs Disabled for sharding and horizontal write scaling, ElastiCache Serverless with ECPU-based pricing, Global Datastore for cross-region replication, Redis data structures beyond simple key-value caching (sorted sets, pub/sub, streams), ElastiCache security (encryption at rest/in transit, AUTH, RBAC, IAM authentication), and cost optimization patterns including reserved nodes, Valkey engine savings, and data tiering. After completing this plan, you will be able to explain why ElastiCache sits as a general-purpose caching layer in front of any data source (RDS, Aurora, DynamoDB, APIs, computed results) while DAX is a DynamoDB-specific zero-code-change cache (building on your DAX deep dive from Mar 26 where you learned about item cache vs query cache, write-through behavior, and the eventually-consistent-only limitation), choose between Redis and Memcached based on data structure needs and persistence requirements, select the right caching strategy for different read/write patterns (connecting to your understanding of Aurora read replicas for read scaling and DynamoDB on-demand vs provisioned for capacity planning), design ElastiCache architectures with Cluster Mode Enabled for write-heavy workloads that need horizontal partitioning across up to 500 shards, and use Global Datastore for cross-region session stores and DR -- paralleling the Aurora Global Database (storage-level replication, sub-second lag) and DynamoDB Global Tables (active-active multi-region writes) patterns you already know.

## Resources

### 1. What Is Amazon ElastiCache? -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/WhatIs.html
- **Type:** Official Docs
- **Summary:** The foundational overview that establishes ElastiCache's two deployment models (serverless with a single endpoint and automatic scaling, vs node-based clusters with fine-grained control over instance types, shards, and replicas) and three supported engines (Valkey 7.2+, Redis OSS 7.1+, Memcached 1.6.22+) -- read this first to understand ElastiCache's positioning as a fully managed in-memory data store that eliminates the operational burden of deploying and maintaining Redis or Memcached yourself (patching, monitoring, backups, failover); after this page, immediately read the linked comparison page at https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/SelectEngine.html which provides a detailed feature table comparing Memcached vs Redis/Valkey across multi-threading, data partitioning, pub/sub, sorted sets, replication, snapshots, encryption, data tiering, and online resharding -- the key decision framework is: choose Memcached only if you need the simplest possible model with multi-threaded architecture for pure key-value caching, choose Redis/Valkey for everything else (data structures, persistence, replication, pub/sub, sorted sets, cluster mode, Global Datastore); note the Valkey transition -- AWS now defaults to Valkey (the Linux Foundation's BSD-licensed fork of Redis 7.2.4) after Redis 8.0 moved to AGPLv3, and Valkey is 20-33% cheaper on ElastiCache

### 2. Caching Strategies -- Official Documentation ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/Strategies.html
- **Type:** Official Docs
- **Summary:** The authoritative guide to the three AWS-documented caching strategies you must internalize for both production architecture and interviews: (1) **Lazy Loading (Cache-Aside)** -- application checks cache first, on miss fetches from database and populates cache, meaning only requested data is cached (no wasted memory) but every cache miss incurs a three-trip penalty (check cache, query DB, write to cache) and data can become stale since the cache is only updated on misses; (2) **Write-Through** -- application writes to cache and database simultaneously on every write, ensuring cache is always current but every write incurs two trips and the cache fills with data that may never be read; (3) **Adding TTL** -- the essential complement to both strategies, setting expiration times so stale data is eventually evicted; the page includes pseudocode examples for each pattern; the critical insight is that lazy loading and write-through are complementary, not competing -- use lazy loading as your baseline everywhere (catches cache misses on reads) and add write-through selectively for hot data that must always be fresh; connect this to your DAX knowledge from yesterday: DAX implements write-through natively (writes go to both DAX and DynamoDB) with TTL-based expiration on the item cache, but the query cache is NOT invalidated on writes -- this is exactly the stale-data problem that the TTL strategy addresses

### 3. Caching Best Practices -- AWS Best Practices Page ⏱️ 15 min
- **URL:** https://aws.amazon.com/caching/best-practices/
- **Type:** Official Docs
- **Summary:** A broader AWS perspective on caching that goes beyond ElastiCache-specific documentation to cover architectural patterns and anti-patterns: reinforces lazy caching and write-through with Python pseudocode examples, then introduces two critical concepts not covered in the ElastiCache docs -- the **Thundering Herd (Dog-Piling) problem** where a cache key expires and hundreds of concurrent requests simultaneously hit the database to repopulate it (mitigated by using short TTLs, pre-warming the cache, or implementing cache locks), and **eviction policies** explaining all eight Redis eviction strategies (volatile-lru, allkeys-lru, volatile-lfu, allkeys-lfu, volatile-random, allkeys-random, volatile-ttl, noeviction) with guidance on when each is appropriate -- the default volatile-lru evicts the least recently used keys that have a TTL set, which means keys without a TTL are never evicted even when memory is full; also covers the **Russian Doll Caching** pattern for nested objects; this page complements the previous resource by adding the failure modes and edge cases that production systems must handle

### 4. Replication: Cluster Mode Disabled vs. Enabled -- Official Documentation ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/Replication.Redis-RedisCluster.html
- **Type:** Official Docs
- **Summary:** The essential architecture reference for understanding ElastiCache Redis/Valkey cluster topology -- **Cluster Mode Disabled** gives you a single shard (one primary node + up to 5 read replicas), meaning all data lives on one node and you scale reads by adding replicas but writes are limited to one node's capacity; **Cluster Mode Enabled** partitions data across up to 500 shards (each with its own primary + up to 5 replicas), enabling horizontal write scaling and larger dataset sizes that exceed a single node's memory; includes a feature comparison table covering data partitioning, scaling mechanics, Multi-AZ support, snapshots, and online resharding; the key architectural parallel: Cluster Mode Disabled is like a single Aurora writer with read replicas (scale reads, not writes), while Cluster Mode Enabled is like DynamoDB's partition-based architecture (data distributed across shards by hash slot, analogous to DynamoDB's partition key hash routing); also read the linked page on shards at https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/Shards.html to understand how Redis uses 16,384 hash slots distributed across shards, and the nodes/shards page at https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/CacheNodes.NodeGroups.html for node type selection; the decision rule: use Cluster Mode Disabled for datasets that fit in a single node's memory with read-heavy patterns, use Cluster Mode Enabled when you need write scaling, larger datasets, or online resharding capability

### 5. ElastiCache Serverless GA Announcement -- AWS News Blog ⏱️ 15 min
- **URL:** https://aws.amazon.com/blogs/aws/amazon-elasticache-serverless-for-redis-and-memcached-now-generally-available/
- **Type:** Blog (AWS Official)
- **Summary:** The AWS News Blog announcement (November 2023) that introduced ElastiCache Serverless, explaining the architecture and pricing model -- Serverless abstracts away all cluster topology decisions (no choosing node types, shard counts, or replica counts) behind a single endpoint, scaling instantly based on workload with zero downtime; pricing is based on two dimensions: data storage (GB-hours, minimum 1 GB for Redis/Memcached or 100 MB for Valkey) and compute measured in ElastiCache Processing Units (ECPUs) where a simple GET/SET of up to 1 KB consumes 1 ECPU, with complex data structure operations consuming proportionally more; always encrypts data at rest and in transit (mandatory, not optional like node-based clusters); the parallel to your existing knowledge is direct: ElastiCache Serverless is to node-based ElastiCache what Aurora Serverless v2 is to provisioned Aurora and what DynamoDB on-demand is to provisioned DynamoDB -- you trade granular control and lower per-unit cost for zero capacity planning and instant scaling; use Serverless for variable/unpredictable cache workloads, new applications, and dev/test; use node-based for steady workloads where reserved node pricing (up to 55% savings) justifies the operational overhead

### 6. Global Datastore for Cross-Region Replication -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/Redis-Global-Datastore.html
- **Type:** Official Docs
- **Summary:** The overview of ElastiCache Global Datastore which replicates Redis/Valkey data from a primary cluster in one region to up to two secondary read-only clusters in different regions with sub-second replication lag -- the primary cluster accepts writes while secondary clusters serve low-latency local reads and act as DR targets; failover promotion of a secondary to primary typically completes in under a minute; requires Cluster Mode Enabled (you cannot use Global Datastore with Cluster Mode Disabled) and is available for node-based clusters only (not Serverless as of March 2026); compare this directly to the three cross-region replication patterns you already know: Aurora Global Database (storage-level replication, sub-second lag, single write region with optional write forwarding, up to 5 secondary regions), DynamoDB Global Tables (active-active writes in all regions with last-writer-wins, sub-second replication), and now ElastiCache Global Datastore (single write region, up to 2 secondary regions, sub-second lag) -- Global Datastore follows the Aurora model (passive secondaries) rather than the DynamoDB model (active-active); also read the companion blog post at https://aws.amazon.com/blogs/database/build-a-multi-region-session-store-with-amazon-elasticache-for-valkey-global-datastore/ for a practical architecture using Global Datastore for session management

### 7. Data Security in Amazon ElastiCache -- Official Documentation ⏱️ 10 min
- **URL:** https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/encryption.html
- **Type:** Official Docs
- **Summary:** The security overview page covering ElastiCache's four security layers: (1) **Encryption in transit** using TLS 1.2+ between clients and cache nodes and between nodes themselves (mandatory for Serverless, optional for node-based -- but cannot be added after creation, like RDS encryption); (2) **Encryption at rest** using AES-256 for on-disk data during sync and backup operations (also mandatory for Serverless, optional for node-based); (3) **Authentication** via three mechanisms: Redis/Valkey AUTH token (simple shared password), IAM authentication (Redis 7.0+, generates short-lived 15-minute tokens, no password management), and user/password with RBAC; (4) **Role-Based Access Control (RBAC)** using access strings that define per-user command and key permissions (e.g., allow user A to read keys matching "session:*" but deny FLUSHALL) -- RBAC requires Redis 6.0+; also follow the link to the RBAC page at https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/Clusters.RBAC.html for deeper detail on user groups and access string syntax; the security model parallels what you learned with RDS: VPC-based deployment, security groups for network access, encryption at creation time only (cannot add later without migration), and IAM integration for authentication

### 8. Redis Data Structures Overview -- Redis Official Documentation ⏱️ 10 min
- **URL:** https://redis.io/docs/latest/develop/data-types/
- **Type:** Official Docs (Redis)
- **Summary:** The Redis.io reference for all data types beyond simple string key-value pairs -- the reason Redis dominates over Memcached for anything beyond pure caching: **Strings** (the basic building block, also used for counters with INCR/DECR), **Lists** (ordered collections for message queues and activity feeds), **Sets** (unique collections for tagging and membership), **Sorted Sets** (scored members for leaderboards, rate limiters, and priority queues -- each member has a score enabling range queries like "top 10 players" or "events in the last hour"), **Hashes** (field-value maps for storing objects, like a user profile with name/email/age fields without serialization overhead), **Streams** (append-only log structures for event sourcing and message streaming, similar to Kafka topics but built into Redis with consumer groups), and **Pub/Sub** (fire-and-forget message broadcasting for real-time notifications); understanding these data structures is what separates "I know ElastiCache is a cache" from "I can design a real-time leaderboard, a rate limiter, and a session store using ElastiCache" -- in interviews, mentioning sorted sets for leaderboards or streams for event processing demonstrates depth beyond the caching strategies covered in resources 2-3

## Study Tips

- Build a comparison matrix of the three cross-region replication patterns you now know: **Aurora Global Database** (storage-level replication, sub-second lag, 1 write region + up to 5 read-only secondary regions, managed switchover for zero-data-loss planned failover, write forwarding available), **DynamoDB Global Tables** (active-active writes in every region, last-writer-wins conflict resolution with MREC or strong consistency with MRSC across 3 regions, 99.999% SLA), and **ElastiCache Global Datastore** (asynchronous replication, sub-second lag, 1 write region + up to 2 read-only secondary regions, promotion in under a minute, Cluster Mode Enabled required). Notice the pattern: Aurora and ElastiCache follow the "single writer with passive secondaries" model, while DynamoDB is the only AWS service offering true active-active multi-region writes with automatic conflict resolution. In a system design interview, you would typically use all three together (Aurora for relational data, DynamoDB for key-value access patterns, ElastiCache for ephemeral cached data and sessions) with each having its own cross-region replication strategy.

- For caching strategy selection, internalize this decision tree: (1) **Start with lazy loading everywhere** as your baseline -- it is safe, memory-efficient, and handles node failures gracefully because a new empty node simply starts accumulating cache entries on misses. (2) **Add write-through selectively** for data that must be fresh immediately after writes (shopping cart contents, user profile updates) -- this eliminates the stale-data window between a database write and the next cache miss. (3) **Always set TTLs** even on write-through data as a safety net against edge cases where the write-through fails silently. (4) **Consider write-behind (write-back)** only for write-heavy workloads where you can tolerate potential data loss -- the application writes to cache first, and a background process asynchronously flushes to the database, which is faster but risks losing unflushed writes if the cache node fails. The key interview trap: candidates often present lazy loading and write-through as an either/or choice, when in practice the best architectures combine both with TTL as the universal safety net -- exactly what DAX does natively with its write-through item cache plus TTL-based expiration.

- For the ElastiCache vs DAX decision, the rule is simple but the nuance matters: **DAX is for DynamoDB-only read acceleration with zero application code changes** (swap the SDK client, done); **ElastiCache is for everything else** (caching RDS/Aurora query results, API responses, computed values, session data, or when you need Redis data structures like sorted sets and streams). But there are two subtle scenarios where you would use ElastiCache even with DynamoDB: (1) when you need to cache aggregated results computed from multiple DynamoDB queries (DAX caches individual queries, not computed aggregations), and (2) when you need cross-service caching where the same cache serves data from both DynamoDB and Aurora in a hybrid architecture like the e-commerce production pattern from your DynamoDB deep dive. Also remember that DAX runs inside your VPC as a cluster of nodes (like ElastiCache node-based), while ElastiCache Serverless offers a fully managed option with zero capacity planning -- there is no "DAX Serverless" equivalent.
