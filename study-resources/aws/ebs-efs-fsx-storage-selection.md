# Block & File Storage (EBS, EFS, FSx) -- 2-Hour Study Plan

**Total Time:** ~125 minutes
**Created:** 2026-03-24
**Difficulty:** Intermediate-Advanced

## Overview

This study plan covers the three pillars of AWS block and file storage -- EBS for single-instance block volumes, EFS for shared POSIX file systems, and FSx for specialized managed file systems (Windows, Lustre, NetApp ONTAP, OpenZFS) -- with deep focus on volume type selection, performance tuning, cost trade-offs, and a cross-service decision framework. After completing this plan, you will be able to select the right EBS volume type based on IOPS vs throughput vs cost requirements, design EFS deployments with appropriate performance and throughput modes, explain when each FSx variant is the correct choice, and articulate a storage selection decision tree covering EBS vs EFS vs FSx vs S3 -- building on your EC2 compute knowledge (Feb 5-12) where EBS attaches to instances, your KMS encryption knowledge (Mar 9) for EBS encryption behavior, and your S3 Advanced deep dive (Mar 23) which completed the object storage tier of your architecture toolkit.

## Resources

### 1. Amazon EBS Volume Types -- Official Specifications Reference ⏱️ 25 min
- **URL:** https://docs.aws.amazon.com/ebs/latest/userguide/ebs-volume-types.html
- **Type:** Official Docs
- **Summary:** The definitive reference page for all six EBS volume types with full specification tables: gp3 (3,000 baseline IOPS and 125 MiB/s included free, independently scalable to 80,000 IOPS and 2,000 MiB/s, 1 GiB-64 TiB, the default choice for most workloads), gp2 (legacy, IOPS scales with volume size at 3 IOPS/GiB up to 16,000, burst credits for small volumes, 1 GiB-16 TiB -- understand why gp3 replaced it), io2 Block Express (all io2 volumes created after November 2023 are Block Express, sub-millisecond latency, up to 256,000 IOPS and 4,000 MiB/s, 99.999% durability which is 100x better than gp3, 4 GiB-64 TiB, supports Multi-Attach to 16 Nitro instances), st1 Throughput Optimized HDD (40 MiB/s per TiB baseline bursting to 250 MiB/s per TiB, 500 MiB/s max, cannot be a boot volume, ideal for MapReduce/Kafka/log processing), and sc1 Cold HDD (12 MiB/s per TiB baseline bursting to 80 MiB/s per TiB, 250 MiB/s max, cannot be a boot volume, lowest cost EBS option for infrequent sequential access); pay close attention to the IOPS-to-volume-size ratios, the critical distinction between IOPS-optimized SSD workloads and throughput-optimized HDD workloads, and the fact that HDD volumes cannot serve as boot volumes -- also follow the links to the gp3 and io2 detail pages for the provisioning formulas (gp3 allows 500 IOPS per GiB up to 80,000; io2 allows 1,000 IOPS per GiB up to 256,000)

### 2. EBS Features -- Snapshots, Multi-Attach, Elastic Volumes, Encryption ⏱️ 20 min
- **URL:** https://aws.amazon.com/ebs/features/
- **Type:** Official Docs
- **Summary:** The features overview page that covers the operational capabilities beyond volume types: EBS Snapshots (incremental backups stored in S3, only changed blocks are saved, Fast Snapshot Restore eliminates lazy-loading latency penalty for critical volumes, Snapshot Archive tier at 75% cost reduction for long-term retention, Data Lifecycle Manager for automated snapshot schedules with application-consistent snapshots via pre/post scripts), Multi-Attach (io2 Block Express only, up to 16 Nitro-based EC2 instances in the same AZ, requires cluster-aware filesystem like GFS2 to avoid corruption, enables shared-storage patterns without EFS), Elastic Volumes (dynamically resize, change type, or adjust IOPS on live volumes with zero downtime -- this is how you migrate from gp2 to gp3 without detaching), and Encryption (uses KMS keys, encrypts data at rest and in transit between instance and volume, cannot directly encrypt an existing unencrypted volume -- must snapshot then create encrypted volume from snapshot, default encryption setting applies account-wide per region); after this page, quickly read the RAID configuration page at https://docs.aws.amazon.com/ebs/latest/userguide/raid-config.html which explains why only RAID 0 is recommended on EBS (RAID 5/6 waste IOPS on parity writes, RAID 1 wastes bandwidth with no write benefit, RAID 0 gives additive IOPS and throughput but total data loss if any volume fails)

### 3. Amazon EFS Performance and Throughput Modes -- Official Specifications ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/efs/latest/ug/performance.html
- **Type:** Official Docs
- **Summary:** The comprehensive performance reference for EFS covering two performance modes and three throughput modes: General Purpose performance mode (default, lowest per-operation latency at ~1ms read / ~2.7ms write, supports up to 2.5 million read IOPS with Elastic throughput, recommended for all workloads) vs Max I/O performance mode (legacy, higher latency but historically needed for massive parallelism -- now deprecated for new workloads since General Purpose with Elastic throughput matches or exceeds its capabilities, not supported for One Zone file systems); Elastic throughput mode (default in console, automatically scales up to 60 GiBps read / 5 GiBps write, pay only for what you use, recommended for most workloads), Provisioned throughput mode (you specify a fixed throughput level independent of storage size, best when you know your exact requirements and want predictable billing), and Bursting throughput mode (throughput scales with stored data at 50 KiBps per GiB baseline with burst credits that allow 100 MiBps per TiB bursts, problematic for small file systems that exhaust burst credits quickly); the key insight is that Elastic throughput mode has fundamentally changed EFS economics -- you no longer need to over-provision storage just to get baseline throughput (the old Bursting problem), but you pay a premium per-MiBps compared to Provisioned mode for sustained high-throughput workloads

### 4. EFS Storage Classes and Lifecycle Management ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/efs/latest/ug/lifecycle-management-efs.html
- **Type:** Official Docs
- **Summary:** Covers EFS's four storage classes and automatic lifecycle tiering: EFS Standard (SSD-backed, lowest latency, highest cost), EFS Infrequent Access (low double-digit millisecond latency, per-access retrieval fee, up to 92% lower storage cost than Standard), EFS Archive (designed for data accessed a few times per year, lowest storage cost), and the One Zone variants of Standard and IA (47% cheaper than Regional equivalents but data exists in a single AZ -- if that AZ fails, data is unavailable or lost); lifecycle policies automatically transition files based on access recency (default: 30 days to IA, 90 days to Archive), with a critical constraint that files smaller than 128 KB are never transitioned and always remain in Standard/One Zone -- the same 128 KB threshold you encountered in S3 lifecycle policies yesterday; also read the storage classes overview at https://aws.amazon.com/efs/storage-classes/ for the cost comparison table showing Standard vs IA vs Archive pricing per region, and note that EFS Intelligent-Tiering (the combination of lifecycle policies with Standard + IA + Archive) is the EFS equivalent of S3 Intelligent-Tiering but implemented through lifecycle policies rather than a separate storage class

### 5. Amazon FSx -- Help Me Choose a File System ⏱️ 15 min
- **URL:** https://aws.amazon.com/fsx/when-to-choose-fsx/
- **Type:** Official Docs
- **Summary:** AWS's official decision guide for selecting among the four FSx file system types, organized by workload requirements: FSx for Windows File Server (SMB protocol, Active Directory integration, Windows ACLs, DFS Namespaces for scaling to hundreds of PB, shadow copies, data deduplication -- choose when you need Windows-native shared file storage with AD-based access control), FSx for Lustre (POSIX-compliant parallel file system, up to 1,000 GBps throughput, seamless S3 integration via data repository associations that lazily load S3 objects as filesystem metadata, scratch deployment for temporary HPC jobs vs persistent deployment for long-term workloads with replication within the AZ -- choose for HPC, ML training, media rendering, or any workload needing massive parallel throughput), FSx for NetApp ONTAP (multi-protocol NAS supporting NFS + SMB + iSCSI + NVMe simultaneously, automatic SSD-to-capacity-pool tiering, SnapMirror replication with 5-minute RPO, FlexClone for instant zero-copy clones, data deduplication and compression -- choose when migrating from on-premises NetApp or need multi-protocol access), and FSx for OpenZFS (NFS protocol, built-in compression, instant snapshots and clones, up to 1 million IOPS -- choose for Linux workloads migrating from on-premises ZFS or needing powerful snapshot/clone capabilities without the NetApp feature set); this single page provides the decision matrix you need for FSx interview questions

### 6. FSx for Lustre Deployment Types and S3 Integration ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/fsx/latest/LustreGuide/using-fsx-lustre.html
- **Type:** Official Docs
- **Summary:** Deep dive into FSx for Lustre's architecture covering the critical distinction between deployment types: Scratch file systems (temporary, no data replication, highest burst throughput at 200 MBps/TiB with 6x burst capability, data lost if a file server fails, ideal for short-lived HPC jobs where the source data lives in S3 and results are written back to S3), Persistent 2 file systems (latest generation, data replicated within the AZ, automatic failover to replacement servers within minutes, supports SSD and Intelligent-Tiering storage classes, choose for long-running workloads requiring data durability), and Persistent 1 (previous generation, supports SSD and HDD); also covers the three storage classes: SSD (sub-millisecond latency for IOPS-intensive workloads), HDD with optional SSD read cache (cost-optimized for throughput-intensive sequential workloads), and Intelligent-Tiering (elastic capacity that auto-tiers data based on 30/90-day access patterns, available only with Persistent 2); the S3 data repository association is the killer feature for HPC -- it lets you present an S3 bucket as a POSIX filesystem, lazily loading object metadata at creation and fetching object data on first access, then exporting results back to S3 via data repository tasks

### 7. FSx for NetApp ONTAP Overview -- Multi-Protocol NAS and Data Management ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/fsx/latest/ONTAPGuide/what-is-fsx-ontap.html
- **Type:** Official Docs
- **Summary:** The conceptual overview of FSx for NetApp ONTAP covering its unique position as the only AWS file service supporting four protocols simultaneously (NFS, SMB, iSCSI, NVMe over TCP), making it the only FSx option that serves both Linux and Windows clients from the same filesystem with the same data; key architectural concepts include the two-tier storage model (high-performance SSD primary storage plus fully elastic capacity pool storage that automatically tiers infrequently accessed data at a fraction of the cost -- similar to S3 Intelligent-Tiering), SnapMirror replication for cross-region DR or on-premises-to-cloud migration with RPO as low as 5 minutes, FlexClone for instant space-efficient clones (ideal for dev/test environments), data deduplication and compression that can reduce storage consumption by 65%+, SnapLock for WORM compliance (the NetApp equivalent of S3 Object Lock), and support for both Single-AZ and Multi-AZ deployments; this is the go-to choice when enterprises are migrating existing NetApp ONTAP infrastructure to AWS because applications require zero code changes -- they connect to FSx for ONTAP using the same NFS/SMB/iSCSI protocols they used on-premises

## Study Tips

- Build a three-column mental model for EBS volume selection: **IOPS-dominant workloads** (databases needing random read/write: gp3 for most, io2 for extreme requirements above 80K IOPS or 99.999% durability), **throughput-dominant workloads** (big data, log processing, streaming: st1 for frequently accessed, sc1 for cold data), and **boot volumes** (always SSD, gp3 default). The most common interview trap is confusing IOPS with throughput -- a database doing thousands of small random reads needs high IOPS (SSD), while a data warehouse scanning terabytes sequentially needs high throughput (HDD). Know the gp3 provisioning formula: 500 IOPS per GiB of volume size, minimum 3,000 free, and that gp3 decouples IOPS from volume size (unlike gp2 where you had to over-provision storage to get more IOPS). For io2, the ratio is 1,000 IOPS per GiB, and Multi-Attach is io2-only (not gp3).

- For EFS vs FSx selection, use this decision tree: need POSIX shared filesystem accessible from multiple AZs with automatic scaling? **EFS**. Need Windows-native SMB with Active Directory? **FSx for Windows**. Need massive parallel throughput for HPC/ML with S3 as the durable data layer? **FSx for Lustre**. Need multi-protocol NAS (NFS + SMB + iSCSI) or migrating from on-premises NetApp? **FSx for NetApp ONTAP**. Need Linux ZFS migration with snapshots and compression? **FSx for OpenZFS**. Need single-instance block storage? **EBS**. Need object storage for unstructured data at internet scale? **S3** (yesterday's topic). The broader storage decision always starts with the access pattern: block (EBS), file (EFS/FSx), or object (S3).

- Connect these storage services to your existing knowledge graph: EBS encryption uses the same KMS infrastructure you studied on Mar 9 (same envelope encryption, same key policies, same cross-account sharing constraints). EBS attaches to the EC2 instances and Nitro architecture you studied Feb 5-9 (Multi-Attach requires Nitro). FSx for Lustre's S3 integration bridges to yesterday's S3 deep dive (the data repository association presents S3 as a POSIX filesystem). EFS mount targets live in VPC subnets you designed on Feb 19, accessible via NFS over the security groups you configured. And EBS snapshots connect to the DR strategies you will study in Week 5 (cross-region snapshot copy enables Pilot Light and Warm Standby patterns for stateful workloads).
