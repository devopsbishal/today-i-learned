## Resources

### 1. Understanding the Lambda Execution Environment Lifecycle -- AWS Official Docs ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/lambda/latest/dg/lambda-runtime-environment.html
- **Type:** Official Docs
- **Summary:** The foundational document for understanding how Lambda actually works. Covers the three lifecycle phases (Init, Invoke, Shutdown) with precise timing constraints: Init is limited to 10 seconds for on-demand invocations (extended to 15 minutes for provisioned concurrency and SnapStart), the Invoke phase runs your handler within the configured timeout (up to 15 minutes), and Shutdown gives extensions up to 2 seconds for cleanup. Key details to internalize: (1) the Init phase has three sub-phases -- Extension init, Runtime init, and Function init (your static initialization code outside the handler) -- and this is where cold start latency lives, (2) Lambda freezes the execution environment between invocations and thaws it for reuse (warm starts), (3) the SnapStart Restore phase replaces Init by resuming from a cached memory/disk snapshot, (4) static initialization code runs only once per execution environment creation so SDK clients, database connections, and configuration loading should happen outside the handler, (5) `/tmp` persists across invocations within the same execution environment but is NOT guaranteed between different environments. Also covers the Runtime API and Extensions API communication model. This page establishes the mental model that every subsequent resource builds upon.

### 2. Understanding and Remediating Cold Starts: An AWS Lambda Perspective -- AWS Compute Blog ⏱️ 20 min
- **URL:** https://aws.amazon.com/blogs/compute/understanding-and-remediating-cold-starts-an-aws-lambda-perspective/
- **Type:** Blog (AWS Official)
- **Summary:** The definitive AWS blog post on cold starts -- goes beyond the docs to explain causes, measurement, and mitigation strategies with practical guidance. Covers what causes cold starts (new function deployment, scaling up to handle increased traffic, no available warm environments after idle period, function configuration change), the factors that affect cold start duration (runtime choice -- Python/Node.js fastest at ~200ms, Java/C# slowest at 1-3s+; deployment package size -- every MB adds latency; memory allocation -- more memory means proportionally more CPU which accelerates Init; VPC attachment -- historically 10+ seconds, now sub-100ms with Hyperplane), and the full mitigation toolkit: (1) Provisioned Concurrency for latency-critical synchronous workloads, (2) SnapStart for Java/Python/.NET, (3) minimize deployment package size by excluding dev dependencies and using Lambda Layers, (4) increase memory allocation for CPU-bound init code, (5) lazy-load heavy libraries only when needed, (6) choose lighter runtimes (Python/Node.js) when cold start matters more than raw execution speed. This post connects the lifecycle phases from Resource 1 to real-world performance engineering decisions.

### 3. Understanding Lambda Function Scaling (Concurrency) -- AWS Official Docs ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/lambda/latest/dg/lambda-concurrency.html
- **Type:** Official Docs
- **Summary:** The complete concurrency model explained with formulas and diagrams. Key formula: Concurrency = (requests per second) x (average duration in seconds). Covers the three concurrency tiers: (1) **Unreserved concurrency** -- the default pool shared across all functions in the account (default account limit: 1,000 concurrent executions per region, requestable increase), (2) **Reserved concurrency** -- sets BOTH the maximum AND minimum allocation for a specific function, guaranteeing capacity but also throttling the function if it exceeds the reservation, no additional cost, does NOT eliminate cold starts, (3) **Provisioned concurrency** -- pre-initializes a specified number of execution environments so they are always warm, eliminates cold starts for those environments, incurs additional charges, can be combined with Application Auto Scaling to adjust based on schedule or utilization. Also covers burst concurrency (initial burst of 500-3000 depending on region, then scales at 500 additional instances per minute), the requests-per-second limit (10x concurrency quota), throttling behavior (synchronous invocations return 429, asynchronous invocations retry then send to DLQ/destination), and how to calculate concurrency needs for your workload. After this page, also read the companion pages on configuring reserved concurrency (https://docs.aws.amazon.com/lambda/latest/dg/configuration-concurrency.html) and provisioned concurrency (https://docs.aws.amazon.com/lambda/latest/dg/provisioned-concurrency.html) for the operational setup details.

### 4. How Lambda Processes Records from Stream and Queue-Based Event Sources -- AWS Official Docs ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/lambda/latest/dg/invocation-eventsourcemapping.html
- **Type:** Official Docs
- **Summary:** Critical for understanding how Lambda integrates with SQS, Kinesis, and DynamoDB Streams -- the event sources you will use heavily this week and in production. Event source mappings are Lambda resources that poll the source service, batch records, and invoke your function. Key mechanics: (1) **Stream sources** (Kinesis, DynamoDB Streams) process records in order per shard, with configurable batch size (up to 10,000 for Kinesis, 10,000 for DynamoDB), batch window (up to 300 seconds to accumulate records), parallelization factor (up to 10 concurrent batches per shard), and error handling via BisectBatchOnFunctionError (splits failed batches in half recursively to isolate the poison record), MaximumRetryAttempts, MaximumRecordAgeInSeconds, and on-failure destinations (SQS, SNS, or S3 as of Nov 2024), (2) **Queue sources** (SQS) scale automatically based on queue depth, support batch sizes up to 10,000 messages, and use visibility timeout for error handling -- failed messages return to the queue and eventually hit the SQS dead-letter queue (configured on the SQS queue, not on Lambda), (3) **All event source mappings** deliver at-least-once semantics requiring idempotent function design. The BisectBatchOnFunctionError + MaximumRetryAttempts + on-failure destination combination is the production-grade error handling pattern for streams -- understand it thoroughly as it appears frequently in interview scenarios.

### 5. Improving Startup Performance with Lambda SnapStart -- AWS Official Docs ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/lambda/latest/dg/snapstart.html
- **Type:** Official Docs
- **Summary:** SnapStart is Lambda's snapshot-based cold start optimization that fundamentally changes the Init phase. Instead of running initialization code on every cold start, Lambda initializes your function once during deployment, captures a Firecracker snapshot of the memory and disk state, caches it, and restores from that snapshot on subsequent cold starts -- reducing init time by up to 90%. Supported runtimes: Java 11+, Python 3.12+, .NET 8+ (with different pricing -- Java is free, Python/.NET incur caching and restoration charges). Critical limitations to memorize: (1) does NOT work with provisioned concurrency (choose one or the other), (2) does NOT support Amazon EFS, (3) ephemeral storage limited to 512 MB (vs 10 GB normally), (4) only works on published versions and aliases (not $LATEST), (5) does NOT support container images. Operational considerations: runtime hooks (beforeCheckpoint/afterRestore for Java, before_snapshot/after_restore for Python) let you handle resources that should not be snapshotted -- database connections must be re-established in the after-restore hook, random number generators must be re-seeded for security, and temporary credentials must be refreshed. SnapStart vs Provisioned Concurrency decision: SnapStart is cheaper (especially for Java) and reduces cold starts significantly but does not eliminate them entirely; Provisioned Concurrency eliminates cold starts completely but costs more. Use SnapStart for cost-sensitive workloads tolerating sub-second cold starts; use Provisioned Concurrency for latency-critical APIs requiring consistent double-digit-millisecond response times.

### 6. Implementing AWS Lambda Error Handling Patterns -- AWS Compute Blog ⏱️ 15 min
- **URL:** https://aws.amazon.com/blogs/compute/implementing-aws-lambda-error-handling-patterns/
- **Type:** Blog (AWS Official)
- **Summary:** Ties together all the error handling mechanisms across Lambda's three invocation models into a unified mental framework -- critical for architectural interviews. Covers: (1) **Synchronous invocation** (API Gateway, ALB) -- caller receives error directly, no built-in retry, implement retry logic in the client or use API Gateway retry configuration, (2) **Asynchronous invocation** (S3 events, SNS, EventBridge) -- Lambda automatically retries twice with backoff (configurable 0-2 retries), then routes to either a DLQ (SQS or SNS, sends only the event payload) or a Destination (SQS, SNS, Lambda, or EventBridge, sends the full invocation record including function response, error details, and stack trace) -- Destinations supersede DLQs as the recommended approach because they support both success AND failure routing and include richer metadata, (3) **Event source mapping** (SQS, Kinesis, DynamoDB Streams) -- error handling differs by source type as covered in Resource 4. The key architectural insight: choose your error handling strategy based on invocation model, not as an afterthought. The blog includes architecture diagrams showing how to compose these patterns with Step Functions for retry orchestration, EventBridge for event routing, and SQS for buffering -- patterns you will study in depth later this week.

### 7. Lambda Canary Deployments Using Weighted Alias Traffic Shifting -- AWS Official Docs ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/lambda/latest/dg/configuring-alias-routing.html
- **Type:** Official Docs
- **Summary:** Covers Lambda's versioning and alias system that enables safe deployment strategies. Key concepts: (1) **Versions** are immutable snapshots of function code + configuration, each with a unique ARN (arn:aws:lambda:region:account:function:name:version), $LATEST is the only mutable version, (2) **Aliases** are named pointers to versions (e.g., prod -> version 5, staging -> version 6), enabling decoupled deployment from consumer configuration -- API Gateway and other triggers reference the alias ARN which never changes, (3) **Weighted aliases** split traffic between two versions (primary + secondary with configurable weight 0.0-1.0) enabling canary deployments -- route 5% to the new version, monitor CloudWatch error rates and latency, then shift to 100% or roll back, (4) **CodeDeploy integration** automates traffic shifting with pre-defined strategies: Canary10Percent5Minutes (shift 10%, wait 5 min, then 100%), Linear10PercentEvery1Minute (gradual 10% increments), AllAtOnce (immediate blue-green), with automatic rollback triggered by CloudWatch Alarms. Also covers pre-traffic and post-traffic hooks for validation Lambda functions that run before and after traffic shifting. This is the Lambda equivalent of the ECS blue-green deployment pattern you studied in February, using aliases instead of target groups. After this page, also read the Lambda versions documentation at https://docs.aws.amazon.com/lambda/latest/dg/configuration-versions.html for the versioning mechanics that underpin alias routing.

## Study Tips

- Build a cold start mitigation decision tree: Start with "Is this a synchronous, latency-sensitive workload?" If yes and budget allows, use Provisioned Concurrency. If yes and cost-sensitive, use SnapStart (Java/Python/.NET only). If neither applies (async processing, batch jobs), cold starts likely do not matter -- optimize for cost instead. For any path, also apply code-level optimizations: minimize package size, initialize SDK clients outside the handler, increase memory for CPU-bound init code, lazy-load conditional dependencies.

- Create a Lambda error handling matrix with three columns (Synchronous, Asynchronous, Event Source Mapping) and rows for: retry behavior, retry count, error routing mechanism, metadata richness, consumer responsibility, and recommended pattern. This single matrix captures the most commonly confused Lambda topic in interviews -- candidates frequently mix up DLQ behavior across invocation models (DLQ on the Lambda function for async vs DLQ on the SQS queue for event source mappings).

- Compare Lambda's concurrency model with ECS scaling from your Feb 16-17 studies. Lambda scales per-invocation (one execution environment per concurrent request) while ECS scales per-task (one container handles multiple requests). This difference drives the architectural choice: Lambda excels at bursty, unpredictable workloads with short duration; ECS excels at steady-state, long-running workloads with connection pooling. Reserved Concurrency is analogous to ECS service minimum/maximum task count, while Provisioned Concurrency has no direct ECS equivalent (ECS tasks are always "warm" once running).
