# DynamoDB Deep Dive -- 2-Hour Study Plan

**Total Time:** ~120 minutes
**Created:** 2026-03-26
**Difficulty:** Intermediate-Advanced

## Overview

This study plan covers Amazon DynamoDB from the ground up for someone transitioning from relational database knowledge (RDS/Aurora) to NoSQL thinking: partition keys and sort keys as the fundamental data access mechanism, Global Secondary Indexes (GSI) and Local Secondary Indexes (LSI) for alternative query patterns, On-Demand vs Provisioned capacity modes with auto-scaling and cost optimization, DynamoDB Accelerator (DAX) for microsecond-latency caching, DynamoDB Streams for change data capture and event-driven architectures, Global Tables for multi-region active-active replication, and single-table design as the paradigm shift from relational normalization. After completing this plan, you will be able to explain why DynamoDB requires you to design your schema around access patterns rather than entity relationships (the inverse of relational modeling), articulate the partition key hash-routing mechanism that enables unlimited horizontal scaling, choose between GSI and LSI based on creation-time constraints and query flexibility requirements, select the right capacity mode based on traffic predictability (connecting to your Aurora Serverless v2 knowledge where variable workloads also drove the serverless choice), understand how DAX differs from ElastiCache (DAX is DynamoDB-native and API-compatible vs ElastiCache being general-purpose), and design multi-region architectures using Global Tables with last-writer-wins conflict resolution -- building on your RDS/Aurora deep dive (Mar 25) where you learned relational HA patterns, your S3 advanced deep dive (Mar 23) where you studied cross-region replication and eventual consistency, and your CloudFront/Global Accelerator knowledge for understanding why low-latency global data access matters.

## Resources

### 1. Core Components of Amazon DynamoDB -- Official Documentation ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/HowItWorks.CoreComponents.html
- **Type:** Official Docs
- **Summary:** The essential starting point that establishes DynamoDB's four core building blocks: tables (schemaless collections where each item can have different attributes -- a fundamental shift from RDS where every row must conform to the table schema you defined), items (analogous to rows), attributes (analogous to columns but flexible per item, nested up to 32 levels), and primary keys -- covering both simple primary keys (partition key only, where DynamoDB hashes the value to determine which physical partition stores the item) and composite primary keys (partition key + sort key, where items sharing the same partition key are stored together and sorted by sort key, enabling efficient range queries with operators like begins_with, between, >, <); also introduces secondary indexes (GSI and LSI) and DynamoDB Streams at a conceptual level; read this page carefully because the partition key hash-routing mechanism is the single most important concept in DynamoDB -- it determines how data is distributed across partitions, which directly governs performance, scalability, and the dreaded "hot partition" problem; coming from RDS/Aurora where the storage layer handled distribution transparently (Aurora's 10 GB segments across 3 AZs), DynamoDB puts distribution responsibility squarely on your primary key design

### 2. Best Practices for Designing and Using Partition Keys Effectively -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/bp-partition-key-design.html
- **Type:** Official Docs
- **Summary:** The official best practices guide for partition key design that explains why uniform distribution of I/O across partitions is critical for DynamoDB performance -- a single partition supports up to 3,000 RCU and 1,000 WCU, and exceeding these limits on a "hot" partition causes throttling regardless of your table's total provisioned capacity; covers the high-cardinality principle (partition keys should have many distinct values like user IDs, device IDs, or session IDs), the write-sharding pattern (appending a random suffix or calculated suffix to spread writes across partitions for high-throughput scenarios like voting or IoT telemetry), and DynamoDB's adaptive capacity feature that automatically isolates frequently accessed items to dedicated partitions; also read the companion page on sort key best practices at https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/bp-sort-keys.html (5 min) which shows how composite sort keys like STATUS#DATE enable hierarchical queries -- this is a fundamentally different way of thinking about data organization compared to the relational WHERE clauses and JOINs you used with RDS/Aurora

### 3. General Guidelines for Secondary Indexes in DynamoDB -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/bp-indexes-general.html
- **Type:** Official Docs
- **Summary:** The authoritative guide to GSI and LSI design that covers the critical differences you must internalize: GSIs have their own partition key and sort key (can be completely different from the base table), can be created or deleted at any time, have their own provisioned throughput separate from the table, and have no size limits -- while LSIs share the base table's partition key (only the sort key differs), must be defined at table creation time (cannot be added later -- this is an irreversible design decision), share the base table's throughput, and enforce a 10 GB limit per partition key value (the "10 GB partition limit" that can silently break your application if item collections grow beyond this); also explains index projections (KEYS_ONLY, INCLUDE, ALL) and their cost implications -- projecting more attributes into an index increases storage costs and write costs (every write to the base table that affects projected attributes triggers an index write), but reduces the need for expensive fetches back to the base table; the practical rule: default to GSI for almost everything since LSIs are inflexible, and only use LSI when you specifically need strong consistency on the alternate sort key (GSIs only support eventually consistent reads)

### 4. Choose the Right Throughput Strategy for Amazon DynamoDB Applications -- AWS Database Blog ⏱️ 15 min
- **URL:** https://aws.amazon.com/blogs/database/choose-the-right-throughput-strategy-for-amazon-dynamodb-applications/
- **Type:** Blog (AWS Official)
- **Summary:** An AWS Database Blog post that provides a decision framework for choosing between DynamoDB's two capacity modes: On-Demand (pay-per-request, no capacity planning, automatically scales to handle spikes up to double the previous peak, ideal for unpredictable or new workloads -- analogous to how you might choose Aurora Serverless v2 for variable relational workloads) and Provisioned (you specify RCU/WCU, can use auto-scaling to adjust capacity based on utilization targets, 25-40% cheaper than on-demand for steady workloads, supports reserved capacity for 1- or 3-year commitments saving up to 75%); the key decision heuristic is the peak-to-average ratio -- if your peak traffic exceeds 4x your average, on-demand is likely cheaper because provisioned mode would require over-provisioning headroom; also covers the switching constraint (on-demand to provisioned unlimited, but provisioned to on-demand limited to 4 times per 24 hours) and the November 2024 on-demand pricing reduction that made on-demand more competitive; read the companion page on capacity mode considerations at https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/bp-switching-capacity-modes.html for switching caveats

### 5. DAX: How It Works -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DAX.concepts.html
- **Type:** Official Docs
- **Summary:** The official explanation of DynamoDB Accelerator's architecture and caching mechanics -- DAX is a fully managed, VPC-based in-memory cache that sits between your application and DynamoDB, using a DAX client SDK that is API-compatible with the standard DynamoDB SDK (meaning you swap the client library and your existing GetItem/Query/Scan calls automatically route through DAX with zero application logic changes); covers the two separate caches: the item cache (stores individual items from GetItem/BatchGetItem with a default 5-minute TTL and LRU eviction) and the query cache (stores full result sets from Query/Scan operations indexed by parameter values, also with configurable TTL); critically, DAX only caches eventually consistent reads -- strongly consistent reads pass through directly to DynamoDB, and write operations use a write-through pattern (written to both DynamoDB and DAX simultaneously); the key architectural decision: DAX runs inside your VPC as a cluster of nodes (primary + read replicas), making it ideal for read-heavy workloads with repeated access to the same items (hot keys), but it adds VPC dependency and is not suitable for write-heavy workloads or workloads requiring strongly consistent reads -- tomorrow you will study ElastiCache which provides general-purpose caching that can work with any data source, not just DynamoDB

### 6. Change Data Capture for DynamoDB Streams -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/Streams.html
- **Type:** Official Docs
- **Summary:** The comprehensive guide to DynamoDB Streams covering how DynamoDB captures a time-ordered, deduplicated sequence of item-level modifications (inserts, updates, deletes) and stores them for exactly 24 hours in a stream composed of shards with parent-child relationships; covers the four StreamViewType options that control what data each stream record contains: KEYS_ONLY (just the primary key -- lightest weight), NEW_IMAGE (complete item after modification), OLD_IMAGE (complete item before modification), and NEW_AND_OLD_IMAGES (both before and after -- most expensive but enables diff computation); the StreamViewType is set once when enabling the stream and cannot be changed (you must disable and re-enable); stream records are guaranteed to appear exactly once and in order per item (but not globally ordered across items); primary integration patterns include Lambda triggers (the most common pattern -- Lambda polls the stream automatically, and these read operations are free), Kinesis Client Library (KCL) for custom processing applications, and Kinesis Data Streams for DynamoDB (an alternative CDC mechanism that writes to a Kinesis data stream instead, offering longer retention up to 365 days and fan-out to multiple consumers); this is the DynamoDB equivalent of Aurora's activity streams, but far more commonly used because DynamoDB's event-driven nature makes stream processing a first-class architectural pattern

### 7. Global Tables: How It Works -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/V2globaltables_HowItWorks.html
- **Type:** Official Docs
- **Summary:** The definitive reference for DynamoDB Global Tables explaining multi-region, multi-active replication where every replica table in every region accepts both reads and writes (true active-active, unlike Aurora Global Database where only the primary region accepts writes unless you use write forwarding at higher latency); covers the two consistency modes: MREC (Multi-Region Eventual Consistency, the default, where writes replicate asynchronously within approximately one second using last-writer-wins conflict resolution based on internal timestamps) and MRSC (Multi-Region Strong Consistency, introduced June 2025, which provides synchronous replication across exactly three regions with RPO=0 but at higher write latency); explains replica table settings synchronization (some settings like capacity mode sync globally while others like auto-scaling and tags are per-replica), the 99.999% availability SLA for global tables (vs 99.99% for single-region), DAX interaction caveats (DAX caches are local to each region and do not replicate), TTL deletes that replicate across regions, and transactions that are isolated per region (transactions cannot span regions); the key architectural comparison: Aurora Global Database replicates at the storage layer with sub-second lag but writes go to one primary region, while DynamoDB Global Tables allows writes to any region with last-writer-wins resolution -- choose based on whether your application can tolerate eventual consistency between regions (DynamoDB) or needs single-region write authority (Aurora)

### 8. Single-Table vs. Multi-Table Design in Amazon DynamoDB -- AWS Database Blog ⏱️ 10 min
- **URL:** https://aws.amazon.com/blogs/database/single-table-vs-multi-table-design-in-amazon-dynamodb/
- **Type:** Blog (AWS Official)
- **Summary:** An AWS Database Blog post that addresses the most paradigm-shifting concept for anyone coming from relational databases: instead of creating a table per entity (Users table, Orders table, Products table) as you would in RDS/Aurora, DynamoDB's single-table design stores multiple entity types in one table with carefully designed partition keys and sort keys that enable retrieving heterogeneous items in a single query -- this compensates for DynamoDB's lack of JOINs by pre-joining related data into item collections; the post compares this approach with multi-table design (one table per entity, more familiar to relational developers, simpler to understand but requires multiple round-trips to assemble related data) and provides guidance on when each approach is appropriate; read Alex DeBrie's companion article at https://www.alexdebrie.com/posts/dynamodb-single-table/ for deeper treatment including the three downsides (steep learning curve, inflexibility when adding new access patterns, complications for analytics); this mental model shift -- designing your schema around access patterns rather than entity relationships -- is the single biggest conceptual hurdle when moving from RDS/Aurora to DynamoDB, and interviewers frequently test whether candidates understand why DynamoDB requires this inverted approach

## Study Tips

- The hardest mental shift from RDS/Aurora to DynamoDB is inverting your design process: with RDS, you normalize your data model first (entities, relationships, foreign keys) and then write queries to retrieve whatever you need using JOINs and WHERE clauses; with DynamoDB, you identify your access patterns first (what queries will the application run?) and then design your primary keys and indexes to serve those exact patterns efficiently. If you discover a new access pattern after deployment, you may need to add a GSI or restructure your data -- there is no ad-hoc query flexibility like SQL provides. When studying partition key design, ask yourself: "what would happen if 80% of my traffic hit a single partition key value?" With RDS/Aurora, the storage layer (EBS or Aurora's distributed volume) handles this transparently; with DynamoDB, it causes throttling. This is why high-cardinality partition keys and write-sharding patterns exist.

- Build a comparison matrix in your notes for the three DynamoDB features that have direct RDS/Aurora parallels: (1) DAX vs Aurora read replicas -- both reduce read load on the primary, but DAX is a cache (data can be stale up to TTL) while Aurora replicas read from the shared storage volume (sub-10ms lag); (2) DynamoDB Global Tables vs Aurora Global Database -- both provide multi-region replication, but Global Tables support active-active writes with last-writer-wins while Aurora Global Database has a single write region with storage-level replication; (3) DynamoDB Streams vs RDS event notifications/Aurora activity streams -- DynamoDB Streams is a far richer CDC mechanism with four granularity levels and native Lambda integration, making it the backbone of event-driven serverless architectures. Drawing these parallels will solidify both your relational and NoSQL knowledge.

- For the GSI vs LSI decision, memorize this simple rule: "LSI is a trap for beginners." LSIs seem simpler because they share the base table's partition key, but they come with three painful constraints that GSIs do not have: (1) they must be created at table creation time (if you forget one, you must recreate the entire table), (2) they impose a 10 GB item collection limit per partition key value (silently breaks your application when you exceed it), and (3) they share the base table's throughput (contention between table reads and index reads). The only advantage LSI has over GSI is supporting strongly consistent reads on the alternate sort key. In practice, default to GSI for everything and only use LSI when you have a proven need for strong consistency on an alternate query path.
