## Overview

This study plan covers the database-specific mechanics of disaster recovery on AWS -- the nitty-gritty of RDS backup internals (automated snapshots + transaction log shipping to S3 every 5 minutes enabling PITR), Aurora Global Database failover procedures (managed switchover with RPO=0 vs unplanned failover with RPO in seconds, the new global writer endpoint that eliminates post-failover endpoint reconfiguration), cross-region read replica promotion for RDS vs Aurora, RDS Proxy's critical role in accelerating failover and managing connections during disruption, and a decision framework for choosing between database DR patterns based on RPO/RTO/cost. After completing this plan, you will be able to design database-tier DR architectures with precise knowledge of failover mechanics, not just high-level strategy selection.

## Resources

### 1. Aurora Global Database -- Switchover and Failover -- Official Documentation ⏱️ 25 min
- **URL:** https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/aurora-global-database-disaster-recovery.html
- **Type:** Official Docs
- **Summary:** The definitive reference for the two Aurora Global Database recovery operations you must distinguish clearly. Managed switchover (formerly "managed planned failover") synchronizes all secondary clusters before switching roles, achieving RPO=0 with zero data loss -- used for planned maintenance, region rotation, and follow-the-sun operations. Managed unplanned failover promotes a secondary cluster immediately without waiting for sync, accepting RPO in seconds based on replication lag at failure time -- used when the primary region is degraded or unavailable. Covers the pre-switchover checklist (version compatibility, replication lag monitoring via AuroraGlobalDBRPOLag CloudWatch metric, parameter group consistency), the exact console and CLI procedures for both operations, write fencing during unplanned failover that prevents split-brain, the automatic snapshot created during unplanned failover (named rds:unplanned-global-failover-*), and the critical difference in topology restoration (switchover preserves topology, failover automatically rebuilds secondaries when the old primary recovers). Read every section carefully -- this is the core of today's study.

### 2. Aurora Global Database Writer Endpoint Deep Dive -- AWS Database Blog ⏱️ 15 min
- **URL:** https://aws.amazon.com/blogs/database/diving-deep-into-the-new-amazon-aurora-global-database-writer-endpoint/
- **Type:** Blog (AWS Official)
- **Summary:** Covers the October 2024 launch of the Aurora Global Database writer endpoint -- a global-level endpoint that automatically updates to point to the current writer instance after switchover or failover, eliminating the post-failover endpoint reconfiguration that was previously the most operationally painful step. Before this feature, applications had to update connection strings or rely on Route53/RDS Proxy endpoint automation after every cross-region failover. Now, the global writer endpoint handles this automatically. This fundamentally changes the database DR operational model by removing a control-plane dependency (updating DNS records or connection strings) from the failover path. Connects directly to the data-plane-vs-control-plane principle from your multi-region study -- the global writer endpoint makes database failover more "data-plane-like" because the endpoint just works without manual intervention.

### 3. RDS Snapshot, Restore, and Recovery Demystified -- AWS Database Blog ⏱️ 20 min
- **URL:** https://aws.amazon.com/blogs/database/amazon-rds-snapshot-restore-and-recovery-demystified/
- **Type:** Blog (AWS Official)
- **Summary:** The essential deep dive into RDS backup mechanics that your previous studies only touched at a high level. Covers how automated backups actually work under the hood: RDS takes a daily storage-level snapshot of the entire DB instance and continuously uploads transaction logs to S3 every 5 minutes. PITR works by restoring the most recent snapshot before your target time and then replaying transaction logs forward to the exact second you specify -- meaning restore time depends on both snapshot restore time AND transaction log replay time (more transactions = longer restore). Critical details include: PITR always creates a NEW DB instance with a new endpoint (it does not restore in-place), the LatestRestorableTime is typically within the last 5 minutes, the 0-35 day automated backup retention period, and how manual snapshots differ (persist until explicitly deleted, no PITR capability, but can be shared cross-account and copied cross-region). Also covers the restore-to-new-instance gotcha that catches many architects -- after PITR you must update application connection strings to point to the new instance.

### 4. RDS Cross-Region Automated Backup Replication -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_ReplicateBackups.html
- **Type:** Official Docs
- **Summary:** Covers the RDS-native feature for replicating automated backups (both snapshots and transaction logs) to a destination Region, enabling cross-region PITR directly from the AWS-managed replicated backups -- without requiring you to set up cross-region read replicas or Aurora Global Database. This is the database-specific version of the AWS Backup cross-region copy rules you studied yesterday, but with a critical advantage: it replicates transaction logs continuously, enabling PITR in the DR region (AWS Backup cross-region copies only copy snapshots, not transaction logs, so they cannot provide PITR). Key limitations to note: not supported for Multi-AZ DB clusters, maximum 20 cross-region backup replications per account, and it supports RDS engines only (not Aurora, which uses Global Database for cross-region DR). This feature maps to the Backup & Restore DR strategy for database tiers that cannot justify the cost of Pilot Light or Warm Standby.

### 5. RDS Proxy Concepts -- Connection Pooling, Multiplexing, and Failover -- Official Documentation ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/rds-proxy.howitworks.html
- **Type:** Official Docs
- **Summary:** Goes deeper into RDS Proxy mechanics than your initial Mar 25 study. Focus on three DR-relevant sections: (1) Connection pooling and multiplexing -- the proxy borrows database connections from the pool at transaction boundaries and returns them after, so thousands of application connections multiplex onto a smaller set of database connections, reducing database memory/CPU overhead. (2) Failover behavior -- during an RDS or Aurora failover, the proxy maintains its IP address and queues incoming requests while the new primary comes online, then redirects traffic transparently. Without a proxy, applications see 60-120 seconds of connection errors during failover; with a proxy, disruption drops to under 10 seconds because the proxy absorbs the DNS propagation delay. (3) Pinning -- when session state (prepared statements, temporary tables, SET variables) prevents safe connection reuse, the proxy pins the session to a specific database connection, which reduces multiplexing efficiency. Understanding pinning is critical because excessive pinning during failover means more connections are disrupted. Also covers IAM authentication integration where applications use short-lived IAM tokens instead of database passwords.

### 6. Financial Customer Case Study -- Aurora Global Database with RDS Proxy for HA/DR -- AWS Database Blog ⏱️ 15 min
- **URL:** https://aws.amazon.com/blogs/database/how-a-large-financial-aws-customer-implemented-ha-and-dr-for-amazon-aurora-postgresql-using-global-database-and-amazon-rds-proxy/
- **Type:** Blog (AWS Official)
- **Summary:** A real-world case study showing how a large financial services customer combined Aurora Global Database with RDS Proxy to achieve both in-region high availability and cross-region disaster recovery for Aurora PostgreSQL. The architecture places RDS Proxy between the application tier and Aurora, so during in-region failover (writer to reader promotion within an Aurora cluster), the proxy queues requests and transparently redirects -- turning what would be application-visible SQL errors into a few seconds of added latency. For cross-region DR, Aurora Global Database provides the replication layer with sub-second lag. This case study is valuable because it shows the complete stack composition: application connects to RDS Proxy endpoint (not directly to Aurora), RDS Proxy handles in-region failover transparently, Aurora Global Database handles cross-region DR, and Route53 routes traffic to the correct region. It is the practical implementation of everything you are studying today.

### 7. Comparing Aurora Replication Solutions -- AWS Prescriptive Guidance ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/prescriptive-guidance/latest/aurora-replication-options/introduction.html
- **Type:** Official Docs (Prescriptive Guidance)
- **Summary:** The decision framework for choosing between Aurora's three replication options: Aurora Replicas (same-region HA, automatic failover, <100ms lag), Cross-Region Aurora Replicas (MySQL only, engine-level binlog replication, few seconds lag, manual promotion, no PostgreSQL support), and Aurora Global Database (storage-level replication, <1s lag, managed switchover/failover, supports both MySQL and PostgreSQL). The comparison table across dimensions -- replication lag, automation, number of replicas, provisioning time -- is the quick-reference decision matrix you need. Critical insight: Aurora PostgreSQL cannot use cross-region read replicas at all, making Global Database the only cross-region DR option for PostgreSQL workloads. For MySQL, cross-region replicas exist but Global Database is superior in every dimension (faster replication, managed failover, storage-level vs engine-level). ## Study Tips

- **Map each database DR mechanism to its DR strategy tier.** As you read, build a mental matrix: (1) Backup & Restore = RDS automated backups with cross-region replication (Resource 4) + manual snapshots copied cross-region, RPO hours, RTO hours. (2) Pilot Light = Cross-region read replica (RDS) or Aurora Global Database secondary with minimal compute (Aurora Serverless v2 at 0.5 ACU), RPO minutes/seconds, RTO tens of minutes. (3) Warm Standby = Aurora Global Database with provisioned compute serving reads, RPO seconds, RTO minutes. (4) Active-Active = Aurora Global Database with write forwarding or DynamoDB Global Tables (from your Mar 26 study), RPO near-zero, RTO near-zero. For each tier, know the exact failover procedure: Backup & Restore requires PITR to a new instance + connection string update; Pilot Light requires read replica promotion (RDS) or Global Database failover (Aurora); Warm Standby uses Global Database managed switchover; Active-Active uses the global writer endpoint and routing controls.

- **Understand the restore-to-new-instance constraint and its DR implications.** Both RDS PITR and snapshot restore create a NEW database instance with a NEW endpoint. This means your DR runbook must include the step "update application connection strings" or "update RDS Proxy target" after every restore. RDS Proxy mitigates this for in-region failover (it tracks the new primary automatically), but for cross-region PITR from replicated backups, you are creating an entirely new database in a new region with a new endpoint. This is why Pilot Light and above -- which use continuously replicating read replicas or Global Database secondaries with stable endpoints -- provide dramatically faster RTO than Backup & Restore.

- **Practice the switchover vs failover distinction until it is automatic.** In an interview, if asked "your primary Aurora region is healthy but you need to rotate to a different region for compliance," the answer is managed switchover (RPO=0, no data loss, topology preserved). If asked "your primary region just went down," the answer is managed unplanned failover (RPO in seconds, data loss possible, write fencing attempted, automatic topology rebuild). The CLI commands are different too: switchover-global-cluster vs failover-global-cluster --allow-data-loss. The --allow-data-loss flag is the exam tell -- if you see it, it is unplanned failover.
