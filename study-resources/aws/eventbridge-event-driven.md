# Amazon EventBridge & Event-Driven Architecture -- 2-Hour Study Plan

**Total Time:** ~125 minutes
**Created:** 2026-04-09

## Overview

This study plan covers Amazon EventBridge as the central event routing layer in AWS serverless architectures -- from the core event bus model (default, custom, partner) with content-based filtering rules and input transformers through Schema Registry/Discovery, Archive/Replay, cross-account and cross-region event delivery, EventBridge Scheduler (rate/cron/one-time with universal targets), EventBridge Pipes (source-filter-enrich-target point-to-point integrations), the EventBridge vs SNS vs SQS decision framework, event-driven architecture patterns (choreography vs orchestration), pricing model, and Terraform implementation patterns. After completing this plan, you will understand how to design production event-driven architectures on AWS, make informed decisions about when to use EventBridge vs SNS vs SQS, and implement EventBridge infrastructure with Terraform at the Solutions Architect / DevOps Engineer level.

## Resources

### 1. EventBridge Core Concepts -- AWS Official Docs ⏱️ 20 min
- **URLs:**
  - Concepts overview: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-what-is-how-it-works-concepts.html
  - Event buses: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-event-bus.html
  - Rules: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-rules.html
  - Event patterns: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-event-patterns.html
- **Type:** Official Docs (four pages, ~5 min each)
- **Summary:** The foundational pages that establish the EventBridge mental model. The concepts page introduces the two primary mechanisms: **event buses** (routers that receive events and deliver them to zero or more targets via rules -- the many-to-many broadcast model) and **Pipes** (point-to-point integrations from a single source to a single target with optional filtering and enrichment). Three event bus types: (1) **Default event bus** -- automatically receives events from AWS services (EC2 state changes, S3 object notifications, GuardDuty findings, etc.) at no cost for AWS management events, (2) **Custom event buses** -- receive events from your applications via PutEvents API ($1/million custom events published), (3) **Partner event buses** -- receive events from SaaS partners (Datadog, PagerDuty, Zendesk, Auth0, etc.) at $1/million partner events. Rules evaluate incoming events against **event patterns** using content-based filtering -- you specify JSON fields and values to match, supporting prefix matching, suffix matching, numeric ranges, exists/not-exists checks, anything-but negation, CIDR matching, and complex combinations with $or logic. Key limits: up to 300 rules per event bus (adjustable via Service Quotas), maximum 5 targets per rule. Each rule can optionally include an **input transformer** that reshapes the event before delivery using input paths (JSON path extractions into variables) and an input template (string with variable placeholders). This is where to understand the difference between EventBridge as an event router (content-based filtering, many producers to many consumers) versus SQS as a message queue (pull-based buffering) and SNS as a pub/sub notification service (topic-based fan-out).

### 2. Targets and Input Transformation -- AWS Official Docs ⏱️ 15 min
- **URLs:**
  - Targets: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-targets.html
  - Input transformation: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-transform-target-input.html
  - Input transformer tutorial: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-input-transformer-tutorial.html
- **Type:** Official Docs (three pages, ~5 min each)
- **Summary:** Targets are the destinations where EventBridge delivers matched events. The targets page catalogs all supported target types -- critical for understanding EventBridge's integration breadth. Key targets include: Lambda functions (most common), Step Functions state machines (StartExecution for Standard, StartSyncExecution for synchronous Express), SQS queues, SNS topics, Kinesis Data Streams, ECS tasks (RunTask), CodeBuild projects, CodePipeline, Batch jobs, SSM Run Command/Automation, API destinations (HTTPS endpoints for external webhooks), **other event buses** (for cross-account and cross-region routing), CloudWatch log groups, Redshift, SageMaker pipelines, and more. Each target can have a **dead-letter queue** (SQS) configured for failed delivery retries -- EventBridge retries for up to 24 hours with exponential backoff, then sends to the DLQ. The input transformation pages cover how to customize the event payload before delivery: define **input paths** that extract values from the event JSON using JSONPath, then construct an **input template** using those extracted values as `<variable>` placeholders. You can define up to 100 variables. Five input options per target: matched event (default), part of the event (input path), constant text, input transformer (paths + template), or no input. This is analogous to Step Functions' Parameters/ResultSelector for reshaping data between states -- same principle of decoupling the event shape from what the consumer expects.

### 3. Schema Registry, Archive, and Replay -- AWS Official Docs ⏱️ 15 min
- **URLs:**
  - Schema Registry: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-schema.html
  - Schema registries: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-schema-registry.html
  - Archive and replay: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-archive.html
- **Type:** Official Docs (three pages, ~5 min each)
- **Summary:** Three operational features that make EventBridge production-ready. **Schema Registry** provides a repository of event schemas (OpenAPI 3.0 and JSONSchema Draft 4 formats) with three registry types: the AWS event schema registry (pre-built schemas for all AWS service events -- invaluable for knowing the exact JSON shape of EC2 state changes, S3 notifications, etc.), custom schema registries (upload your own schemas), and discovered schemas (when **schema discovery** is enabled on an event bus, EventBridge automatically detects and catalogs the schema of every event flowing through, including versioning when schemas evolve -- powerful for documenting event contracts in microservice architectures). You can generate code bindings (Java, Python, TypeScript) from discovered schemas for type-safe event handling. **Archive** saves events matching an optional event pattern to internal storage (S3-backed, not directly accessible) with configurable retention (indefinite by default, or specify days). **Replay** re-sends archived events back to the source event bus within a specified time window -- critical for debugging (reproduce a production issue), hydrating new services with historical events, recovering from downstream failures, and testing. Key constraints: maximum 10 concurrent replays per account per region, replays do not preserve event ordering, archived events replay to the source bus only (not to a different bus), and replayed events are treated as new events (consumers must be idempotent). Replay costs the same as publishing custom events ($1/million). Schema discovery costs $0.10 per million events ingested for discovery.

### 4. EventBridge Scheduler -- AWS Official Docs ⏱️ 15 min
- **URLs:**
  - What is Scheduler: https://docs.aws.amazon.com/scheduler/latest/UserGuide/what-is-scheduler.html
  - Schedule types: https://docs.aws.amazon.com/scheduler/latest/UserGuide/schedule-types.html
  - Universal targets: https://docs.aws.amazon.com/scheduler/latest/UserGuide/managing-targets-universal.html
- **Type:** Official Docs (three pages, ~5 min each)
- **Summary:** EventBridge Scheduler is a separate, purpose-built scheduling service (launched Nov 2022) that supersedes the legacy EventBridge scheduled rules. Three schedule types: (1) **Rate-based** -- recurring at fixed intervals (`rate(5 minutes)`, `rate(1 hour)`, `rate(7 days)`), (2) **Cron-based** -- fine-grained recurring patterns with timezone support (`cron(0 8 ? * MON-FRI *)` for 8 AM weekdays), including wildcard operators and day-of-week/month selectors, (3) **One-time** -- single invocation at a specific date/time (`at(2026-04-15T09:00:00)`) with IANA timezone support, ideal for deferred actions like "send reminder in 48 hours". All schedule types support **timezone configuration** via IANA Time Zone Database names with automatic DST handling (spring-forward skips invocation, fall-back runs once only). **Universal targets** enable invoking 270+ AWS services and 6,000+ API operations directly -- format: `arn:aws:scheduler:::aws-sdk:service:apiAction`. This means Scheduler can call any AWS API on a schedule without Lambda in the middle -- same cost optimization pattern as Step Functions direct SDK integrations and API Gateway direct AWS service integrations you already studied. Four **templated targets** (Lambda, SQS, SNS, EventBridge) provide simplified configuration. **Flexible time windows** (OFF, 5-15 min) disperse invocations to reduce thundering herd effects. Built-in retry policy (configurable max retries and max retention time) plus DLQ support for failed invocations. Scheduler vs legacy scheduled rules: Scheduler supports one-time schedules, timezone-aware scheduling, flexible time windows, universal targets, higher throughput (millions of schedules), and schedule groups for organization -- always use Scheduler for new workloads. Pricing: $1.00 per million invocations with a generous 14 million invocations/month free tier.

### 5. EventBridge Pipes -- AWS Official Docs ⏱️ 15 min
- **URLs:**
  - Pipes overview: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-pipes.html
  - Pipes concepts: https://docs.aws.amazon.com/eventbridge/latest/userguide/pipes-concepts.html
  - Pipes enrichment: https://docs.aws.amazon.com/eventbridge/latest/userguide/pipes-enrichment.html
- **Type:** Official Docs (three pages, ~5 min each)
- **Summary:** EventBridge Pipes (launched Dec 2022) provide **point-to-point integrations** with a four-stage pipeline: Source -> Filter -> Enrich -> Target. Unlike event buses (many-to-many broadcast), Pipes connect exactly one source to exactly one target -- think of it as a dedicated pipeline vs a broadcast router. **Sources** include DynamoDB Streams, Kinesis Data Streams, SQS queues (Standard and FIFO), Amazon MQ (ActiveMQ and RabbitMQ), Apache Kafka (MSK and self-managed), and Kinesis Data Firehose (added 2024). **Filtering** uses the same event pattern syntax as event bus rules, applied before enrichment -- you only pay for events that pass the filter ($0.40/million filtered requests), making filtering a cost optimization lever. **Enrichment** is optional and synchronous -- EventBridge invokes an enrichment service (Lambda function, Step Functions state machine, API Gateway endpoint, or API destination), waits for the response, and passes the enriched data to the target. Enrichment responses are limited to 6 MB. **Targets** include most event bus targets plus additional services. Pipes maintain event ordering within batches and support batching and concurrency configuration. The critical mental model: Pipes replace the pattern of "event source mapping -> Lambda function that just transforms data and forwards it" with a managed, codeless integration. Compare to your Lambda event source mapping knowledge -- Pipes serve a similar purpose (polling sources, batching, filtering) but without requiring Lambda as the compute layer. For simple transformations, Pipes eliminate Lambda entirely; for complex enrichment, Pipes use Lambda (or Step Functions/API Gateway) only for the enrichment step, not for the routing logic. Pipes vs event bus rules: use Pipes for dedicated source-to-target integrations with enrichment; use event bus rules when multiple consumers need the same events or when you need fan-out.

### 6. Cross-Account and Cross-Region Event Delivery -- AWS Official Docs + AWS Compute Blog ⏱️ 15 min
- **URLs:**
  - Cross-region: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-cross-region.html
  - Event bus permissions: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-event-bus-perms.html
  - Cross-region routing blog: https://aws.amazon.com/blogs/compute/expanding-cross-region-event-routing-with-amazon-eventbridge/
- **Type:** Official Docs + Blog (AWS Official)
- **Summary:** EventBridge supports sending events across AWS accounts and regions -- essential for multi-account Organizations architectures. **Cross-account delivery**: configure a resource-based policy on the target account's event bus to allow `events:PutEvents` from the source account. The source account creates a rule with the target event bus ARN as a target. As of early 2025, EventBridge also supports **cross-account targets** directly -- you can target SQS, Lambda, Kinesis, SNS, and API Gateway in another account from a rule, without needing an intermediate event bus in the target account (the target resource must have a resource-based policy allowing the EventBridge service principal). This simplifies architectures that previously required a two-hop pattern (source bus -> target bus -> target resource). **Cross-region delivery**: EventBridge can route events from any commercial region to any other commercial region (excluding GovCloud and China partitions). Create a rule in the source region with a target event bus in the destination region. Key constraint: cross-region targets are limited to other event buses -- you cannot directly target a Lambda function or SQS queue in another region from a rule. For cross-region + cross-account, use the intermediate event bus pattern: source bus (us-east-1, Account A) -> target bus (eu-west-1, Account B) -> rule on target bus -> local target. The event bus permissions page covers resource-based policies in detail, including the Organization-level condition key `aws:PrincipalOrgID` for granting access to all accounts in your Organization -- a pattern you already know from your IAM Advanced and Organizations/SCPs studies.

### 7. SQS vs SNS vs EventBridge Decision Guide -- AWS Official Decision Guide ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/decision-guides/latest/sns-or-sqs-or-eventbridge/sns-or-sqs-or-eventbridge.html
- **Type:** Official Docs (AWS Decision Guide)
- **Summary:** The authoritative AWS decision framework for choosing between the three messaging/eventing services -- a frequent interview question and essential architectural knowledge. The guide evaluates across several dimensions: **Communication model** -- SQS is pull-based (consumers poll), SNS is push-based (real-time delivery to subscribers), EventBridge is event-driven (content-based routing via rules). **Persistence** -- SQS persists messages until consumed or retention expires (up to 14 days), SNS and EventBridge do not persist (real-time delivery only, though EventBridge Archive provides optional persistence). **Ordering** -- SQS FIFO and SNS FIFO guarantee strict ordering, EventBridge provides no ordering guarantees. **Filtering** -- SQS has no native filtering (consumers see all messages), SNS has subscription filter policies (attribute-based), EventBridge has advanced content-based event pattern matching (the most powerful filtering of the three). **Delivery guarantees** -- all three provide at-least-once delivery. **Throughput** -- SQS Standard has nearly unlimited throughput, SNS handles millions of subscribers, EventBridge handles thousands of events per second per region (soft limit, requestable increase). **When to use each**: SQS for decoupling with buffering, rate limiting, and reliable processing (the consumer controls pace); SNS for fan-out notification to multiple subscribers with basic filtering; EventBridge for advanced content-based routing, multi-source event aggregation, third-party SaaS integration, cross-account/cross-region delivery, archive/replay, and schema discovery. **Common compositions**: SNS + SQS fan-out (SNS distributes to multiple SQS queues, each with independent consumers), EventBridge + SQS (EventBridge routes by content, SQS buffers for reliable processing), EventBridge + SNS (EventBridge filters, SNS fans out). This directly prepares you for tomorrow's SQS/SNS deep dive.

### 8. Orchestration vs Choreography and Terraform Patterns -- AWS Prescriptive Guidance + Spacelift Blog ⏱️ 15 min
- **URLs:**
  - Choosing coordination approach: https://docs.aws.amazon.com/prescriptive-guidance/latest/modernization-integrating-microservices/choosing-approach.html
  - Orchestration and choreography patterns: https://docs.aws.amazon.com/prescriptive-guidance/latest/cloud-design-patterns/orchestration-choreography.html
  - Terraform EventBridge guide: https://spacelift.io/blog/terraform-eventbridge
- **Type:** Official Docs (AWS Prescriptive Guidance) + Blog (Spacelift)
- **Summary:** Two complementary topics to close the study plan -- architectural patterns and infrastructure-as-code. The AWS Prescriptive Guidance pages establish the **choreography vs orchestration** decision framework: **Choreography** (decentralized, event-driven) -- each service publishes events and reacts to events independently, no central coordinator, implemented with EventBridge or SNS, choose when working within a single microservice boundary with simple 2-4 step workflows. **Orchestration** (centralized, workflow-driven) -- a central coordinator (Step Functions) manages the workflow, calls services in sequence/parallel, handles retries and compensation (saga pattern), choose when coordinating across microservice boundaries with complex multi-step transactions requiring rollback. The key insight connecting yesterday's Step Functions study to today: orchestration and choreography are complementary, not competing -- use Step Functions to orchestrate complex business processes (order fulfillment saga) and EventBridge to choreograph loosely-coupled domain events (order.completed event triggers shipping, billing, and analytics services independently). The Spacelift blog covers practical Terraform implementation patterns: `aws_cloudwatch_event_bus` for custom event buses, `aws_cloudwatch_event_rule` with `event_pattern` for content-based filtering, `aws_cloudwatch_event_target` linking rules to targets (Lambda, SQS, Step Functions) with `input_transformer` blocks, `aws_cloudwatch_event_archive` for archive configuration, `aws_cloudwatch_event_connection` and `aws_cloudwatch_event_api_destination` for API destinations, `aws_scheduler_schedule` for Scheduler with universal targets, DLQ configuration on targets, and IAM role/policy setup for EventBridge to invoke targets. Note: EventBridge resources in the AWS provider still use the `aws_cloudwatch_event_*` prefix (legacy naming from when EventBridge was CloudWatch Events) -- the `aws_pipes_pipe` resource uses the newer naming convention.

## Study Tips

- Build an EventBridge component decision matrix with three columns (Event Bus Rules, Pipes, Scheduler) and rows for: pattern (many-to-many vs point-to-point vs time-triggered), sources, filtering, enrichment, targets, ordering guarantees, retry behavior, DLQ support, and pricing model. This single artifact captures the most commonly confused EventBridge topic -- candidates frequently conflate event bus rules with Pipes or do not know when to use Scheduler vs scheduled rules. The critical distinction: event bus rules are routers (one event, many consumers), Pipes are pipelines (one source, one target, with enrichment), and Scheduler is a cron/rate/one-time trigger (time-based, not event-based).

- Map the complete event flow for a production scenario end-to-end. Example: "User uploads file to S3" -> S3 sends event to default event bus -> Rule 1 matches `s3:ObjectCreated:*` for `.csv` files and targets a Step Functions workflow (orchestration: validate -> transform -> load) -> Rule 2 matches the same event and targets an SQS queue for audit logging -> Step Functions publishes `processing.completed` to a custom event bus -> Rule 3 targets SNS for notification fan-out -> Rule 4 targets a Lambda function for analytics. Trace the IAM permissions, DLQ configuration, and input transformers at each hop. This exercise connects today's EventBridge knowledge with Lambda (Day 1), API Gateway (Day 2), and Step Functions (Day 3) into a cohesive architecture.

- Internalize the pricing differences because they drive architectural decisions: EventBridge custom/partner events cost $1/million published (AWS service events are free), Scheduler costs $1/million invocations (14M/month free tier), Pipes cost $0.40/million filtered requests. Compare with SNS ($0.50/million publishes) and SQS ($0.40/million requests after 1M free). EventBridge is roughly 2x the cost of SNS for simple fan-out -- you pay the premium for content-based routing, schema discovery, archive/replay, and cross-account/cross-region delivery. If you only need basic topic-based fan-out, SNS is cheaper; if you need advanced filtering or SaaS integration, EventBridge justifies the cost.
