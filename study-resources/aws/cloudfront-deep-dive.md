# CloudFront Deep Dive -- 2-Hour Study Plan

**Total Time:** ~120 minutes
**Created:** 2026-03-13
**Difficulty:** Intermediate

## Overview

This study plan covers Amazon CloudFront's full architecture and feature set: the two-tier POP/regional-edge-cache hierarchy, origin types (S3, ALB, custom HTTP, VPC origins), cache behaviors with path-pattern routing, the cache-policy-vs-origin-request-policy separation, Origin Access Control (OAC) replacing legacy OAI, edge compute with Lambda@Edge and CloudFront Functions, content security (signed URLs, signed cookies, geo-restriction), cache invalidation vs versioned file names, and logging/monitoring options. After completing it, you will understand how to design a CloudFront distribution that serves both static S3 assets and dynamic ALB-backed content under a single domain, how to secure origins so only CloudFront can reach them, how to run lightweight logic at the edge without a full Lambda function, and how cache policies and origin request policies work together to maximize cache hit ratios while still forwarding necessary data to your origin -- all of which connects to the WAF integration you studied on Mar 10 (WAF Web ACLs attach to CloudFront distributions) and the Route53 alias records you covered on Mar 11 (Route53 alias records point to CloudFront distribution domain names with zero-cost, instant resolution).

## Resources

### 1. How CloudFront Delivers Content -- Official Conceptual Overview ⏱️ 10 min
- **URL:** https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/HowCloudFrontWorks.html
- **Type:** Official Docs
- **Summary:** The essential starting page that explains CloudFront's three-step request flow (DNS routes viewer to nearest POP, POP checks cache, cache miss goes to regional edge cache before origin) and the two-tier caching architecture -- 750+ POPs serve content directly to viewers while 15 regional edge caches sit between POPs and your origin with larger cache capacity and longer object retention; pay attention to the detail that proxy HTTP methods (PUT, POST, PATCH, DELETE) and dynamic requests bypass regional edge caches and go straight to the origin, which is critical for understanding why CloudFront handles static and dynamic content differently; this page builds the mental model everything else depends on

### 2. Cache Behavior Settings -- Official Reference ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/DownloadDistValuesCacheBehavior.html
- **Type:** Official Docs
- **Summary:** The definitive reference for every setting within a cache behavior -- path patterns (wildcard matching with * and ?, processed in order with first match winning, default behavior is * and cannot be changed), viewer protocol policy (HTTP+HTTPS, redirect HTTP to HTTPS, HTTPS-only), allowed HTTP methods (GET/HEAD vs full CRUD), TTL settings (Minimum/Default/Maximum and how they interact with origin Cache-Control headers), query string forwarding and caching, cookie forwarding, header-based caching, field-level encryption, automatic compression (gzip and Brotli), and Lambda function associations at four trigger points; the key architectural insight is that each behavior maps to exactly one origin (or origin group for failover), so you use multiple behaviors with different path patterns to route /api/* to your ALB and /static/* to your S3 bucket -- this is the mechanism that makes CloudFront work as both a CDN and a reverse proxy under a single domain

### 3. Cache Policies and Origin Request Policies -- How They Work Together ⏱️ 15 min
- **URL (start here):** https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/cache-key-understand-cache-policy.html
- **Type:** Official Docs (read two pages sequentially)
- **Summary:** Read the cache policy page first to understand the three components of a cache policy -- TTL settings (Minimum/Default/Maximum TTL that override or complement origin Cache-Control headers), cache key settings (which headers, cookies, and query strings become part of the cache key), and compression support (normalizing Accept-Encoding for gzip/Brotli); then read the companion page on how cache policies and origin request policies work together (https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/understanding-how-origin-request-policies-and-cache-policies-work-together.html) to grasp the critical separation: the cache policy controls what goes into the cache key (fewer values = better cache hit ratio), the origin request policy controls what additional headers/cookies/query strings get forwarded to the origin (more values = richer context for your backend), and everything in the cache policy is automatically included in origin requests so you never need to duplicate values between the two policies; also skim the managed cache policies page (https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/using-managed-cache-policies.html) to see the pre-built policies like CachingOptimized (recommended for S3) and CachingDisabled (for dynamic API content)

### 4. Restricting Access to S3 Origins -- OAC vs Legacy OAI ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/private-content-restricting-access-to-s3.html
- **Type:** Official Docs
- **Summary:** The complete guide to ensuring only CloudFront can access your S3 bucket origin, covering both Origin Access Control (OAC, recommended) and the legacy Origin Access Identity (OAI, deprecated); OAC wins on every dimension -- it supports all AWS regions including opt-in regions launched after December 2022, works with SSE-KMS encrypted objects (OAI cannot decrypt KMS-encrypted content, which directly connects to your KMS deep dive from Mar 9), supports dynamic requests (PUT, POST, DELETE) for upload-through-CloudFront patterns, and uses short-term credentials with frequent rotation instead of a static identity; the page includes the exact S3 bucket policy statements you need for both read-only and read-write OAC configurations, the KMS key policy addition required for SSE-KMS, and a step-by-step migration path from OAI to OAC; when you build the Terraform tonight, use `aws_cloudfront_origin_access_control` not the legacy `aws_cloudfront_origin_access_identity`

### 5. VPC Origins and CloudFront with ALB/NLB Patterns ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/private-content-vpc-origins.html
- **Type:** Official Docs
- **Summary:** CloudFront VPC Origins (launched late 2024) is a game-changer for securing ALB/NLB origins -- instead of the old pattern of making your ALB public and validating a custom header to ensure requests come from CloudFront, VPC Origins lets you place your ALB or NLB in a private subnet with no public IP and CloudFront connects to it over the AWS backbone via a service-managed ENI; supported origin types are ALBs (including internal), NLBs (with restrictions: no dual-stack, no TLS listeners, must have security group), and EC2 instances in private subnets; the security benefit is that CloudFront becomes the sole ingress point and your origin has zero public internet exposure; be aware of current limitations: no WebSocket support, no gRPC, and no origin request/response Lambda@Edge triggers with VPC origins; for the Terraform build tonight, this is the pattern to use with your ALB origin rather than the legacy custom-header approach

### 6. CloudFront Functions vs Lambda@Edge -- Choosing Edge Compute ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/edge-functions-choosing.html
- **Type:** Official Docs
- **Summary:** The official comparison page that defines exactly when to use each edge compute option across six dimensions: CloudFront Functions run in 225+ edge locations (closer to viewers, lower latency) while Lambda@Edge runs in 13 regional edge caches; CloudFront Functions are limited to viewer request and viewer response triggers only while Lambda@Edge also supports origin request and origin response triggers; CloudFront Functions must complete in sub-millisecond time with 2MB memory and 10KB code in JavaScript ES5.1 only, while Lambda@Edge gets up to 5 seconds for viewer triggers and 30 seconds for origin triggers with up to 10GB memory in Node.js or Python; CloudFront Functions cannot make network calls, access the request body, or use the AWS SDK, while Lambda@Edge can do all of these; the decision framework is clear -- use CloudFront Functions for simple, high-volume manipulations (URL rewrites, header normalization, cache key manipulation, simple JWT validation with KeyValueStore) and Lambda@Edge for anything requiring network access, external service calls, or complex processing (A/B testing with database lookups, image transformation at the origin, authentication against an external IdP)

### 7. Serving Private Content -- Signed URLs, Signed Cookies, and Geo-Restriction ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/PrivateContent.html
- **Type:** Official Docs
- **Summary:** The overview page for CloudFront's content access control mechanisms covering the three-step setup (restrict direct origin access via OAC, specify trusted key groups that can create signatures, implement signed URLs or signed cookies in your application); after reading the overview, follow the link to the signed URLs vs signed cookies decision page to understand the selection criteria -- use signed URLs when you want to restrict access to individual files or your users use a client that does not support cookies (like a mobile app making API calls), and use signed cookies when you want to provide access to multiple restricted files (like all HLS video segments for a subscriber) without changing URLs; the page also covers canned policies (simpler, one file, one expiration) vs custom policies (IP address restriction, access window with start time, wildcard path matching); additionally read the geo-restriction section (https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/georestrictions.html) to understand CloudFront's built-in country-level blocking using either an allow-list or deny-list approach -- this is separate from Route53 geolocation routing, which you already know, and operates at the CDN edge before the request ever reaches your origin

### 8. Cache Invalidation and Versioned File Names ⏱️ 10 min
- **URL:** https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/Invalidation.html
- **Type:** Official Docs
- **Summary:** Covers the mechanics and cost trade-offs of cache invalidation: invalidation removes objects from edge caches before TTL expiration, the first 1,000 invalidation paths per month are free with $0.005 per path after that, using /* counts as a single path (cheap full-distribution invalidation), and you can have up to 3,000 invalidation paths in progress at once; the critical recommendation is to prefer versioned file names (e.g., style-v2.css or style.abc123.css) over invalidation for several reasons -- invalidation does not clear browser caches or corporate proxy caches, versioned files enable instant rollback by pointing back to the old version, and build tools like webpack/Vite automatically generate content-hashed filenames; the practical pattern is a two-tier approach: set long TTLs (1 year) on versioned static assets and short TTLs (60 seconds) on HTML entry points that reference those versioned assets, so users always get fresh HTML that points to the correct asset versions without any invalidation needed

### 9. Logging, Monitoring, and Real-Time Logs ⏱️ 5 min
- **URL:** https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/logging.html
- **Type:** Official Docs
- **Summary:** A quick orientation page covering CloudFront's three logging mechanisms: standard access logs (v2) that deliver detailed per-request records to S3, CloudWatch Logs, or Data Firehose with a delay of minutes; real-time logs that deliver records within seconds to Kinesis Data Streams for near-instant analysis (configurable sampling rate to control cost and volume, useful for monitoring cache hit ratios and identifying 4xx/5xx spikes); and edge function logs that capture CloudFront Functions and Lambda@Edge execution output to CloudWatch Logs; also note the CloudWatch metrics that CloudFront publishes by default (Requests, BytesDownloaded, BytesUploaded, 4xxErrorRate, 5xxErrorRate, TotalErrorRate) and the additional metrics available for a per-distribution fee (cache hit rate, origin latency, error rate by status code); this connects to the CloudWatch monitoring you will deepen in Week 6

## Study Tips

- Build a mental architecture diagram with four layers as you read: viewers at the top connect to POPs (225+ edge locations), POPs connect to regional edge caches (15 locations), regional edge caches connect to Origin Shield (optional, single additional cache layer in one AWS Region), and Origin Shield connects to your origins (S3, ALB, custom HTTP); every cache miss walks down one layer, and every cache hit stops the walk and returns content upward. Understanding this layered hierarchy is the single most important concept for CloudFront -- every other feature (cache policies, behaviors, OAC, edge compute) plugs into a specific layer of this hierarchy.
- When studying cache policies vs origin request policies, think of it as a Venn diagram: the cache policy defines the inner circle (what makes cached objects unique), and the origin request policy defines the outer ring (additional context your origin needs but that should not fragment the cache). The cardinal sin is putting too many values in the cache key -- every additional header, cookie, or query string parameter multiplies cache fragmentation and tanks your hit ratio. The managed CachingOptimized policy includes only the minimum (no headers, no cookies, no query strings) for exactly this reason.
- Connect CloudFront security to the services you already know: WAF Web ACLs (Mar 10) attach to CloudFront distributions to filter requests at the edge before they reach your origin, OAC uses IAM-style SigV4 signing to authenticate requests to S3 (connecting to your IAM knowledge from Mar 5), signed URLs/cookies use RSA or ECDSA key pairs for content access control, and Shield Standard protects every CloudFront distribution automatically at no cost with Shield Advanced available for enhanced L7 DDoS protection with SRT support.
