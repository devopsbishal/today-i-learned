## Resources

### 1. Amazon SQS Basic Architecture and Core Concepts -- AWS Official Docs ⏱️ 15 min
- **URLs:**
  - SQS basic architecture: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-basic-architecture.html
  - Standard queues: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/standard-queues.html
  - Message lifecycle: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-message-lifecycle.html
- **Type:** Official Docs (three short pages)
- **Summary:** The foundational pages that lock in the SQS mental model before you dig into the operational knobs. The basic architecture page establishes the producer/consumer/queue triangle, distributed storage across multiple servers and Availability Zones (message durability is free -- you do not configure replication), and the at-least-once delivery contract for Standard queues (duplicates are possible but rare and must be handled by idempotent consumers). The Standard queues page makes the ordering guarantee explicit: "best-effort ordering" means a message sent second *might* be received first, and you must not build logic that depends on ordering in a Standard queue. Pay attention to the throughput numbers: Standard queues support "nearly unlimited" API calls per second per action (SendMessage, ReceiveMessage, DeleteMessage) across the queue, while FIFO queues are capped at 300 messages/second without batching or 3,000 messages/second with batched API calls in default mode, or 70,000+ transactions/second in high-throughput mode (covered later). The message lifecycle page is the most important of the three -- it walks through the five-step lifecycle: (1) producer calls SendMessage, (2) SQS distributes the message redundantly across multiple servers, (3) consumer calls ReceiveMessage (short or long poll), (4) SQS returns the message and starts the visibility timeout, (5) consumer processes and calls DeleteMessage before the timeout expires, or the message reappears. This lifecycle is the mental model you will use for every subsequent topic -- visibility timeout tuning, DLQ maxReceiveCount, in-flight message limits (120,000 for Standard, 20,000 for FIFO), and partial batch failure handling all derive from it.

### 2. Visibility Timeout, Long Polling, and Dead-Letter Queues -- AWS Official Docs ⏱️ 20 min
- **URLs:**
  - Visibility timeout: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-visibility-timeout.html
  - Short and long polling: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-short-and-long-polling.html
  - Dead-letter queues: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-dead-letter-queues.html
- **Type:** Official Docs (three pages, ~7 min each)
- **Summary:** The three operational knobs you will tune on every production SQS queue. **Visibility timeout** (default 30 seconds, range 0 seconds to 12 hours, max cumulative extension 12 hours regardless of ChangeMessageVisibility calls) is the window during which a received message is hidden from other consumers. The critical Lambda interaction rule: set visibility timeout to at least 6x your Lambda function timeout plus MaximumBatchingWindowInSeconds. If the Lambda takes longer than the visibility timeout, SQS re-delivers the message to a second Lambda invocation while the first is still running, causing duplicate processing -- the most common production bug when connecting SQS to Lambda. Use ChangeMessageVisibility for dynamically-extending long-running work (a worker reports "still processing" heartbeats to push the deadline out). **Short polling** returns immediately (even with zero messages) and samples a subset of SQS servers -- it generates empty responses you still pay for and is rarely the right choice. **Long polling** (set ReceiveMessageWaitTimeSeconds to 20, the maximum) waits up to 20 seconds for at least one message to appear, samples all servers for better message visibility, reduces empty-response costs by ~97%, and is the default you should enable on every queue. Critical gotcha: even with long polling, ReceiveMessage can still return zero messages when the queue is non-empty because SQS samples a distributed subset -- consumers must loop and keep polling, never treat an empty response as "queue is empty". **Dead-letter queues** capture messages that exceeded maxReceiveCount (typically 3-5 for production workloads) without being deleted. Three non-obvious rules: (1) the DLQ type must match the source queue (FIFO->FIFO, Standard->Standard), (2) a DLQ is just a regular SQS queue with a redrive policy on the source pointing to it -- it is not a special queue type, (3) the redrive-allow-policy on the DLQ controls which source queues are allowed to send to it (byQueue, allowAll, or denyAll) and you need manual redrive permissions (StartMessageMoveTask) to push messages from the DLQ back to the source for reprocessing. Configure a CloudWatch alarm on the DLQ ApproximateNumberOfMessagesVisible metric -- a non-zero DLQ means something broke and you need to investigate.

### 3. SQS FIFO Queues: Exactly-Once, MessageGroupId, and High Throughput -- AWS Official Docs ⏱️ 15 min
- **URLs:**
  - FIFO queues overview: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-fifo-queues.html
  - Exactly-once processing: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/FIFO-queues-exactly-once-processing.html
  - High throughput for FIFO: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/high-throughput-fifo.html
- **Type:** Official Docs (three pages, ~5 min each)
- **Summary:** The deep dive on the "ordering plus exactly-once" story that differentiates FIFO from Standard. FIFO queues guarantee two properties Standard cannot: strict ordering within a message group and exactly-once delivery within a 5-minute deduplication window. **MessageGroupId** is the partition key -- messages with the same group ID are processed strictly in order, while messages in different groups can be processed in parallel. The canonical use case: per-customer or per-tenant ordering (use customer ID as the group ID) where you need strict order within each customer but want parallel throughput across customers. If you use a single group ID for all messages, you get strict global ordering but your consumer throughput is limited to one at a time per group. **MessageDeduplicationId** (either explicitly provided or content-based via SHA-256 hash of the body) prevents duplicate SendMessage calls from introducing duplicates into the queue -- if you retry SendMessage within 5 minutes with the same deduplication ID, SQS silently accepts but does not enqueue the duplicate. Note the precise semantics: "exactly-once delivery" means SQS will not introduce a duplicate into the *queue*, not that the consumer will only see the message once. If a consumer receives a message, fails to delete it, and the visibility timeout expires, the message will still be re-delivered -- so consumer-side idempotency remains necessary even with FIFO. **Default FIFO throughput** is 300 messages/second per API action or 3,000 with batching. **High throughput mode** (enabled via the DeduplicationScope and FifoThroughputLimit attributes) scales to 70,000+ transactions/second by partitioning the deduplication scope to the message group level rather than the entire queue -- but this weakens the deduplication guarantee: duplicates are only detected within each message group, not across the whole queue. Use high throughput mode when you have a large number of independent message groups (many customers) and do not need cross-group deduplication. In-flight message limit: 20,000 for FIFO (vs 120,000 for Standard) -- if you hit this limit, new ReceiveMessage calls return an error until you delete some in-flight messages.

### 4. Amazon SNS Fanout, Subscription Filter Policies, and Payload-Based Filtering -- AWS Official Docs + AWS Compute Blog ⏱️ 20 min
- **URLs:**
  - Fanout scenario: https://docs.aws.amazon.com/sns/latest/dg/sns-common-scenarios.html
  - Subscription filter policies: https://docs.aws.amazon.com/sns/latest/dg/sns-subscription-filter-policies.html
  - Filter policy constraints: https://docs.aws.amazon.com/sns/latest/dg/subscription-filter-policy-constraints.html
  - Payload-based filtering blog: https://aws.amazon.com/blogs/compute/introducing-payload-based-message-filtering-for-amazon-sns/
- **Type:** Official Docs + AWS Compute Blog
- **Summary:** The canonical SNS pattern and its two modes of filtering. The fanout scenario page establishes the pattern: a single SNS topic with N SQS subscriptions, where each queue receives a copy of every published message and can be processed by an independent consumer with its own scaling and visibility timeout. This is the workhorse pattern for decoupling one producer from many consumers (classic examples: order-placed event fanned out to inventory, billing, analytics, notifications). **Subscription filter policies** reduce cost and consumer load by only delivering messages that match a JSON pattern. Two scopes determined by the FilterPolicyScope attribute: (1) **MessageAttributes** (the default and historical mode) filters on the structured attributes you attach to a Publish call -- you cannot filter on message body content, so publishers must duplicate routing information into attributes, (2) **MessageBody** (announced at re:Invent 2022) filters on the JSON body directly without requiring duplicated attributes, using the same pattern syntax as EventBridge event patterns (prefix, suffix, numeric ranges, exists, anything-but, $or). Payload-based filtering is the modern default for new workloads because it eliminates the "must remember to populate attributes" footgun. Filter policy constraints to note: up to 5 attribute names per policy, up to 200 filter policies per topic (soft), up to 150 values per attribute, nested JSON depth of 5 levels for body filtering. **Gotcha**: filter policy changes take up to 15 minutes to fully propagate -- do not expect instant delivery behavior changes in tests. The decision trade-off vs EventBridge: EventBridge has richer filter semantics (larger patterns, more operators, schema registry integration) and supports cross-account/cross-region/archive-replay, but SNS+SQS fanout is approximately half the cost at scale ($0.50/M SNS publishes + SQS receive charges vs $1.00/M EventBridge custom events) and has simpler semantics for basic pub/sub use cases. If you only need to broadcast to 2-5 consumers with simple filtering, SNS+SQS is usually the right choice; if you need content-based routing across multiple sources with replay or schema discovery, EventBridge earns its premium. Re-read the decision guide from yesterday's study plan if you need a refresher on the full comparison matrix.

### 5. SNS FIFO Topics and Subscription Dead-Letter Queues -- AWS Official Docs ⏱️ 15 min
- **URLs:**
  - SNS FIFO topics: https://docs.aws.amazon.com/sns/latest/dg/sns-fifo-topics.html
  - FIFO message grouping: https://docs.aws.amazon.com/sns/latest/dg/fifo-message-grouping.html
  - SNS dead-letter queues: https://docs.aws.amazon.com/sns/latest/dg/sns-dead-letter-queues.html
  - Configuring SNS DLQ: https://docs.aws.amazon.com/sns/latest/dg/sns-configure-dead-letter-queue.html
- **Type:** Official Docs (four short pages)
- **Summary:** Two SNS features that have subtle, often-missed semantics. **SNS FIFO topics** guarantee strict ordering (using MessageGroupId exactly like SQS FIFO) and exactly-once publication (using MessageDeduplicationId with a 5-minute window). The critical constraint vs Standard SNS: FIFO topics can only deliver to SQS FIFO queues and (as of September 2023) to SQS Standard queues -- but NOT to Lambda, HTTP/S, email, SMS, Kinesis Firehose, or mobile push endpoints. The reason is that only SQS queues can preserve ordering semantics downstream; pushing directly to Lambda cannot guarantee order because Lambda scales invocations in parallel. The MessageGroupId propagates from the FIFO topic into the subscribed SQS FIFO queues, preserving per-group ordering end-to-end. Use SNS FIFO topics when you need fanout (multiple subscribers receive the same ordered stream) AND strict ordering -- but if you only need ordering to a single consumer, skip the topic and use an SQS FIFO queue directly. **SNS subscription DLQs** are the most commonly misunderstood SNS feature. Key points: (1) the DLQ is configured at the subscription level via a RedrivePolicy JSON object on the subscription, NOT at the topic level -- each subscription has its own DLQ, so a topic with five subscribers can have five different DLQs, (2) the DLQ receives messages that SNS failed to deliver to that specific subscription after exhausting the delivery retry policy (default 3 retries with exponential backoff for HTTP/S, variable for other protocols), (3) the DLQ MUST be an SQS queue (not another SNS topic, not EventBridge), (4) the SQS DLQ must have a resource-based policy allowing the SNS service principal to SendMessage, (5) messages filtered out by subscription filter policies are NEVER sent to the DLQ because filtering happens before delivery is even attempted -- filtering is not "failure". Contrast with EventBridge: EventBridge targets have per-target DLQs with similar semantics. Compare with SQS: SQS DLQs activate on consumer-side maxReceiveCount failures, whereas SNS DLQs activate on delivery-side failures (the subscriber never successfully received the message) -- these are two distinct failure modes, and a production architecture often needs both.

### 6. SQS vs SNS vs EventBridge Decision Guide and Cross-Service Patterns -- AWS Docs + AWS Compute Blog ⏱️ 15 min
- **URLs:**
  - Decision guide: https://docs.aws.amazon.com/decision-guides/latest/sns-or-sqs-or-eventbridge/sns-or-sqs-or-eventbridge.html
  - Choosing messaging services blog: https://aws.amazon.com/blogs/compute/choosing-between-messaging-services-for-serverless-applications/
  - Building resilient patterns blog: https://aws.amazon.com/blogs/compute/building-resilient-no-code-serverless-patterns-by-combining-messaging-services/
- **Type:** Official Docs + Two AWS Compute Blog posts
- **Summary:** The critical synthesis resource that ties today's material back to yesterday's EventBridge study. You already read the decision guide yesterday -- re-read it today through the SQS/SNS lens, paying attention to the comparison dimensions: communication model (pull vs push vs event-driven), persistence (SQS persists up to 14 days, SNS/EventBridge do not persist beyond delivery attempts unless you use EventBridge Archive), ordering (SQS FIFO and SNS FIFO guarantee order, EventBridge does not), filtering power (EventBridge > SNS payload filter > SNS attribute filter > SQS has none), throughput, and delivery targets. The "Choosing between messaging services" blog adds decision trees with concrete scenarios: "I need to decouple a producer from a consumer with buffering" (SQS), "I need to broadcast an event to multiple independent consumers" (SNS topic fanout), "I need content-based routing across multiple sources with replay" (EventBridge), "I need a managed scheduled job" (EventBridge Scheduler). The "Building resilient serverless patterns" blog shows composition patterns that combine all three: EventBridge rules routing to SNS topics that fan out to SQS queues (using EventBridge for routing logic, SNS for fanout, SQS for consumer buffering), EventBridge rules directly targeting SQS queues with DLQ for reliable event consumption, and Lambda functions processing SQS messages that were populated by SNS subscriptions. The pattern to internalize: these services compose rather than compete. A common production architecture is "EventBridge for routing -> SNS for fanout -> SQS for buffering -> Lambda for compute", where each service plays to its strengths. The decision is rarely "SQS or SNS or EventBridge" -- it is usually "where in the pipeline does each service fit". Cost intuition to carry into interviews: SQS ($0.40/M requests after 1M free tier) is cheapest because consumers pull, SNS ($0.50/M publishes plus downstream SQS costs) adds a small premium for push-based fanout, EventBridge ($1.00/M custom events) is roughly 2x SNS for the content-based routing, schema registry, and cross-account/cross-region features. AWS service events flowing through EventBridge are free.

### 7. Well-Architected Best Practices for SQS and the SNS+SQS Fan-Out Throttle Pattern -- AWS Compute Blog + Jeremy Daly ⏱️ 15 min
- **URLs:**
  - Well-Architected best practices for SQS (Part 2): https://aws.amazon.com/blogs/compute/implementing-aws-well-architected-best-practices-for-amazon-sqs-part-2/
  - Jeremy Daly -- SNS+SQS distribute and throttle pattern: https://www.jeremydaly.com/how-to-use-sns-and-sqs-to-distribute-and-throttle-events/
- **Type:** AWS Compute Blog + Expert Community Blog (free)
- **Summary:** Two practitioner-level resources that move from "what SQS and SNS do" to "how to operate them in production." The Well-Architected blog walks through the four Well-Architected pillars applied to SQS: **Security** (SSE-SQS for AWS-managed encryption at no extra cost vs SSE-KMS with customer-managed keys for cross-account and fine-grained key policies, VPC interface endpoints for private connectivity without NAT, IAM resource policies for cross-account access, least-privilege on SendMessage/ReceiveMessage/DeleteMessage), **Reliability** (DLQs with alarmed ApproximateNumberOfMessagesVisible metric, visibility timeout tuning to 6x Lambda timeout, idempotent consumers for at-least-once semantics, partial batch failure via ReportBatchItemFailures), **Performance** (long polling with 20-second wait time, SendMessageBatch/DeleteMessageBatch up to 10 messages per call to amortize API costs, message attributes for routing metadata to avoid parsing the body, FIFO high-throughput mode when order-within-group is sufficient), and **Cost Optimization** (free tier of 1M requests/month, batching drops per-message cost by up to 10x, consider SQS extended client library for messages over 256 KB to use S3 for the payload and SQS for the pointer rather than paying for higher-cost alternatives). The Jeremy Daly blog (published January 2019, still canonical) explains the **distribute and throttle pattern**: SNS topic -> multiple SQS queues (each with its own filter policy and its own consumer) as a way to both fan out events and throttle downstream processing independently. The key insight: SQS queues act as shock absorbers between a bursty producer and downstream systems with different capacity constraints -- a relational database write may handle 100 TPS, a third-party API call may rate-limit to 10 RPS, and a long-running analytics job may process batches of 50. By placing an SQS queue in front of each consumer, you can tune the consumer's pull rate (Lambda reserved concurrency, worker count, batch size) independently and absorb spikes without overwhelming the downstream. This pattern is how most production systems handle asymmetric downstream capacity and it directly connects to the Lambda reserved concurrency and event source mapping knowledge from Monday.

## Study Tips

- **Build a three-column decision matrix: SQS, SNS, EventBridge.** Rows: communication model (pull/push/event), persistence (14 days / none / none+archive optional), ordering (FIFO opt-in / FIFO opt-in / none), filtering (none / attribute+body / content-based pattern), delivery guarantee (at-least-once / at-least-once / at-least-once), max message size (256 KB / 256 KB / 256 KB), max retention (14 days / N/A / archive), throughput (unlimited/millions fan-out/thousands per sec per region), cost per million ($0.40 / $0.50 / $1.00 custom). Then add a fourth "composition" column showing how each service wires to the others (SQS as target for SNS subscriptions, SQS/SNS as targets for EventBridge rules, SNS as target for EventBridge rules). This single artifact answers 80% of messaging interview questions and forces you to internalize the trade-offs instead of memorizing service facts.

- **Trace the visibility timeout math for a Lambda+SQS integration end-to-end.** Scenario: Lambda function timeout 30 seconds, MaximumBatchingWindowInSeconds 5 seconds, SQS visibility timeout ???. AWS recommends 6 * 30 + 5 = 185 seconds minimum. Walk through what happens at each edge case: (1) Lambda finishes in 20 seconds and deletes messages -- everything works, (2) Lambda hits 30-second timeout without deleting -- SQS waits another 155 seconds before re-delivering, (3) SQS re-delivers and the second invocation also fails and does not delete -- maxReceiveCount increments to 2, (4) after maxReceiveCount attempts, SQS moves the message to the DLQ. Now trace the broken version: Lambda timeout 30 seconds, SQS visibility timeout 30 seconds. Lambda runs for 28 seconds, SQS re-delivers at second 30 to another Lambda invocation while the first is still running, so you have two Lambda invocations processing the same message with different idempotency tokens. This is the most common SQS+Lambda production bug and you should be able to draw the failure timing diagram from memory.

- **Map the SNS+SQS fan-out pattern to a real scenario and enumerate every DLQ.** Scenario: `OrderPlaced` event published to SNS topic `orders-topic` with three subscriptions -- inventory SQS queue, billing SQS queue, analytics SQS queue. Each SQS queue triggers a Lambda. Enumerate every possible failure mode and the corresponding DLQ: (1) SNS fails to deliver to the billing queue because of a misconfigured IAM policy -> SNS subscription DLQ for the billing subscription, (2) the analytics Lambda crashes on a malformed message -> SQS main-queue DLQ for the analytics queue after maxReceiveCount attempts, (3) the inventory Lambda is async-invoked downstream and its async destination fails -> Lambda async failure destination, (4) the billing filter policy has a typo and filters out all messages -> no DLQ anywhere (filtering is not a failure), you discover this via CloudWatch NumberOfMessagesPublished dropping to zero. This exercise forces you to internalize the difference between delivery failures, processing failures, and silent filter misconfigurations, which is often the highest-value SQS/SNS interview question.
