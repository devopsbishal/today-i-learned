# S3 Advanced Deep Dive -- 2-Hour Study Plan

**Total Time:** ~130 minutes
**Created:** 2026-03-23
**Difficulty:** Intermediate-Advanced

## Overview

This study plan covers the advanced operational features of Amazon S3 that go beyond basic bucket and object operations: the eight storage classes and how to transition between them cost-effectively, lifecycle policies for automated data management, cross-region and same-region replication for DR and compliance, access points for delegated access management, Object Lock for WORM compliance, event notifications for event-driven architectures, versioning with MFA Delete protection, Transfer Acceleration, S3 Select, and the critical distinction between server access logging and CloudTrail data events. After completing this plan, you will be able to design S3 storage strategies that optimize cost across the full data lifecycle, architect replication topologies for multi-region resilience, and answer interview questions about storage class transition constraints, replication gotchas, and Object Lock compliance requirements -- building on your CloudFront OAC knowledge (Mar 13) where S3 served as an origin, and setting the foundation for the DR and multi-region architecture topics coming in Week 5.

## Resources

### 1. Understanding and Managing Amazon S3 Storage Classes -- Official Overview ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/AmazonS3/latest/userguide/storage-class-intro.html
- **Type:** Official Docs
- **Summary:** The foundational reference page documenting all eight production storage classes with their durability, availability, AZ count, minimum storage duration, minimum object size, and retrieval fees in a single comparison table: S3 Standard (the default, 99.999999999% durability, 99.99% availability, no minimums), S3 Intelligent-Tiering (automatic tiering across Frequent, Infrequent, and Archive Instant Access tiers with optional Deep Archive, no retrieval fees but $0.0025 per 1,000 objects/month monitoring fee), S3 Standard-IA (30-day minimum, 128 KB minimum, per-GB retrieval fee), S3 One Zone-IA (same as Standard-IA but single AZ -- 99.5% availability instead of 99.9%), S3 Glacier Instant Retrieval (90-day minimum, millisecond retrieval, highest retrieval cost among Glacier classes), S3 Glacier Flexible Retrieval (90-day minimum, minutes-to-hours retrieval via Expedited/Standard/Bulk tiers), S3 Glacier Deep Archive (180-day minimum, 12-48 hour retrieval, lowest storage cost at ~$0.00099/GB/month), and S3 Express One Zone (directory buckets, single-digit millisecond latency, session-based auth, co-located with compute); read the full comparison table carefully as storage class selection questions are among the most common S3 interview topics

### 2. Transitioning Objects Using Amazon S3 Lifecycle -- Constraints and Waterfall ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/AmazonS3/latest/userguide/lifecycle-transition-general-considerations.html
- **Type:** Official Docs
- **Summary:** The single most important page for S3 interviews -- contains the storage class transition waterfall diagram showing which classes can transition to which (transitions only flow "downward": Standard can go anywhere, but Glacier Deep Archive is terminal and cannot transition to anything via lifecycle rules), along with every constraint you need to memorize: 30-day minimum before transitioning from Standard to Standard-IA or One Zone-IA, objects smaller than 128 KB are excluded from transitions by default (overridable with ObjectSizeGreaterThan/ObjectSizeLessThan filters), early deletion charges apply if you transition out before the minimum storage duration expires (30 days for IA classes, 90 for Glacier/Glacier IR, 180 for Deep Archive), each archived object incurs 8 KB of metadata stored at S3 Standard rates plus 32 KB at Glacier rates (making archival of millions of tiny objects potentially more expensive than keeping them in Standard), and when multiple lifecycle rules conflict the deletion action takes precedence over transition; this page alone will answer 80% of lifecycle-related interview questions

### 3. Lifecycle Configuration Elements and Examples ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonS3/latest/userguide/intro-lifecycle-rules.html
- **Type:** Official Docs
- **Summary:** The implementation reference for lifecycle rules covering the XML/JSON structure you need to understand: each rule has a filter (prefix, tag, object size, AND combinations), a status (Enabled/Disabled), and one or more actions (Transition, Expiration, NoncurrentVersionTransition, NoncurrentVersionExpiration, AbortIncompleteMultipartUpload); critical details include how NoncurrentVersionExpiration works with versioned buckets (deletes noncurrent versions after a specified number of days, with an optional NewerNoncurrentVersions parameter to retain the N most recent noncurrent versions), how AbortIncompleteMultipartUpload auto-cleans orphaned multipart uploads after a configurable number of days (a cost-saving essential since incomplete uploads consume storage silently), and how ExpiredObjectDeleteMarker removes delete markers when they are the only remaining version of an object; skim the examples page (https://docs.aws.amazon.com/AmazonS3/latest/userguide/lifecycle-configuration-examples.html) afterward for practical patterns

### 4. Replicating Objects Within and Across Regions -- CRR, SRR, RTC, and Batch Replication ⏱️ 20 min
- **URL:** https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication.html
- **Type:** Official Docs
- **Summary:** The conceptual overview of S3 replication covering when to use each type: CRR (Cross-Region Replication) for compliance requirements mandating geographically separated copies, latency reduction by placing data closer to users, and DR with warm standby in another region; SRR (Same-Region Replication) for log aggregation from multiple source buckets into a single bucket, live replication between production and test accounts, and data sovereignty requirements keeping copies within the same region; Replication Time Control (RTC) for SLA-backed replication with 99.99% of objects replicated within 15 minutes (costs extra but essential for compliance workloads); and S3 Batch Replication for retroactively replicating existing objects that were created before replication was enabled, re-replicating objects that previously failed, and replicating to newly added destinations; key requirements: versioning must be enabled on both source and destination buckets, IAM role must grant S3 permission to replicate, and by default only new objects are replicated after the rule is created -- follow the link to the requirements page (https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication-requirements.html) for the full list of what does and does not replicate (SSE-C encrypted objects do not replicate, lifecycle actions do not replicate, objects in the source bucket before replication was enabled do not replicate unless you use Batch Replication)

### 5. S3 Object Lock -- Governance Mode, Compliance Mode, Legal Hold, and WORM ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-lock.html
- **Type:** Official Docs
- **Summary:** The complete reference for S3 Object Lock and its WORM (Write Once Read Many) storage model, covering the three protection mechanisms: Retention Period with Governance Mode (prevents most users from deleting or overwriting, but users with s3:BypassGovernanceRetention permission and the x-amz-bypass-governance-retention:true header can override -- useful for testing retention policies before committing to compliance mode), Retention Period with Compliance Mode (absolutely no one can delete or overwrite, not even the root account, and the retention period cannot be shortened -- the only way out is to wait for expiration or delete the entire AWS account; assessed for SEC 17a-4, CFTC, and FINRA regulatory compliance), and Legal Hold (a binary on/off flag independent of retention periods, with no expiration date, toggled by users with s3:PutObjectLegalHold permission -- commonly used for litigation holds where the duration is unknown); critical prerequisite: Object Lock must be enabled at bucket creation time and requires versioning to be enabled; an object can have both a retention period AND a legal hold simultaneously, and both must be cleared before the object can be deleted

### 6. S3 Multi-Region Access Points and Access Points ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonS3/latest/userguide/MultiRegionAccessPoints.html
- **Type:** Official Docs
- **Summary:** Covers Multi-Region Access Points (MRAPs) which provide a single global endpoint (using AWS Global Accelerator under the hood -- connecting to your GA knowledge from Mar 18) that automatically routes S3 requests to the closest bucket based on network latency, with failover controls that let you shift traffic between regions in minutes for active-active or active-passive configurations; also read the access points overview page (https://docs.aws.amazon.com/AmazonS3/latest/userguide/access-points.html) which explains single-bucket access points -- named network endpoints attached to a bucket, each with its own IAM policy that simplifies managing access for shared datasets by creating per-application or per-team access points with distinct permissions instead of a single complex bucket policy, with optional VPC restriction that limits access to requests originating from a specific VPC (eliminates the need for VPC endpoint policies to restrict bucket access); understanding the progression from bucket policies to access points to multi-region access points shows how S3 access management has evolved from monolithic to delegated

### 7. S3 Event Notifications, Versioning, Transfer Acceleration, and Logging ⏱️ 15 min
- **URL:** https://docs.aws.amazon.com/AmazonS3/latest/userguide/EventNotifications.html
- **Type:** Official Docs
- **Summary:** The event notifications overview covering the two delivery paths: classic notifications (directly to SNS, SQS standard queues, or Lambda -- note that SQS FIFO queues are NOT supported as a direct destination) and EventBridge integration (sends all S3 events to EventBridge where you can create rules with advanced filtering, fan-out to multiple targets, archive and replay events, and route to SQS FIFO queues); key gotchas: S3 delivers events at least once (design for idempotent processing), events may arrive out of order, and if your notification target writes back to the same bucket that triggers the notification you can create an infinite loop (use prefix filters or separate buckets to prevent this); after this page, quickly skim three additional pages: Versioning workflows (https://docs.aws.amazon.com/AmazonS3/latest/userguide/versioning-workflows.html) for how GET/PUT/DELETE behave with versioned objects and how MFA Delete requires root account credentials via CLI only; Transfer Acceleration (https://docs.aws.amazon.com/AmazonS3/latest/userguide/transfer-acceleration.html) for how uploads route through CloudFront edge locations over the AWS backbone to the bucket region; and the logging comparison page (https://docs.aws.amazon.com/AmazonS3/latest/userguide/logging-with-S3.html) which contains the definitive server access logging vs CloudTrail data events comparison table -- the key distinction is that server access logs are free (you pay only for storage), delivered within hours, and capture authentication failures, while CloudTrail data events cost money, deliver within 5-15 minutes, support log integrity validation and cross-account delivery, but do not capture authentication failures

### 8. AWS Storage Blog -- Applying S3 Object Lock at Scale for Petabytes of Existing Data ⏱️ 10 min
- **URL:** https://aws.amazon.com/blogs/storage/applying-amazon-s3-object-lock-at-scale-for-petabytes-of-existing-data/
- **Type:** Blog (AWS Official)
- **Summary:** A practical AWS Storage Blog post that goes beyond the conceptual docs to show how organizations actually implement Object Lock at enterprise scale using S3 Batch Operations to apply retention settings to billions of existing objects -- this bridges the gap between understanding Object Lock modes (Resource 5) and knowing how to operationalize them; covers the pattern of using S3 Inventory to generate a manifest of existing objects, then S3 Batch Operations to apply governance or compliance retention to every object in the manifest, which is the only practical approach when you have petabytes of pre-existing data that needs retroactive WORM protection; this pattern frequently appears in architecture discussions about migrating regulated workloads to S3

## Study Tips

- Build a storage class decision matrix with these columns: storage class name, minimum storage duration, minimum object size, retrieval time, retrieval cost, and ideal use case. The interview trick question is about the 128 KB minimum object size for transitions and the metadata overhead (8 KB Standard + 32 KB Glacier per object) that makes archiving millions of tiny files counterproductive -- always calculate whether the metadata overhead exceeds the storage savings before recommending Glacier for small-object workloads. For Intelligent-Tiering, know that it has NO retrieval fees (unlike Standard-IA and the Glacier classes) but charges a monitoring fee per object, making it ideal when access patterns are unpredictable but not cost-effective for buckets with millions of objects that are known to be rarely accessed (use Glacier directly instead).

- Trace the replication data flow end to end: source bucket receives a PUT, S3 checks replication rules (prefix/tag filters), assumes the IAM replication role, reads the object from the source, and writes it to the destination bucket in the destination region (for CRR) or same region (for SRR). Critical gotchas to internalize: (1) replication is asynchronous by default and does NOT replicate existing objects unless you use Batch Replication, (2) lifecycle actions on the source do NOT replicate to the destination -- you need separate lifecycle policies on each bucket, (3) delete markers are replicated by default but permanent deletes (version ID deletes) are NOT replicated to prevent malicious cross-region deletion, (4) SSE-C encrypted objects cannot be replicated, and (5) replication does not cascade -- if bucket A replicates to B, and B replicates to C, objects replicated from A to B will NOT automatically replicate from B to C.

- Connect S3 features to services you already know: S3 event notifications feed into the Lambda/SQS/SNS/EventBridge ecosystem you will study in Week 6, Transfer Acceleration uses the same CloudFront edge locations you studied on Mar 13 (but for uploads rather than cached downloads), Multi-Region Access Points use the Global Accelerator infrastructure you studied on Mar 18, OAC from your CloudFront deep dive controls how CloudFront accesses S3 origins, and Object Lock connects to your KMS encryption knowledge (Mar 9) since locked objects can also be encrypted with SSE-KMS. Building these cross-service mental models is what separates strong architecture answers from surface-level responses.
